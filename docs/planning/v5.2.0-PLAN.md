### H4. verify-release.sh version consistency gate
The script catches stale counts but not stale version numbers. The B1 slip (manifest showing 4.9.0/PAI after v5.1.0 shipped) happened because no gate cross-checks version sources. Also: `VERSION` file currently says `4.9.0` while `preferences.jsonc` and `manifest.json` say `5.1.0` — a third source to include.

**Single source of truth: `manifest.json`** — per architecture decisions, manifest.json is a generated data file that reflects the system's actual state. All other version-bearing files must match it, not the other way around. So the check reads manifest.json first, then validates others against it.

Uses the existing `fail` helper defined at the top of `verify-release.sh` (same convention as all other checks in the script).

Add to section 4 (Ground Truth Counts):
```bash
# manifest.json is the canonical version SoT — all others must match it
MANIFEST_VER=$(grep -E '"version"\s*:' manifest.json | grep -oE '[0-9]+\.[0-9]+\.[0-9]+' | head -1)

if [[ -n "$MANIFEST_VER" ]]; then
  PREFS_VER=$(grep -E '"version"\s*:' config/preferences.jsonc | grep -oE '[0-9]+\.[0-9]+\.[0-9]+' | head -1)
  FILE_VER=$(cat VERSION 2>/dev/null | tr -d '[:space:]')
  INSTALL_VER=$(grep "Installer v" install.sh | head -1 | grep -oE '[0-9]+\.[0-9]+' | head -1)

  if [[ -n "$PREFS_VER" && "$PREFS_VER" != "$MANIFEST_VER" ]]; then
    fail "preferences.jsonc version ($PREFS_VER) != manifest.json version ($MANIFEST_VER) — run BuildManifest.ts"
  fi
  if [[ -n "$FILE_VER" && "$FILE_VER" != "$MANIFEST_VER" ]]; then
    fail "VERSION file ($FILE_VER) != manifest.json version ($MANIFEST_VER)"
  fi
  # install.sh banner shows major.minor only — allow major.minor match
  if [[ -n "$INSTALL_VER" && "$INSTALL_VER" != "${MANIFEST_VER%.*}" && "$INSTALL_VER" != "$MANIFEST_VER" ]]; then
    fail "install.sh banner version ($INSTALL_VER) != manifest.json version ($MANIFEST_VER)"
  fi
fi
```

**File:** `scripts/verify-release.sh`

## Phase R: Routing Intelligence

These are the v5.1.0 NEXT-STEPS items 1-3, carried forward.

### R1. ReadTracker.hook.ts
`PostToolUse:Read` hook. Async (no stdout, no session impact).

```typescript
// Writes to MEMORY/STATE/read-log.jsonl
{ timestamp, session_id, path, project_dir }
```

- Only tracks paths inside `PAI_DIR` — skip external files
- Deduplicates within a session (same path multiple times = 1 entry)
- Register in `config/hooks.jsonc` as async `PostToolUse:Read`
- **Log rotation:** trim entries older than `--days` window on each write (default: keep 90 days). Also cap file at 1MB — if exceeded, drop oldest 25% before appending. Prevents unbounded growth on heavy-use installs.

**File:** `hooks/ReadTracker.hook.ts` (new)

### R2. RoutingCandidates.ts
CLI tool. Reads `MEMORY/STATE/read-log.jsonl`, aggregates by path, cross-references `CONTEXT_ROUTING.md`, outputs candidates.

```
bun PAI/Tools/RoutingCandidates.ts [--threshold 3] [--days 30]
```

Output: paths read in ≥N distinct sessions that have no routing entry. Includes read count and last-seen date.

**File:** `PAI/Tools/RoutingCandidates.ts` (new)

### R3. RoutingAudit.ts propose mode
Add `propose` subcommand to existing `PAI/Tools/RoutingAudit.ts`:

```
bun PAI/Tools/RoutingAudit.ts propose [--threshold 3]
```

- Reads RoutingCandidates output
- For each candidate, reads first 200 chars of file content
- Uses **Haiku** (fast tier) inference to generate a 3-5 word topic label — one-shot labeling, latency matters
- **Offline fallback:** if inference is unavailable (no API key, budget exhausted), use the filename stem as the label (e.g., `CONTEXT_ROUTING.md` → `context routing`) and mark it `[needs-label]` in output so user knows it needs manual editing
- Outputs copy-paste ready routing table rows

Optional: add nudge to `LoadContext.hook.ts` — if proposals exist and last nudge >7 days ago, append one-liner to session context.

**File:** `PAI/Tools/RoutingAudit.ts` (modify), `hooks/LoadContext.hook.ts` (optional nudge)

### R4. BuildSettings dry-run mode
```
bun hooks/handlers/BuildSettings.ts --dry-run
```
Runs `buildSettings()`, diffs against current `settings.json`, prints unified diff, exits without writing. Uses JS-native line diff (no external dep).

**File:** `hooks/handlers/BuildSettings.ts`

## Phase S: Steering Enforcement Design Spike

Deferred from v5.1.0. The failure mode: "diagnosis flowing into obvious fix flowing into execution overrides the plan-means-stop rule."

**Design spike (no code, just a document):**

Leading candidate: `UserPromptSubmit` hook that detects plan-presentation patterns in the last assistant message and injects a "waiting for approval" context reminder if the next user message looks like "go ahead" / "do it" / "ok start" without a plan hash in state.

Key open questions to answer in the spike:
1. How do we detect "last response was a plan" without reading full transcript? Single structural markers (`## Phase` alone) fire on any multi-step explanation. Require **≥2 of these signals** in the last assistant message to classify as "plan presented":
   - `## Execution Order` or `## Phase \d` section header
   - A completion gate block (lines matching `- \[ \]` checklist with gate language)
   - An explicit phase table with estimated times
   Requiring co-occurrence drastically reduces false positives on structured-but-non-plan responses.
2. What state persists between UserPromptSubmit and Stop? (session state file)
3. What approval signals constitute "user approved"? Explicit: "go ahead", "do it", "start", "execute", "approved". Implicit approval threshold TBD.
4. Should the gate be a Continue=false block or just an additionalContext injection? (Recommendation: additionalContext — softer, lower false-positive risk)

**Deliverable:** `docs/planning/steering-enforcement-design.md` — design doc with proposed approach, failure modes, and implementation spec ready for v5.3.0 implementation.

**File:** `docs/planning/steering-enforcement-design.md` (new)
