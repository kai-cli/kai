## Feature Details

### [2] Instinct file-revert detection (session write ledger)

**What:** Track files PAI writes during a session. When user subsequently writes to the same
file and the diff reverts PAI's changes, create an instinct with source `'revert'`.

**Why:** File reverts are strong correction signals that the current regex patterns miss.
User undoing PAI's work is unambiguous negative feedback.

**Implementation:**
- New `hooks/WriteTracker.hook.ts` (PostToolUse) — watches Write/Edit calls, maintains
  an in-memory `Map<string, LedgerEntry>` with `{path, timestamp, contentHash, snippet}`
- Extend `hooks/InstinctCapture.hook.ts` — on user Write/Edit to a tracked file, diff
  against PAI's last version. If diff reverts ≥50% of PAI's changed lines → create instinct
- Pure in-memory — no file written to disk. Map lives for the hook process lifetime
  (which matches the Claude Code session). No cleanup needed.

**Multi-edit sequencing:** The ledger tracks PAI's *most recent* write to each path. Each
PAI write overwrites the previous entry for that path. So the baseline for revert detection
is always the last thing PAI wrote, not the first. Sequence:
```
PAI writes foo.ts (v1) → ledger: {foo.ts: hash(v1)}
User edits foo.ts (non-revert) → no instinct (below threshold)
PAI writes foo.ts (v2) → ledger: {foo.ts: hash(v2)}  ← overwrites v1
User reverts foo.ts → compared against v2, not v1
```

**Session lifecycle:** The ledger is an in-memory `Map<string, LedgerEntry>` inside
WriteTracker.hook.ts. It lives for the duration of the hook process (which matches the
Claude Code session lifetime). No file cleanup needed — no SessionCloseGuard dependency.

**Context extraction for instinct text:** The instinct text is derived from:
1. The file path (what was reverted)
2. A summary of what PAI's change did (extracted from the `snippet` field — first 120 chars
   of the diff PAI introduced)
3. Template: `"Reverted: [snippet summary] in [filename]"`

This produces instincts like: `"Reverted: added error handling wrapper to parseConfig() in config.ts"`
— actionable enough for /evolve clustering without requiring conversation replay.

**Revert threshold:** 50% of PAI's changed lines must be removed or replaced. Rationale:
- <50%: user is editing/extending PAI's work (not a correction)
- ≥50%: user is undoing the core of what PAI did (correction signal)
- 100% exact revert: highest confidence (0.5 instead of default 0.3)

### [1] Semantic instinct dedup via embeddings

**What:** Before creating a new instinct, embed the candidate text and compare against all
active instincts. If cosine similarity > threshold → reinforce existing instinct instead
of creating a duplicate.

**Why:** Without dedup, slightly-reworded corrections create near-duplicate instincts that
waste the 100-instinct cap and dilute /evolve clustering. Tag-based dedup (exact text match)
misses semantic equivalents like "don't mock the database" vs "use real DB in tests".

**Implementation:**
- Extend `hooks/lib/instinct-store.ts` — `createInstinct()` calls dedup check first
- New `hooks/lib/instinct-dedup.ts` — embed candidate, compare against active instincts
- Reuses `hooks/lib/semantic-fallback.ts` pipeline (already loads jina model)

**Latency model:** InstinctCapture fires on UserPromptSubmit (async hook). The dedup check
runs inside the async path — it does NOT block the user's message from reaching Claude.
Worst case (~500ms for embed + compare against 100 instincts) means the instinct is created
slightly after the prompt is processed. This is acceptable: instinct creation is a side effect,
not user-facing latency.

**Dedup threshold:** 0.85 cosine similarity. Rationale:
- 0.9+ only catches near-identical rephrasing (too conservative)
- 0.8 risks merging genuinely different corrections that share vocabulary
- 0.85 is the midpoint; tunable in settings.json if empirically wrong

**Fallback:** When embedding model not installed, skip dedup entirely — allow duplicates.
Graceful degradation to v5.6 behavior (no dedup). Log warning once per session.

**Migration for pre-existing instincts:** Existing instincts have no stored embeddings.
On first dedup check, `instinct-dedup.ts` embeds all active instincts and caches vectors
in `MEMORY/STATE/embeddings/instinct-vectors.jsonl`. Subsequent checks only embed the new
candidate (cache invalidated when instincts are archived/created). First-run cost: ~2s for
100 instincts. Subsequent: ~50ms per candidate.

### [3] /evolve embedding-based clustering

**What:** Replace tag-overlap clustering (≥2 shared tags) with embedding-based grouping.
Cluster instincts by cosine similarity, surface tighter thematic groups for promotion.

**Why:** Tag-based clustering is limited by the hardcoded ~80 keyword vocabulary. Instincts
about the same behavioral pattern with different wording won't share tags. Embeddings
capture semantic similarity regardless of word choice.

**Implementation:**
- New `hooks/lib/instinct-cluster.ts` — agglomerative clustering algorithm
- Extend `hooks/lib/instinct-store.ts` — add `getClusteredInstincts()` export
- Update `skills/Evolve/SKILL.md` — change prompt instructions from "group by shared tags"
  to "call `getClusteredInstincts()` for embedding-based groups"
- Reuses instinct embedding cache from [1] (`instinct-vectors.jsonl`)

**SKILL.md change clarification:** The Evolve skill is an AI instruction document (not
executable code). The change is updating the clustering step instructions: instead of
telling the model to group by ≥2 shared tags, it tells the model to read the output of
`getClusteredInstincts()` (which the model calls via Read on a generated JSON file, or
the function is exposed via a tool-use pattern). The actual clustering runs in
`instinct-cluster.ts` — SKILL.md just references it.

**Clustering algorithm:**
- Agglomerative with 0.7 cosine threshold (group instincts with >0.7 pairwise similarity)
- **Max cluster size: 8** — prevents degenerate case where homogeneous instinct stores
  collapse into one mega-cluster. If a cluster exceeds 8, split by highest internal distance.
- **Min cluster size for promotion: 2** — single instincts aren't clusters.
- Uses cached vectors from `instinct-vectors.jsonl` (same cache as [1])

**Edge case — small/homogeneous store:** With <10 instincts that are all similar (e.g.,
all about "run tests first"), the algorithm may produce 1 large cluster. The max-size guard
splits it, but more importantly: /evolve already requires confidence ≥0.8 AND trigger_count ≥3
for promotion eligibility. A cluster of low-confidence instincts won't surface as promotable
regardless of clustering quality.

**Files:**
- `hooks/lib/instinct-cluster.ts` — new: clustering algorithm + max-size split
- `hooks/lib/instinct-store.ts` — add: `getClusteredInstincts()` export
- `skills/Evolve/SKILL.md` — update: reference embedding clustering in workflow steps

## Design Decisions (Resolved)

| Decision | Resolution | Tunable? |
|----------|-----------|----------|
| Dedup threshold | 0.85 cosine | Yes, `settings.json` |
| Revert threshold | ≥50% of PAI's changed lines reverted | Hardcoded v1 |
| Clustering threshold | 0.7 cosine, max 8 per cluster | Hardcoded v1 |
| Context extraction | Template: "Reverted: [snippet] in [file]" | N/A |
| Dedup latency | Async (non-blocking) — runs in UserPromptSubmit async path | N/A |
| Session ledger lifecycle | In-memory Map, dies with process | N/A |
| Pre-existing instinct migration | First-run embed + cache (~2s one-time cost) | N/A |

All thresholds are initial values. If empirically wrong, [1] and [3] thresholds are
trivial to adjust (single constant). [2]'s revert threshold may need revision after
observing false positive/negative rates in real sessions.
