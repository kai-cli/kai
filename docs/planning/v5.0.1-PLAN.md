## BLOCKERS — Must fix before v5.0.0 tag

These are release-integrity issues. verify-release.sh says 15/15 but is lying on at least 3 checks.

### B1. config/preferences.jsonc has wrong version and productName
- `version: "4.9.0"` → should be `"5.0.0"`
- `productName: "PAI"` → should be `"KAI"`
- This is the source of truth BuildManifest.ts reads from. **Fix first — everything else cascades.**
- pai-config keeps `"PAI"` (correct for dev repo); kai must be `"KAI"`.
- sync-to-kai.sh brand transforms should handle this, but verify.

### B2. manifest.json is stale (version, productName, hook count)
- Committed: `version: "4.9.0"`, `productName: "PAI"`, `hooks: 35`
- Actual: version 5.0.0, product KAI, 36 hooks (HealthCheck added in 00f33a9)
- Fix: after B1, run `PAI_DIR=$PWD bun PAI/Tools/BuildManifest.ts` in kai

### B3. Marker regions claim 35 hooks in 4 files
- `README.md:30`, `docs/QUICKSTART.md:75,111`, `docs/WHATS-DIFFERENT.md:14,71`, `scripts/deploy.ts:484` all have `<!-- KAI:counts:hooks:begin -->35<!-- end -->`
- CHANGELOG.md was hand-updated to 36 but marker-managed docs were not
- Fix: after B2, run `PAI_DIR=$PWD bun PAI/Tools/BuildDocs.ts` in kai

### B4. verify-release.sh count check has a bug (blind to marker syntax)
- `grep -oE '[0-9]+ hook'` requires "35 hooks" as space-separated text
- Marker syntax: `<!-- KAI:counts:hooks:begin -->35<!-- KAI:counts:hooks:end --> lifecycle hooks` — number is never followed by space+"hook"
- Pattern never matches → check always passes → false confidence
- Fix: strip marker tags before matching, or extract value between begin/end tags and compare to filesystem count

### B5. Multi-author check is warn, not fail
- `verify-release.sh` line 94: `warn "Multiple authors..."` — should be `fail`
- Currently passes because kai has one author, but the gate doesn't hold structurally
- Plan explicitly required promotion to FAIL after Round 4 PII regression

### B6. CHANGELOG claims 338 tests; suite has 367
- `CHANGELOG.md:18`: "338 tests passing" — actual count is 367
- Feature-claim verifier only checks hook names, not test count
- **Two deliverables:**
  1. Update CHANGELOG test count to actual (367)
  2. Add `counts.tests` to `BuildManifest.ts` and add test count verification to `verify-release.sh` — without this, B6 recurs every time tests are added. The manifest already tracks `counts.skills`/`counts.hooks`/`counts.agents`; `counts.tests` is a one-field extension of the same pattern

### B7. docs/planning/RELEASE-BLOCKERS.md missing from kai
- `release.sh` Gate 6 silently `gate_skip`s when file absent (should `gate_fail`)
- File exists in pai-config but is in EXCLUDE_PATHS for sync (pai-config-only)
- **Two deliverables:**
  1. Create `docs/planning/RELEASE-BLOCKERS.md` in kai with all items resolved (since we're releasing)
  2. Change `release.sh` Gate 6 from `gate_skip` to `gate_fail` when file is absent — a gate that silently skips is structurally the same as no gate

### B8. sync-to-kai.sh missing from kai (by design)
- Review flagged this, but it's intentionally excluded — sync script is pai-config-only tooling
- **Not a blocker.** The sync mechanism is rsync with an exclude list (`EXCLUDE_PATHS` in `sync-to-kai.sh`). New files sync by default unless explicitly excluded. All Phase 5.1 artifacts (`release.sh`, `scripts/hooks/pre-commit`, `scripts/hooks/pre-push`, `scripts/install-hooks.sh`, `BuildManifest.ts`, `BuildDocs.ts`, `.github/workflows/test.yml`) are NOT in the exclude list and therefore sync automatically. Only `RELEASE-BLOCKERS.md` is excluded (line 85), which is covered by B7.
- Reviewer note acknowledged: "Phase 5 has not been started" is incorrect — it was completed in pai-config, just not visible from kai.
