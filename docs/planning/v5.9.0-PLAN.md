## Feature Details

### [1] Skills Lock File (`skills-lock.json`)

**What:** A manifest that records every installed skill's source, version (content hash), and path — enabling reproducible installations, drift detection, and eventual multi-machine sync.

**Why:** PAI has 83+ skills with no way to detect drift between kai (source of truth) and ~/.claude (live installation). KAI distribution relies on `sync-to-kai.sh` doing full file copies without knowing what changed. A lock file makes the skill inventory explicit and auditable.

**Inspired by:** Warp's `skills-lock.json` with content hashing and `npx skills` CLI.

**Implementation:**

```typescript
// skills-lock.json (generated, committed)
{
  "version": 1,
  "generated": "2026-05-22T...",
  "skills": {
    "Research": {
      "source": "kai",
      "path": "skills/Research/SKILL.md",
      "hash": "sha256:a1b2c3...",
      "workflows": ["QuickResearch", "StandardResearch", "ExtensiveResearch"],
      "workflowHashes": {
        "QuickResearch": "sha256:...",
        "StandardResearch": "sha256:...",
        "ExtensiveResearch": "sha256:..."
      },
      "specializes": null
    }
  }
}
```

**Hashing policy:**
- Hash is SHA-256 of the **parsed content** (YAML frontmatter deserialized + body stripped of trailing whitespace), not raw file bytes. This prevents whitespace-only edits from triggering false drift alerts.
- Workflow hashes use the same normalization: parse frontmatter (if any), strip trailing whitespace, hash the result.
- Hashing the parsed representation means the lock file is stable across editors that differ on trailing newlines, but catches every semantic change.

**Drift response policy:**

| Context | Response |
|---------|----------|
| `verify` run locally | Exit code 1, print diff of drifted skills to stdout. Does NOT block skill loading. |
| `verify` in CI (`--strict` flag) | Exit code 1, CI fails. Enforces lock discipline on pull requests. |
| IntegrityCheck hook (SessionEnd) | Logs warning to `/tmp/pai-hooks/IntegrityCheck.log`. Does NOT block session. |
| `generate` after edits | Regenerates lock. This is the intended workflow — edit skill, then regenerate. |

The lock is informational by default, strict in CI. It never blocks skill loading at runtime — that would make the system fragile.

**Existing state:** `scripts/skills-lock.ts` and `skills-lock.json` already exist and are
called by `verify-release.sh`. This feature is enhancement/completion work — adding `--verify --strict`,
parsed-content hashing (replacing raw file hashing), and the drift response table above. NOT greenfield.
The ~200 LOC estimate applies to bringing it to spec, not building from scratch.

**Files:**

| File | Action |
|------|--------|
| `scripts/skills-lock.ts` | UPDATE — add `--verify --strict`, parsed-content hashing, drift response |
| `skills-lock.json` | UPDATE — regenerate with new hash format |
| `hooks/IntegrityCheck.hook.ts` | Update — verify lock matches installation |

**CLI interface:**
```bash
bun scripts/skills-lock.ts generate    # Regenerate lock from current skills
bun scripts/skills-lock.ts verify      # Check installation matches lock (exit 0/1)
bun scripts/skills-lock.ts verify --strict  # Same but for CI (verbose, fails hard)
bun scripts/skills-lock.ts diff        # Show what changed since last generate
```

**Validation:**
- `generate` produces deterministic output (same skills → same hash, regardless of trailing whitespace)
- `verify` detects when a skill file was edited in ~/.claude but not in kai
- `verify` does NOT fail on whitespace-only changes (parsed hash is stable)
- `diff` shows human-readable changes (added/removed/modified skills)
- Lock file updates are part of the commit workflow for skill changes

### [2] Skill Specialization (`specializes:` frontmatter)

**What:** A `specializes` field in SKILL.md frontmatter that declares inheritance from a parent skill. Project-specific skills override or extend system skills without duplicating the base.

**Why:** Multiple projects need slightly different versions of common skills (Research for firmware vs Research for general). Currently this requires full skill duplication. Specialization lets a project skill say "I'm a Research variant for firmware context" and inherit the parent's workflows while adding/overriding specifics.

**Inspired by:** Warp's `specializes: review-pr` pattern in `review-pr-local`.

**Implementation:**

```yaml
# ~/.claude/projects/-Users-your.name-Projects-your-project/skills/Research/SKILL.md
---
name: Research
description: "Firmware-context research with build system knowledge"
specializes: Research   # Resolves to system-level skill at ~/.claude/skills/Research/
use_when: "Research in your-project context. Adds build system and Jenkins context."
overrides:
  - StandardResearch    # This workflow replaces the parent's
extends:
  - FirmwareBuildSearch # This workflow is added (not in parent)
---
```

**Parent resolution:**
- `specializes: Research` resolves by searching in this order:
  1. System skills: `~/.claude/skills/Research/SKILL.md`
  2. If not found: error at lock generation time (not at runtime)
- The path is unambiguous because project skills live in `~/.claude/projects/{project}/skills/` and system skills live in `~/.claude/skills/`. A project skill named "Research" that declares `specializes: Research` always references the system-level skill — it cannot specialize itself.
- SkillGuard resolves context: when in a project directory, project skill takes priority. Outside project context, system skill is used.

**Error handling:**

| Condition | Behavior |
|-----------|----------|
| `overrides` lists a workflow not in parent | **Error at `skills-lock.ts generate`** — prints: "Research (project:your-project) overrides 'NonExistent' but parent has no such workflow. Available: [QuickResearch, StandardResearch, ExtensiveResearch]". Lock generation fails. |
| Parent skill doesn't exist | **Error at `skills-lock.ts generate`** — prints: "Research (project:your-project) specializes 'Research' but no system skill found at ~/.claude/skills/Research/". Lock generation fails. |
| Runtime: parent was deleted after lock | SkillGuard falls back to child-only (no inheritance). Logs warning. Next `verify` catches the drift. |

Errors are caught at lock generation time (development workflow), not at runtime. This keeps the system robust in production.

**Resolution order (runtime, via SkillGuard):**
1. Project skill workflows override same-named parent workflows
2. Project skill `extends` workflows are added to parent's set
3. Unmentioned parent workflows pass through unchanged
4. `use_when` from child replaces parent's (more specific context)

**Files:**

| File | Action |
|------|--------|
| `PAI/SKILLSYSTEM.md` | Update — document `specializes` field |
| `hooks/SkillGuard.hook.ts` | Update — resolve specialization chain |
| `scripts/skills-lock.ts` | Update — track parent-child relationships, validate at generate time |

**Validation:**
- A specialized skill inherits parent workflows not explicitly overridden
- `overrides` referencing a non-existent parent workflow → error at lock generation
- SkillGuard resolves the specialization chain correctly at runtime
- skills-lock.json records the `specializes` relationship
- Ambiguous resolution is impossible (project → system path hierarchy is deterministic)

### [3] Settings Schema (Canonical Schema, Validated Instances)

**What:** A handwritten JSON Schema that canonically defines all valid PAI/KAI settings — their types, constraints, enums, optionality, and defaults. `settings.json` is validated AGAINST this schema, not derived from it.

**Why:** The `update-config` skill currently hardcodes knowledge of what settings exist. When new settings are added (hooks, permissions, env vars), the skill doesn't know about them until manually updated. A canonical schema makes settings self-describing and enables validation at multiple points (skill, CI, hooks).

**Inspired by:** Warp's `define_settings_group!` macro that generates both typed accessors and JSON Schema. PAI uses a handwritten schema (TypeScript definitions is where constraints live; schema-from-instance is fragile because it can't express enums, required-vs-optional, or constraints not already in the example).

**Implementation:**

```typescript
// settings-schema.ts — canonical schema definition
// This is the source of truth. settings.json is validated against it.

const SETTINGS_SCHEMA = {
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "properties": {
    "permissions": {
      "type": "object",
      "properties": {
        "allow": {
          "type": "array",
          "items": { "type": "string" },
          "description": "Tool patterns to auto-approve (e.g., 'Bash(git *)', 'Read')"
        },
        "deny": {
          "type": "array",
          "items": { "type": "string" },
          "description": "Tool patterns to always block"
        }
      }
    },
    "hooks": {
      "type": "object",
      "propertyNames": {
        "enum": ["SessionStart", "UserPromptSubmit", "PreToolUse", "PostToolUse",
                 "Stop", "SessionEnd", "SubagentStop", "PreCompact",
                 "ConfigChange", "WorktreeCreate", "WorktreeRemove",
                 "TaskCompleted", "TeammateIdle"]
      },
      "additionalProperties": {
        "type": "array",
        "items": { "$ref": "#/$defs/hookEntry" }
      }
    },
    "env": {
      "type": "object",
      "additionalProperties": { "type": "string" },
      "description": "Environment variables injected into all hook/tool processes"
    },
    "mcpServers": {
      "type": "object",
      "additionalProperties": { "$ref": "#/$defs/mcpServer" }
    }
  },
  "$defs": {
    "hookEntry": {
      "type": "object",
      "required": ["hooks"],
      "properties": {
        "matcher": { "type": "string", "description": "Tool name to match (PreToolUse/PostToolUse only)" },
        "hooks": {
          "type": "array",
          "items": {
            "type": "object",
            "required": ["type", "command"],
            "properties": {
              "type": { "enum": ["command"] },
              "command": { "type": "string" },
              "async": { "type": "boolean", "default": false }
            }
          }
        }
      }
    },
    "mcpServer": {
      "type": "object",
      "properties": {
        "command": { "type": "string" },
        "args": { "type": "array", "items": { "type": "string" } },
        "env": { "type": "object", "additionalProperties": { "type": "string" } },
        "url": { "type": "string", "format": "uri" },
        "type": { "enum": ["stdio", "sse"] }
      }
    }
  }
};
```

**Validation points:**
1. `bun scripts/settings-validate.ts` — validates settings.json against schema (CI, pre-commit)
2. `update-config` skill — introspects schema to enumerate valid settings, types, and constraints before editing
3. `ConfigChange.hook.ts` — validates after any settings.json edit

**Files:**

| File | Action |
|------|--------|

### [4] Command Risk Classification

**What:** Enhance SecurityValidator hook with structured risk metadata on every shell command: `is_read_only`, `is_risky`, `is_destructive`, `uses_pager`, `modifies_git`.

**Why:** SecurityValidator currently uses pattern matching to block dangerous commands. Adding risk classification metadata enables more nuanced decisions: read-only commands can auto-approve, risky commands get extra scrutiny, and the classification data can feed into SecurityValidator's learning.

**Inspired by:** Warp's `AIAgentActionType::RequestCommandOutput` with `is_read_only`, `is_risky`, `uses_pager` fields.

**Implementation:**

```typescript
// hooks/lib/risk-classifier.ts
interface CommandRisk {
  command: string;
  is_read_only: boolean;     // ls, cat, grep, git status, git log
  is_risky: boolean;         // rm, git push, docker rm, kill
  is_destructive: boolean;   // rm -rf, git push --force, DROP TABLE
  uses_pager: boolean;       // less, more, man
  modifies_git: boolean;     // git commit, push, reset, rebase
}

function classifyCommand(command: string): CommandRisk { ... }
```

**Classification heuristics:**

| Category | Positive signals | Negative signals (NOT this) |
|----------|-----------------|----------------------------|
| `is_read_only` | `ls`, `cat`, `grep`, `find`, `git status/log/diff/show`, `echo`, `which`, `type`, `wc`, `head`, `tail` | Any write flag (`-w`, `>`, `>>`, `tee`) |
| `is_risky` | `rm` (without -rf), `kill`, `pkill`, `docker rm`, `git push` (without --force), network tools (`curl -X POST/PUT/DELETE`) | Read variants of same tools |
| `is_destructive` | `rm -rf`, `--force`/`-f` on git push/reset, `DROP`, `TRUNCATE`, `mkfs`, `dd`, `> /path/to/important-file` (redirect overwriting a specific file that isn't /dev/null) | `> /dev/null` (discarding stdout is safe, not destructive) |
| `uses_pager` | `less`, `more`, `man`, `git log` without `--no-pager` | Same commands with `--no-pager` or piped |
| `modifies_git` | `git commit`, `git push`, `git reset`, `git rebase`, `git merge`, `git cherry-pick`, `git tag` | `git status`, `git log`, `git diff`, `git show` |

**No `confidence` field in MVP.** The classifier is heuristic-only — every command gets a deterministic boolean classification based on pattern matching. A probabilistic confidence score would require a training set and threshold tuning that isn't justified for the first iteration. If we find the heuristics produce too many false positives in practice, we add confidence in a follow-up.

**Files:**

| File | Action |
|------|--------|
| `hooks/lib/risk-classifier.ts` | NEW — classification engine |
| `hooks/SecurityValidator.hook.ts` | Update — use classifier for decisions |
| `tests/risk-classifier.test.ts` | NEW — unit tests for classification |

**Validation:**
- `ls -la` → `is_read_only: true, is_risky: false`
- `rm -rf /tmp/foo` → `is_destructive: true`
- `git push --force origin main` → `is_destructive: true, modifies_git: true`
- `echo "test" > /dev/null` → `is_read_only: false, is_destructive: false` (discarding output is benign)
- `echo "test" > ~/.claude/settings.json` → `is_destructive: true` (overwriting a real file)
- SecurityValidator uses classification to auto-approve read-only commands
- All existing SecurityValidator tests still pass (no behavior regression)

### [7] Hot-Reload Rules

**What:** Detect changes to CLAUDE.md and steering rules files mid-session and inject updated context on the next prompt, without requiring session restart.

**Why:** Currently, changing CLAUDE.md requires starting a new session to pick up changes. During PAI development, this means constant session cycling. Hot-reload lets rules evolve mid-session.

**Inspired by:** Warp's file watchers on WARP.md/AGENTS.md with instant rule application.

**Implementation (mtime-based, realistic path):**

Claude Code hooks are one-shot processes — they cannot maintain persistent file watchers. The practical implementation is an mtime check on every UserPromptSubmit:

```typescript
// hooks/lib/rules-watcher.ts
const WATCHED_FILES = [
  `${process.env.HOME}/.claude/CLAUDE.md`,
  `${process.env.HOME}/.claude/PAI/AISTEERINGRULES.md`,
  // Project CLAUDE.md resolved from cwd
];

const MTIME_CACHE_PATH = "/tmp/pai-hooks/rules-mtime.json";

interface MtimeCache {
  [path: string]: number;  // last-known mtime in ms
}

function checkForChanges(): { changed: boolean; files: string[]; delta?: string } {
  const cache = readMtimeCache();
  const changed: string[] = [];
  
  for (const file of WATCHED_FILES) {
    const currentMtime = fs.statSync(file, { throwIfNoEntry: false })?.mtimeMs;
    if (currentMtime && currentMtime > (cache[file] ?? 0)) {
      changed.push(file);
      cache[file] = currentMtime;
    }
  }
  
  if (changed.length > 0) {
    writeMtimeCache(cache);
    return { changed: true, files: changed, delta: computeDelta(changed) };
  }
  return { changed: false, files: [] };
}
```

- On every UserPromptSubmit: check mtime of watched files against cached values (~1ms)
- If mtime increased: re-read the changed file, compute a summary of what changed
- Inject via `additionalContext`: "Rules updated in CLAUDE.md: [summary of changes]"
- Only injects the delta (what changed), not the full file (Claude already has the original)

**Future path (aspirational):** If Claude Code adds support for persistent background processes or long-lived hooks, migrate to `fs.watch()` for instant detection instead of polling on each prompt. This would eliminate the one-prompt latency between edit and detection.

**Files:**

| File | Action |
|------|--------|
| `hooks/lib/rules-watcher.ts` | NEW — mtime-based change detection |
| `hooks/LocalContextFirst.hook.ts` | Update — integrate rules change detection into existing hook |

**Validation:**
- Edit CLAUDE.md mid-session → next prompt picks up changes via additionalContext
- No performance regression on UserPromptSubmit (mtime stat is <1ms per file)
- Only changed content is injected (not full file every time)
- Files that haven't changed → no additionalContext injected (no noise)
- Missing files (e.g., project CLAUDE.md doesn't exist) → silently skipped

### [8] ModeClassifier Calibration (Low-Effort Fix)

**What:** Targeted pattern fix in the existing ModeClassifier hook to reduce false NATIVE classifications for natural-language requests that imply complex work.

**Why:** The ALGORITHM classifier currently requires "action verb + technical object OR multi-step complexity." This misses natural phrases like "fix the auth bug" (no explicit technical object), "update my profile" (possessive reference), or "clean up the tests" (contextual reference). These get classified as NATIVE when they should trigger ALGORITHM.

**Implementation (~20 LOC):**

Add pattern recognition for:
- Possessive/contextual references as implicit technical objects: "my", "the", "our" + noun
- Common dev-intent verbs that always imply multi-step: "fix", "debug", "refactor", "investigate", "migrate"
- Phrases with implicit scope: "clean up", "set up", "wire up"

This is a targeted calibration pass, NOT a rewrite. v6.0 Feature [1] (Input Classification) will replace the whole classifier — this just reduces the miss rate in the interim.

| File | Action |

| `hooks/ModeClassifier.hook.ts` | Update — add possessive/contextual patterns |

- "fix the auth bug" → ALGORITHM (was: NATIVE)
