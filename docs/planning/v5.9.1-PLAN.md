### [1] Installer: API Key Setup Wizard

**What:** After identity setup (Step 3), add a new step that offers to configure API keys. Shows which keys are already set in the environment, which are missing, and collects values interactively.

**Why:** .env.example exists but nobody reads it at install time. The QUICKSTART mentions it in passing. Users discover they need keys only when a skill fails.

**Implementation:**

```typescript
// PAI-Install/main.ts — new Step 4 (renumber existing steps)
step(4, TOTAL_STEPS, "API Keys & Services");

const API_KEYS = [
  { key: "ANTHROPIC_API_KEY", name: "Anthropic (Claude)", required: true,
    hint: "Get one at https://console.anthropic.com/settings/keys" },
  { key: "GITHUB_TOKEN", name: "GitHub", required: false,
    hint: "For PR review, repo skills. Create at https://github.com/settings/tokens" },
  { key: "OPENAI_API_KEY", name: "OpenAI (Codex researcher)", required: false,
    hint: "https://platform.openai.com/api-keys" },
  { key: "GEMINI_API_KEY", name: "Google Gemini", required: false,
    hint: "https://aistudio.google.com/apikey" },
  { key: "XAI_API_KEY", name: "xAI Grok", required: false },
  { key: "PERPLEXITY_API_KEY", name: "Perplexity", required: false },
  { key: "DEEPSEEK_API_KEY", name: "DeepSeek", required: false },
  { key: "MISTRAL_API_KEY", name: "Mistral", required: false },
];

// For each key:
// 1. Check if already in environment → show "✅ Already set"
// 2. If not set and required → prompt with hint URL
// 3. If not set and optional → "Skip? (y/N)"
// 4. Collect values → write to shell profile OR .env file

// User chooses destination:
// - Option A: Append to ~/.zshrc (persistent across all terminals)
// - Option B: Write to ~/.claude/.env (loaded by KAI hooks only)
```

**Key decisions:**
- Required keys (ANTHROPIC_API_KEY) get prompted unconditionally
- Optional keys shown as a checklist: "Which would you like to configure? [space to select]"
- Keys are never stored in tracked files — only in shell profile or gitignored .env
- If using Bedrock (already configured in Step 4-old), skip ANTHROPIC_API_KEY prompt
- **Deduplication guard:** Before appending `export KEY=...` to shell profile, grep for existing `export KEY=` line. If found, sed-replace the value instead of appending. Prevents accumulation on repeated installs.
- **Precedence (env var resolution order):** Shell profile (sourced into all terminals) takes priority over `.env` file. Hooks that load `.env` must skip keys already present in `process.env`. Document this order in the .env file header comment.

**Files:**

| File | Action |
|------|--------|
| `PAI-Install/main.ts` | Update — add API key step |
| `PAI-Install/lib/api-keys.ts` | NEW — key detection, dedup-aware shell profile writer, .env loader with precedence |

**Validation:**
- Installer detects pre-existing keys and shows them as "already set"
- Keys written to ~/.zshrc are properly quoted and exported
- Running the installer again updates (not duplicates) existing key exports
- A key in shell profile is NOT overridden by the same key in .env
- Bedrock users are not prompted for ANTHROPIC_API_KEY

### [3] Installer: Notification Setup

**What:** Optional step that configures notification channels (ntfy, Discord) by collecting topic/webhook values and enabling routing.

**Why:** Notifications config exists with env var placeholders but no user ever fills them in. The feature is invisible.

**Implementation:**

```typescript
step(N, TOTAL_STEPS, "Notifications (optional)");

info("KAI can send push notifications for long tasks, errors, and security events.");
const wantsNotifications = await confirm("Configure notifications?", false);

if (wantsNotifications) {
  // ntfy (simplest — just a topic name)
  const ntfyTopic = await prompt("ntfy.sh topic name (free push notifications)", "");
  if (ntfyTopic) {
    writeKeyToDestination("NTFY_TOPIC", ntfyTopic); // uses api-keys.ts dedup writer
    enableNotificationChannel(paiDir, "ntfy", ["longTask", "error", "backgroundAgent"]);
  }

  // Discord (webhook URL)
  const discordWebhook = await prompt("Discord webhook URL (optional)", "");
  if (discordWebhook) {
    writeKeyToDestination("DISCORD_WEBHOOK", discordWebhook);
    enableNotificationChannel(paiDir, "discord", ["error", "security"]);
  }
}

// enableNotificationChannel() implementation:
// 1. Read notifications.jsonc as text
// 2. Parse JSONC → find routing.[eventType] arrays
// 3. For each event in enabledEvents: push channelName if not already present
// 4. Write back with comment preservation (regex-based splice, not full reserialize)
//
// This is the hardest part of this feature — JSONC modification without losing comments.
// Approach: targeted regex replacement on the specific routing lines rather than
// full parse/reserialize. Each routing line is a simple JSON array on one line.
// Pattern: /"eventType": \[([^\]]*)\]/ → append channel to the array.
```

**Files:**

| File | Action |

| `PAI-Install/lib/notifications-setup.ts` | NEW — notification channel config |

**Validation:**

### [4] Fresh Install: Local Overrides File

**What:** Always create `config/preferences.local.jsonc` during fresh install (not just upgrades), with explanatory comments.

**Why:** Currently only created during migration from existing settings. Fresh users don't know this pattern exists for machine-specific customization.

**Implementation:**

```typescript
// In createUserScaffold() or separate function
function ensureLocalPreferences(paiDir: string) {
  const path = join(paiDir, "config", "preferences.local.jsonc");
  if (existsSync(path)) return; // Don't overwrite existing

  const rebuildCmd = `bun ${paiDir}/hooks/handlers/BuildSettings.ts`;
  const content = `// Machine-Specific Overrides
// This file is gitignored — it stays on this machine only.
// Values here merge ON TOP of config/preferences.jsonc at build time.
//
// Use this for:
//   - Extra environment variables specific to this machine
//   - MCP servers only available on this machine
//   - API keys you don't want in your shell profile
//   - Dev server port overrides
//
// After editing, rebuild settings:
//   ${rebuildCmd}
//
// Examples:
//   {
//     "env": {
//       "MY_CUSTOM_VAR": "value",
//       "DEV_API_URL": "http://localhost:8080"
//     },
//     "mcpServers": {
//       "my-server": { "url": "http://localhost:3000/mcp" }
//     }
//   }

{}
`;
  writeFileSync(path, content, "utf-8");
}
```

### [5] Fresh Install: .env Bootstrap

**What:** During install, copy `.env.example` to `.env` if it doesn't already exist. Tell the user it's there and what to do with it.

**Why:** QUICKSTART references `.env.example` but nobody creates a `.env` from it. The installer should do this automatically.

**Implementation:**

```typescript
// After API key wizard (Step 4)
const envExample = join(paiDir, ".env.example");
const envFile = join(paiDir, ".env");

if (existsSync(envExample) && !existsSync(envFile)) {
  // Copy example to .env
  copyFileSync(envExample, envFile);
  info("Created .env from .env.example");
  info("Keys you configured in Step 4 are in your shell profile — they're already active.");
  info(".env is for ADDITIONAL keys you add later (alternative to editing your shell profile).");
  info("Shell profile keys always take precedence over .env values.");
}
```

**Key decision:** The .env file is gitignored. It's a convenience for users who prefer file-based secrets over shell profile exports. The API key wizard (Step 4) writes to the shell profile by default — .env is the alternative path for future keys.

**Precedence (documented in .env header):** `process.env` (from shell profile) > `.env` file. The .env loader (in hooks/lib/) must skip any key already present in process.env. This means Step 4 keys are authoritative and .env never silently overrides them.

**Files:**

| File | Action |
|------|--------|
| `PAI-Install/main.ts` | Update — copy .env.example |

**Validation:**
- .env created on fresh install if .env.example exists
- .env NOT overwritten if it already exists (upgrade case)
- Both .env and shell profile are valid key sources (hooks check both)

### [7] Guide: How Memory Works (`docs/MEMORY.md`)

**What:** A user-facing guide explaining the MEMORY/ directory structure, what auto-populates, retention policies, and what the user should/shouldn't touch.

**Content outline:**

```markdown
# How Memory Works

## Overview
KAI builds up knowledge about you and your work over time. This happens
automatically — you don't need to do anything.

## Directory Structure
| Directory | What's Stored | Auto-populated? | Retention |
|-----------|--------------|-----------------|-----------|
| KNOWLEDGE/ | Domain expertise distilled from sessions | Yes (KnowledgeSync hook) | Permanent (committed) |
| WISDOM/FRAMES/ | Curated lessons and insights | Manual curation | Permanent (committed) |
| STATE/ | Session metadata, work tracking | Yes | 30 days (auto-cleaned) |
| WORK/ | Active PRDs and work sessions | Yes | Until completed |
| LEARNING/ | Raw failure captures, reflections | Yes | Archived monthly |
| RELATIONSHIP/ | Interaction patterns with user | Yes | Permanent |
| CAPABILITIES/ | Learned skills and patterns | Yes | Permanent |
| STAGING/ | Draft insights awaiting curation | Yes (ReflectionHarvester) | 14 days |
| RESEARCH/ | Research session outputs | Yes | Per-session |
| SECURITY/ | Security scan results | Yes | Per-session |

## What You Should Know
- KNOWLEDGE/ and WISDOM/ are committed to git — review before committing
- If your sessions involve sensitive context (internal project names, colleague names,
  proprietary systems), review KNOWLEDGE/ entries before pushing. KnowledgeSync distills
  technical patterns, but domain keywords can leak context. Run `git diff MEMORY/KNOWLEDGE/`
  before committing to verify.
- Everything else is gitignored (machine-local, may contain PII)
- You never need to manually create files in these directories
- On fresh install, directories exist but are empty — they fill over ~5-10 sessions
- The `SessionCleanup` hook enforces retention automatically

## Customization
- To add domain knowledge manually: edit MEMORY/KNOWLEDGE/{domain}.md
- To add a wisdom frame: create MEMORY/WISDOM/FRAMES/{name}.md
- To change retention: edit `memory.retention` in config/preferences.jsonc

## Troubleshooting
- "My knowledge isn't growing": KnowledgeSync fires on session end — complete sessions
- "State/ is huge": SessionCleanup runs daily — if not, run manually: ...
```

### [8] Guide: Local Overrides (`docs/CONFIGURATION.md`)

**What:** Comprehensive guide to the configuration system — domain files, build pipeline, local overrides, and when to use each.

**Content outline:**

```markdown
# Configuration Guide

## Architecture
KAI uses domain-specific config files that merge into a single settings.json:

  config/identity.jsonc       → Who you are, who KAI is
  config/preferences.jsonc    → Env vars, tech stack, runtime settings
  config/hooks.jsonc          → Hook registrations
  config/permissions.jsonc    → Tool allow/deny/ask rules
  config/notifications.jsonc  → Alert channels and routing
  config/preferences.local.jsonc → Machine-specific overrides (gitignored)

## The Build Pipeline
  config/*.jsonc → BuildSettings.ts → settings.json (generated, gitignored)

## When to Use Each File
| I want to... | Edit this file |
|---|---|
| Change my name or timezone | config/identity.jsonc |
| Add an environment variable | config/preferences.jsonc (global) or preferences.local.jsonc (this machine) |
| Add an MCP server | config/preferences.local.jsonc |
| Allow a new tool | config/permissions.jsonc |
| Set up notifications | config/notifications.jsonc |
| Override something just on this machine | config/preferences.local.jsonc |

## preferences.local.jsonc
This file is for anything machine-specific that shouldn't be in git:
- MCP servers only available on this machine
- API keys (alternative to shell profile)
- Dev server ports
- Project paths that differ between machines

It merges ON TOP of preferences.jsonc — same keys override, new keys are added.

## After Any Config Change
  bun ~/.claude/hooks/handlers/BuildSettings.ts
(Also runs automatically at session start if config files changed.)

## Never Edit Directly
- settings.json — generated, overwritten on rebuild
- CLAUDE.md — generated from CLAUDE.md.template
```

### [9] Guide: MCP Server Configuration (`docs/MCP-GUIDE.md`)

**What:** Guide covering how to add MCP servers, the three config mechanisms, OAuth flows, and troubleshooting.

**Content outline:**

```markdown
# MCP Server Guide

## What Are MCP Servers?
Model Context Protocol servers give KAI access to external tools — databases,
APIs, browsers, infrastructure. They're like plugins that expose new tool calls.

## Adding a Server

### Remote (HTTP) servers
Add to config/preferences.local.jsonc:
  "mcpServers": {
    "my-server": { "url": "https://example.com/mcp" }
  }

### Local (stdio) servers
  "mcpServers": {
    "my-server": {
      "command": "npx",
      "args": ["@some/mcp-server"],
      "env": { "API_KEY": "${MY_API_KEY}" }
    }
  }

## Available Servers
| Server | Install | What It Does |
|--------|---------|-------------|
| Cloudflare | Remote (https://mcp.cloudflare.com/mcp) | Workers, KV, R2, DNS |
| GitHub | `npx @anthropic-ai/mcp-github` | PRs, issues, search |
| Playwright | `npx @anthropic-ai/mcp-playwright` | Browser automation |
| SQLite | `npx @anthropic-ai/mcp-sqlite` | Database queries |
| Filesystem | `npx @anthropic-ai/mcp-filesystem` | Sandboxed file access |

NOTE: Verify install commands against each server's README at implementation time.
npx-based servers are self-installing. Some servers (GitHub) may have alternative
install methods (Docker, binary download) — document the npx path as primary since
it requires no global install.

## OAuth Flows (Cloudflare, etc.)
1. Add the server URL to config
2. Rebuild: bun ~/.claude/hooks/handlers/BuildSettings.ts
3. Start a session — Claude Code will prompt for OAuth
4. Complete the flow in your browser
5. Auth token cached in mcp-needs-auth-cache.json (gitignored)

## Troubleshooting
- "Server connected but no tools": Check `claude mcp list`
- "Connection refused": Verify the server process is running
- "Auth expired": Delete `~/.claude/mcp-needs-auth-cache.json`, restart session
  (this file is at the PAI_DIR root, gitignored, auto-recreated on next OAuth flow)
- "Env var not found": Add to preferences.local.jsonc env block

## Project-Specific Servers
Add a .mcp.json to your project root. Enable in preferences.jsonc:
  "enableAllProjectMcpServers": true
```

### [13] First-Session Onboarding Hook

**What:** A one-time SessionStart hook that detects "this is the user's first session after install" and injects a brief orientation into context — what to try, where to get help, how to customize.

**Why:** After install, the user starts `claude` and gets a greeting banner but no guidance on what KAI can actually do. The first session is the highest-leverage moment for showing value. A one-time orientation bridges the gap between "installed" and "productive."

**Implementation:**

```typescript
// hooks/FirstSessionOnboarding.hook.ts (SessionStart, sync)
import { existsSync, writeFileSync } from "fs";
import { join } from "path";

const PAI_DIR = process.env.PAI_DIR!;
const FLAG_FILE = join(PAI_DIR, "MEMORY", "STATE", ".onboarding-complete");

// Only fire once — ever
if (existsSync(FLAG_FILE)) {
  process.exit(0);
}

// Mark as complete so this never fires again
writeFileSync(FLAG_FILE, new Date().toISOString(), "utf-8");

// Inject orientation context
const orientation = `
Welcome to KAI — your first session.

Quick orientation:
• Type normally — KAI handles simple and complex tasks automatically
• /help — see all available skills and commands
• /research <topic> — multi-agent parallel research
• /end — session wrap-up with memory save

Customize anytime:
• Edit ~/.claude/PAI/USER/ABOUTME.md to tell KAI about yourself
• Reconfigure: bun ~/.claude/scripts/kai-setup.ts
• Verify install: bun ~/.claude/scripts/kai-doctor.ts

This message appears once. KAI is ready.
`.trim();

console.log(JSON.stringify({ additionalContext: orientation }));
```

**Key decisions:**
- Fires exactly ONCE (flag file in gitignored MEMORY/STATE/)
- Short enough to not overwhelm (~10 lines of orientation)
- Actionable — every bullet is something the user can do right now
- References actual script paths (`bun ~/.claude/scripts/kai-setup.ts`) not aliases, since v5.9.2 aliases may not be installed yet. If the alias exists, mention it parenthetically.
- Sync: the hook syncs to KAI; the flag file is machine-local
- **Hook output format:** SessionStart hooks support `additionalContext` in JSON stdout (verified: StartupGreeting.hook.ts, LoadContext.hook.ts already use this pattern). If Claude Code changes the contract, this hook fails silently (no output = no error).

**Files:**

| File | Action |
|------|--------|
| `hooks/FirstSessionOnboarding.hook.ts` | NEW — one-time orientation hook |
| `config/hooks.jsonc` | Update — register at SessionStart (sync, after StartupGreeting) |

**Validation:**
- First session after install: orientation appears in context
- Second session: hook exits immediately (flag file exists)
- Deleting flag file re-triggers orientation (useful for testing)
- Content is generic (no PII, works for any KAI user)
- Hook adds <50ms latency (just a file existence check)
