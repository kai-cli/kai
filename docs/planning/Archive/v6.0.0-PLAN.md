## Feature Details

### [1] Input Classification (Unified Prompt)

**What:** Replace the current prefix-based mode routing (ModeClassifier hook) with a probabilistic classifier that determines whether user input is a shell command, an AI query, or a PAI skill invocation — enabling a single unified input box.

**Why:** Currently, users type in Claude Code's prompt and the ModeClassifier hook classifies into NATIVE/ALGORITHM/MINIMAL based on task complexity. But the user still has to think about whether they're talking to the shell or to AI. A unified classifier would route `git status` to shell and "fix the auth bug" to AI seamlessly.

**Inspired by:** Warp's heuristic + ONNX model with `p_shell` / `p_ai` probability scores.

**Architectural constraint (critical):** Claude Code sends all input to the model first. There is no pre-routing hook that intercepts input BEFORE it reaches Claude. This means:
- **Path A (hook-based hint):** UserPromptSubmit hook classifies input and injects `additionalContext` that strongly hints "this is a shell command, execute it directly." Claude still receives and processes the input — the hint improves behavior but cannot guarantee shell-only execution.
- **Path B (Claude Code pre-route API):** If Claude Code adds a hook event that fires BEFORE model inference and can redirect input, implement proper routing there.

**v6.0 ships Path A.** The UX improvement is real but bounded: classified shell commands get executed faster with less "let me help you with that" preamble, but Claude still processes every input. Path B is tracked as a Claude Code feature request.

**User feedback loop:** When input is auto-classified as shell and executed:
- Success output is shown normally (same as `!command` today)
- If the command fails or produces unexpected output, Claude offers to help interpret
- Users can always override: `!command` forces shell, no prefix forces AI (existing behavior preserved)

**Implementation:**

```typescript
// hooks/lib/input-classifier.ts
interface ClassificationResult {
  p_shell: number;     // Probability: this is a shell command (0-1)
  p_ai: number;        // Probability: this is an AI query (0-1)
  p_skill: number;     // Probability: this is a skill invocation (0-1)
  classification: "shell" | "ai" | "skill" | "ambiguous";
}
```

**Classification layers:**

1. **Deterministic layer** (fast, no ambiguity):
   - Starts with `/` → skill invocation (`p_skill: 1.0`)
   - Starts with `!` → shell command (`p_shell: 1.0`, existing Claude Code convention)
   - These bypass all heuristics — explicit prefixes always win

2. **Heuristic layer** (word + syntax analysis):
   - First token is a known binary (scanned from PATH on first use, cached to `/tmp/pai-hooks/path-cache.json`) → `p_shell += 0.6`
   - High shell metacharacter density (`|`, `>`, `$`, `&&`, `;`) → `p_shell += 0.2` per token
   - English word ratio > 0.7 (using a word list, not NLP) → `p_ai += 0.4`
   - Contains question words ("what", "how", "why", "can you") → `p_ai += 0.3`
   - Multi-word sentence structure without flags/pipes → `p_ai += 0.2`

3. **Threshold logic:**
   - If `p_shell > 0.85` → classify as shell, inject hint
   - If `p_ai > 0.85` → classify as AI, proceed normally (ModeClassifier still handles NATIVE/ALGORITHM/MINIMAL)
   - Otherwise → classify as AI (safe default — user can always prefix with `!`)

**Files:**

| File | Action |
|------|--------|
| `hooks/lib/input-classifier.ts` | NEW — classification engine |
| `hooks/lib/command-database.ts` | NEW — PATH scan + caching |
| `hooks/ModeClassifier.hook.ts` | REWRITE — integrate classifier as first stage |
| `tests/input-classifier.test.ts` | NEW — extensive test suite |

**Validation:**
- `git status` → classified as shell with p_shell > 0.9
- `fix the authentication bug in auth.ts` → classified as AI with p_ai > 0.9
- `/research Warp terminal` → classified as skill with p_skill = 1.0
- `npm install express` → classified as shell
- `what does this error mean` → classified as AI
- Ambiguous single words (`docker`) → default to AI (safe; user can `!docker` for shell)
- Existing `!` prefix behavior completely unchanged
- ModeClassifier still handles NATIVE/ALGORITHM/MINIMAL for AI-classified input

### [2] Multi-Harness Agent Orchestration

**What:** A typed `Harness` abstraction that lets PAI orchestrate multiple AI backends (Claude Code, OpenAI Codex, Google Gemini, local models) through a unified interface, with both local and remote execution modes.

**Why:** PAI currently only orchestrates Claude via the Agent tool. As AI models specialize (Codex for code generation, Gemini for multimodal, local models for privacy), PAI should be able to dispatch to the right model for the job — not just the one it's embedded in.

**Inspired by:** Warp's `Harness` trait with `ClaudeCode`, `Codex`, `Gemini`, `OpenCode` implementations and `OrchestrationConfig` supporting local/remote modes.

**Implementation:**

```typescript
// agents/harnesses/harness.ts
interface Harness {
  name: string;
  
  /** Check if this harness is available (API key set, binary installed, etc.) */
  isAvailable(): Promise<boolean>;
  
  /** Execute a single prompt and return the response */
  execute(opts: {
    prompt: string;
    systemPrompt?: string;
    tools?: ToolDefinition[];
    maxTokens?: number;
    temperature?: number;
  }): Promise<HarnessResponse>;
  
  /** Execute a multi-turn conversation (streaming) */
  chat(messages: Message[]): AsyncGenerator<string>;
  
  /** Get static capabilities declaration for this harness */
  capabilities(): HarnessCapabilities;
}

interface HarnessCapabilities {
  supportsTools: boolean;
  supportsStreaming: boolean;
  supportsImages: boolean;
  maxContextWindow: number;
}

// Execution configuration per-task
interface OrchestrationConfig {
  harness: string;           // "claude" | "codex" | "gemini" | "local"
  executionMode: "local" | "remote";
  model?: string;            // Model ID override within the harness
  fallback?: string;         // Harness to use if primary fails/unavailable
}
```

**Design decisions:**

1. **No `costTier` in interface.** Pricing changes constantly — hardcoding tiers in TypeScript creates stale routing decisions. Cost awareness is handled via external config (`agents/harnesses/config.json`) that maps harness+model to a cost bucket, updatable without code changes.

2. **TaskDescription construction (the hard problem).** The router needs to know what kind of task it's looking at. Rather than requiring a fully-typed `TaskDescription` to be constructed before routing, the router accepts a lightweight signal bag:

```typescript
// agents/harnesses/router.ts
interface RoutingSignals {
  hasImages?: boolean;         // Detected from prompt content (base64, file refs)
  privacyLevel?: "public" | "internal" | "sensitive";  // From project config
  taskType?: "code-gen" | "reasoning" | "research" | "multimodal" | "general";
  preferredHarness?: string;   // User override from skill/workflow config
  maxBudget?: "low" | "medium" | "high";  // From task effort level
}

function selectHarness(signals: RoutingSignals): OrchestrationConfig {
  // Explicit preference always wins
  if (signals.preferredHarness) return { harness: signals.preferredHarness, ... };
  
  // Privacy constraint
  if (signals.privacyLevel === "sensitive") return { harness: "local", ... };
  
  // Capability match
  if (signals.hasImages) return { harness: "gemini", ... };
  
  // Default
  return { harness: "claude", executionMode: "local" };
}
```

Signal construction is lightweight: `hasImages` is a regex check for base64/file patterns, `privacyLevel` comes from project config, `taskType` is inferred from the skill/workflow that triggered the agent. The router doesn't need to deeply understand the task — it routes on surface signals and lets the harness handle complexity.

3. **Tool format translation.** Each harness handles its own tool_use format internally:
   - Claude: native Anthropic tool_use format
   - Codex/GPT: OpenAI function_calling format
   - Gemini: Google function_declarations format
   - Local: depends on model (Ollama supports tools for some models)
   
   The harness interface presents a unified `ToolDefinition` to callers; the concrete harness translates to/from its provider's format.

**Concrete harnesses (MVP):**

| Harness | Backend | Availability Check | Primary Use |
|---------|---------|-------------------|-------------|
| `claude` | Anthropic API (direct) | `ANTHROPIC_API_KEY` env var | Default for complex reasoning |
| `claude-code` | Claude Code CLI (embedded) | Always available (we're inside CC) | Agent spawning (current behavior) |
| `codex` | OpenAI API | `OPENAI_API_KEY` env var | Code generation tasks |
| `gemini` | Google Gemini API | `GOOGLE_AI_KEY` env var | Multimodal, large context |
| `local` | Ollama HTTP | `curl localhost:11434/api/tags` succeeds | Privacy-sensitive, offline |

| `agents/harnesses/claude.ts` | NEW — Anthropic API harness |

### [3] Workflow Templates ("Light Skills")

**What:** A new automation primitive — YAML parameterized command templates that are simpler than full skills but richer than shell aliases. Invoked with argument substitution, searchable from the Board.

**Why:** Full skills (SKILL.md + Workflows/ + Tools/) are overkill for simple parameterized commands. Many automation needs are just "run this command with these arguments" — firmware builds, test runs, deploy scripts. Workflow templates fill the gap between raw shell and full skill orchestration.

**Inspired by:** Warp's `.warp/workflows/` YAML templates with parameter interpolation.

**The three skill tiers after v6.0:**

| Tier | Format | Complexity | Use Case |
|------|--------|-----------|----------|
| **Workflow** | Single YAML file | Parameterized command | "Run this with these args" |
| **Skill** | SKILL.md + Workflows/ | Multi-step orchestration | Complex domain operations |
| **Agent** | Full agent personality | Autonomous work | Independent workstreams |

**Implementation:**

```yaml
# workflows/firmware-build.yaml
name: firmware-build
description: "Build firmware for a specific target board"
command: |
  cd ~/Projects/feed-bbf && \
  make defconfig BOARD={{board}} && \
  make -j$(nproc)
arguments:
  - name: board
    description: "Target board (e.g., m62, m80, m33)"
    default: m62
    options: [m62, m80, m33, m6x-eu]
tags: [firmware, build, openwrt]
```

```yaml
# workflows/jenkins-trigger.yaml
name: jenkins-trigger
description: "Trigger a Jenkins build for a branch"
command: |
  bun ~/.claude/skills/Utilities/Tools/JenkinsTrigger.ts \
    --branch {{branch}} \
    --target {{target}}
arguments:
  - name: branch
    description: "Git branch to build"
    required: true
  - name: target
    description: "Build target"
    default: "m62-dev"
tags: [jenkins, ci, build]
```

**Template syntax:** `{{variable}}` — double-brace interpolation. This does NOT conflict with:
- Hook system: uses environment variables (`$VAR` / `${VAR}`)
- Skill system: uses YAML frontmatter fields (no interpolation in content)
- Shell: `${var}` and `$(cmd)` — double-brace is never valid shell syntax
- GitHub Actions: uses `${{ }}` (dollar prefix differentiates)

**Directory structure:**
```
workflows/                    # System-level templates
├── firmware-build.yaml
├── jenkins-trigger.yaml
└── mcp-doctor.yaml

~/.claude/projects/{project}/workflows/   # Project-level templates
├── run-tests.yaml
└── deploy.yaml
```

**Security model:**
- System workflows (`~/.claude/workflows/`) are always trusted (user-owned)
- Project workflows (`~/.claude/projects/{project}/workflows/`) are trusted (user explicitly created them for that project)
- **Workflows from cloned repos are NOT auto-trusted.** If a `.workflows/` or `workflows/` directory exists in a cloned repo root, it is NOT auto-discovered. Only workflows in PAI's managed directories are executed. This prevents supply-chain attacks via repo-bundled workflows.
- To use a repo's workflows: user must explicitly copy them into the project's PAI workflow directory (conscious trust decision)

**Resolution order:** Project workflows shadow system workflows (same name → project wins).

**Invocation:**
- From Claude: "run the firmware-build workflow for m80" (natural language)
- From command palette (Board): search workflows, fill params interactively
- Via CLI: `bun scripts/workflow-run.ts firmware-build --board m80`
- Via shell: `bun scripts/workflow-run.ts --list` (enumerate available workflows)

**Files:**

| File | Action |
|------|--------|
| `scripts/workflow-run.ts` | NEW — CLI runner with arg substitution |
| `workflows/` | NEW directory — system templates |
| `PAI/SKILLSYSTEM.md` | Update — document workflow tier |
| `tests/workflow-run.test.ts` | NEW — template parsing + substitution tests |

**Validation:**
- `workflow-run.ts firmware-build --board m62` executes the substituted command
- Missing required argument → clear error with usage help
- `--help` on any workflow shows description + arguments + options
- `--list` shows all available workflows with descriptions
- Workflows appear in Board command palette search results
- Project workflows override system workflows by name
- `{{variable}}` in command is substituted; literal `${shell_var}` passes through unchanged
- Workflow file in a git-cloned repo root → NOT auto-discovered (security boundary)

### [4] Spec-Driven Development

**What:** A `specs/` directory pattern where product and technical specs are written BEFORE implementation, with dedicated workflows that read specs and generate implementation plans + ISC criteria automatically.

**Why:** PAI's Algorithm already uses ISC (Ideal State Criteria) for every task. But criteria are generated ad-hoc from user requests. Specs formalize the input: when a feature has a written spec, the Algorithm can derive higher-quality criteria from it. This closes the loop: spec → criteria → implementation → verification against spec.

**Inspired by:** Warp's `specs/` directory with product/tech specs indexed by ticket ID, plus `implement-specs` and `write-tech-spec` skills.

**Implementation:**

```markdown
# specs/v6.0-input-classification.md
---
id: PAI-601
type: feature
status: draft
priority: high
---

## Problem
[What problem does this solve?]

## Requirements
- [ ] REQ-1: Classify shell commands with >90% accuracy
- [ ] REQ-2: Classify AI queries with >90% accuracy
- [ ] REQ-3: Ambiguous input defaults to AI (safe)

## Design
[Technical approach, key decisions, constraints]

## Files Affected
[Specific file paths and what changes]

## Acceptance Criteria
[Maps to ISC criteria the Algorithm will use]
```

**Status transitions:**

| Transition | Triggered By | Automation |
|-----------|-------------|-----------|
| `draft` → `approved` | User explicitly approves | Manual (user sets status) |
| `approved` → `implementing` | `ImplementSpec` workflow starts | **Automatic** — ImplementSpec updates frontmatter |
| `implementing` → `shipped` | All ISC criteria pass in Algorithm VERIFY | **Automatic** — Algorithm LEARN phase updates spec |

Only `draft → approved` is manual (requires human judgment). The rest are automated by the workflows.

**Algorithm integration:**

The Algorithm's OBSERVE phase gains a new pre-flight check:

```
- **Spec lookup (when task references a spec or matches a spec title):** Read `specs/` directory
  for relevant specs. If a matching spec exists with status `approved`:
  - Import requirements as ISC criteria (REQ-N → ISC-N)
  - Import acceptance criteria as additional ISC
  - Set spec status to `implementing`
  - Reference spec in PRD Context section
```

This is a behavioral addition to the Algorithm prompt, not a code change to a hook or tool. It modifies `PAI/Algorithm/v3.13.0.md` (or its successor version).

**Two new workflows (in Development skill):**

1. **WriteSpec** — guides spec authoring with structure enforcement, validates required sections
2. **ImplementSpec** — reads a spec, generates PRD with ISC criteria from requirements, transitions status to `implementing`, starts Algorithm

**Files:**

| File | Action |
|------|--------|
| `specs/` | NEW directory (starts empty, populated as specs are written) |
| `skills/Development/Workflows/WriteSpec.md` | NEW — spec authoring workflow |
| `skills/Development/Workflows/ImplementSpec.md` | NEW — spec-to-implementation workflow |
| `PAI/Algorithm/v{next}.md` | Update — add spec lookup to OBSERVE pre-flight |

**Validation:**
- `WriteSpec` produces well-structured spec with all required sections
- `ImplementSpec` reads spec → generates ISC criteria covering all REQ-N items
- Spec status transitions automatically: approved → implementing (on ImplementSpec) → shipped (on Algorithm completion)
- Algorithm OBSERVE pre-flight finds relevant specs when task matches
- Spec with status `draft` is NOT auto-imported (must be `approved` first)

### [5] PAI→KAI Internal Rename

**What:** Complete the internal brand consolidation that was deferred from v5.x. All internal references to "PAI" become "KAI" for the public distribution. The private repo retains "PAI" as the development name.

**Why:** Currently PAI uses "Option C" (user-facing KAI, internal PAI/). v6.0 is the right time to fully consolidate because: (a) it's a major version with breaking changes anyway, (b) the skills-lock from v5.9 makes it easier to verify completeness, (c) all new v6.0 features can be built with the final naming from the start.

**Scope of rename:**

| Layer | Current | v6.0 |
|-------|---------|------|
| Public package name | KAI | KAI (unchanged) |
| Public docs/README | KAI | KAI (unchanged) |
| Internal directory | `PAI/` | `KAI/` (in kai repo only) |
| Internal references | Mixed PAI/KAI | KAI in public, PAI in private |
| Algorithm references | "PAI Algorithm" | "The Algorithm" (brand-neutral) |
| Hook references | `PAI_DIR` env var | `KAI_DIR` in kai, `PAI_DIR` in kai |
| Skill descriptions | "PAI system" | "system" (brand-neutral) |

**Rename categories (each needs different handling):**

| Category | Examples | Approach |
|----------|----------|----------|
| Env vars in code | `process.env.PAI_DIR` | Simple find/replace |
| Env vars in shell conditionals | `if [ -n "$PAI_DIR" ]` | Regex-aware replace |
| Comments and documentation | `# PAI system`, `/// PAI hook` | Bulk replace, allow residual in CHANGELOG |
| CHANGELOG / git history | Prior release notes | **Do not touch** — historical record stays as-is |
| Binary/script names | `pai-curate`, `pai-board` | Rename files + update all call sites |
| Test fixtures | Strings in test assertions | Update to match new names |
| Generated artifacts | `settings-schema.json` references | Re-generate after rename |

**Case variants to sweep:**
- `PAI` (uppercase)
- `pai` (lowercase)
- `Pai` (title case)
- `PAI_DIR`, `PAI_HOME` (env vars)
- `kai` (repo name — stays as-is in private repo)

**Dry-run mode for sync-to-kai.sh:**
```bash
# New flag: --dry-run outputs diff without applying
./scripts/sync-to-kai.sh --dry-run > rename-preview.diff
# Review the diff, then:
./scripts/sync-to-kai.sh
```

**Implementation approach:**
- Add `--dry-run` flag to `sync-to-kai.sh` that outputs a unified diff of all transforms without writing
- Human reviews the diff before committing
- skills-lock.json `verify` confirms no PAI references leaked into KAI output
- Explicit allowlist for acceptable residuals (CHANGELOG, attribution lines, `Co-Authored-By` in git history)

**Files:**

| File | Action |
|------|--------|
| `scripts/sync-to-kai.sh` | Update — complete brand transforms + `--dry-run` flag |
| `skills-lock.json` | Verify — no PAI references in KAI output post-rename |

**Validation:**
- `sync-to-kai.sh --dry-run` produces a reviewable diff
- `grep -ri "PAI" ~/Projects/kai/` returns ONLY allowlisted hits (CHANGELOG, attribution)
- `grep -ri "pai_dir" ~/Projects/kai/` returns zero (env var fully renamed to KAI_DIR)
- All KAI tests pass after rename
- skills-lock.json verifies integrity post-rename
- sync-to-kai.sh is idempotent (running twice produces same output)
- Case variants covered: PAI, pai, Pai, PAI_DIR, PAI_HOME all transformed
