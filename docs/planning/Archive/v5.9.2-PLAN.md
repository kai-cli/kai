# PAI v5.9.2 — Re-runnable Setup & Self-Service Configuration

**Created:** 2026-05-22
**Theme:** Post-install self-service — users can add, change, and verify configuration anytime
**Prerequisite:** v5.9.1 shipped (guides, templates, installer enhancements)
**Bridges to:** v6.0.0 (workflow templates can wrap these as `kai setup` shortcuts)

---

## Overview

v5.9.1 makes the first-run experience comprehensive. v5.9.2 makes configuration a continuous, self-service activity. Users should never need to read source code or reverse-engineer config files — they run `kai setup` or ask KAI directly.

```
[1] kai-setup Command          → Re-runnable interactive configurator
[2] kai-doctor Command         → Verify installation health + report
[3] kai-keys Command           → Manage API keys (add/list/verify)
[4] Config Validation Hook     → Catch invalid config before it breaks
[5] Guided MCP Onboarding Skill → In-session MCP setup via conversation
[6] Environment Verification   → Startup hook reports missing optional deps
[7] kai-reset Command          → Rebuild config from source; --hard nukes runtime state
[8] kai-upgrade Command        → Pull latest + rebuild in one step
[9] Hook Integration Tests     → Priority hook coverage for v6.0 readiness
[10] Release Audit Script      → Automated PII/brand/sync checks for releases
```

**Naming convention:** All CLI commands use hyphenated names (`kai-setup`, `kai-doctor`, etc.)
for shell aliases. In prose and user-facing docs, we write `kai setup` (space) as the
natural-language invocation. The alias maps the hyphenated form; both work because the alias
definition is: `alias kai-setup="bun ~/.claude/scripts/kai-setup.ts"`. Users can also invoke
directly with `bun ~/.claude/scripts/kai-setup.ts` if aliases aren't installed.

### [1] `kai setup` — Re-runnable Interactive Configurator

**What:** A post-install command that lets users reconfigure any aspect of KAI without re-running the full installer. Menu-driven with specific subcommands.

**Why:** The installer runs once. But users want to add API keys weeks later, change their timezone, add an MCP server, or reconfigure notifications. Today this requires knowing which JSONC file to edit and running BuildSettings manually.

**Implementation:**

```typescript
// scripts/kai-setup.ts
// Usage:
//   bun scripts/kai-setup.ts              # Interactive menu
//   bun scripts/kai-setup.ts identity     # Jump to identity config
//   bun scripts/kai-setup.ts keys         # Jump to API keys
//   bun scripts/kai-setup.ts mcp          # Jump to MCP servers
//   bun scripts/kai-setup.ts notifications # Jump to notifications
//   bun scripts/kai-setup.ts bedrock      # Jump to AWS Bedrock

const MENU = [
  { key: "1", label: "Identity", desc: "Change your name, DA name, timezone" },
  { key: "2", label: "API Keys", desc: "Add or update service credentials" },
  { key: "3", label: "MCP Servers", desc: "Add, remove, or test MCP connections" },
  { key: "4", label: "Notifications", desc: "Configure alert channels" },
  { key: "5", label: "AWS Bedrock", desc: "Enable/disable Bedrock routing" },
  { key: "6", label: "Preferences", desc: "Tech stack, dev server, timeouts" },
  { key: "7", label: "Verify All", desc: "Run kai doctor" },
];
```

**Each submenu reuses installer modules** from v5.9.1:
- Identity → `writeIdentityConfig()` from PAI-Install/main.ts
- API Keys → `PAI-Install/lib/api-keys.ts`
- MCP → `PAI-Install/lib/mcp-setup.ts`
- Notifications → `PAI-Install/lib/notifications-setup.ts`

**v5.9.1 module refactor requirement:** v5.9.1 Features [1-3] MUST export their logic as
importable functions (not just as top-level script execution). This means each `lib/*.ts` file
should have a `main()` function guarded by `if (import.meta.main)` and separate exported
functions for each reusable operation. The v5.9.1 LOC estimate already accounts for this
(`PAI-Install/lib/*.ts | Refactor — export functions for reuse`), but implementers should
treat this as a v5.9.1 deliverable, not a v5.9.2 surprise.

After any change: auto-runs `BuildSettings.ts` to rebuild settings.json.

**Shell alias (installed by installer):**
```bash
# Added to ~/.zshrc by installer
alias kai-setup="bun ~/.claude/scripts/kai-setup.ts"
```

**Files:**

| File | Action |
|------|--------|
| `scripts/kai-setup.ts` | NEW — menu-driven configurator |
| `install.sh` | Update — add alias to shell profile |
| `PAI-Install/lib/*.ts` | Refactor — export functions for reuse |

- Each submenu works independently (no full-installer required)

### [2] `kai doctor` — Installation Health Check

**What:** Diagnostic command that verifies the entire KAI installation is healthy: config valid, hooks loadable, keys accessible, MCP responsive, dependencies present.

**Why:** When something breaks, users have no way to diagnose. "It just stopped working" is the common report. A doctor command gives them (and us) actionable diagnostics.

**Implementation:**

```typescript
// scripts/kai-doctor.ts
// Output format:
//   ✅ Config: settings.json valid (schema check passed)
//   ✅ Identity: PAI for YourName (America/Los_Angeles)
//   ✅ Symlink: ~/.claude → ~/Projects/kai
//   ⚠️  API Keys: 3/8 optional keys not set (OPENAI, GEMINI, GROK)
//   ✅ Hooks: 48/48 hook files exist and parse
//   ✅ Skills: 84/84 skills have valid SKILL.md
//   ⚠️  MCP: cloudflare connected, github not responding
//   ✅ Git hooks: pre-commit and pre-push installed
//   ✅ Runtime: bun 1.3.9, node v26, claude found
//   ❌ Notifications: NTFY_TOPIC not set (notifications disabled)
//
//   Summary: 7 passed, 2 warnings, 1 failure
//   Run `kai setup notifications` to fix the failure.

interface Check {
  name: string;
  check: () => Promise<CheckResult>;
  category: "required" | "recommended" | "optional";
}

interface CheckResult {
  status: "pass" | "warn" | "fail";
  message: string;
  fix?: string; // Suggested fix command
}
```

**Check categories:**

| Category | Checks |
|----------|--------|
| Required | Installation exists (`~/.claude/` — symlink OR directory, not hardcoded as symlink) |
| Required | settings.json exists + valid, CLAUDE.md exists, bun installed |
| Required | At least one Claude access method: ANTHROPIC_API_KEY OR Bedrock configured (check `CLAUDE_CODE_USE_BEDROCK=1` in settings.json env). These are mutually exclusive — having Bedrock configured satisfies the requirement even without ANTHROPIC_API_KEY. |
| Recommended | GitHub token, git hooks installed, hook files parseable |
| Optional | Optional API keys, MCP servers, notifications, devices.json |

**Check module discovery:** `scripts/kai-doctor-checks/` contains one file per check group
(e.g., `required.ts`, `recommended.ts`, `optional.ts`). The runner imports all `.ts` files
via `readdirSync` + dynamic import. Each file exports a `checks: Check[]` array. This means
adding a new check = adding it to the appropriate file, no registration step.

**Files:**

| File | Action |
|------|--------|
| `scripts/kai-doctor.ts` | NEW — diagnostic runner (glob-imports check modules) |
| `scripts/kai-doctor-checks/required.ts` | NEW — required checks |
| `scripts/kai-doctor-checks/recommended.ts` | NEW — recommended checks |
| `scripts/kai-doctor-checks/optional.ts` | NEW — optional checks |
| `install.sh` | Update — add `kai-doctor` alias |

**Validation:**
- Clean install → all required pass, some optional warn
- Deliberately break settings.json → doctor catches it with fix suggestion
- Native (non-symlinked) install at ~/.claude/ → installation check passes
- Bedrock user without ANTHROPIC_API_KEY → Claude access check passes (not fails)
- Output is copy-paste friendly for bug reports

### [4] Config Validation Hook

**What:** A SessionStart hook that validates settings.json against the schema (from v5.9.0 Feature [3]) and warns if invalid fields exist.

**Why:** Users edit config files and typos silently break things. A validation hook catches errors at the start of the session with clear guidance.

**Implementation:**

```typescript
// hooks/ConfigValidation.hook.ts (SessionStart, async)
import { readFileSync, existsSync } from "fs";
import { join } from "path";

const PAI_DIR = process.env.PAI_DIR!;
const settingsPath = join(PAI_DIR, "settings.json");
const schemaPath = join(PAI_DIR, "settings-schema.json");

// Only validate if schema exists (v5.9.0 shipped)
if (existsSync(schemaPath) && existsSync(settingsPath)) {
  let settings: unknown;
  
  // Phase 1: Can we parse it at all?
  try {
    settings = JSON.parse(readFileSync(settingsPath, "utf-8"));
  } catch (e) {
    // Malformed JSON — this is worse than schema violations
    const msg = e instanceof SyntaxError ? e.message : "unknown parse error";
    console.log(JSON.stringify({
      additionalContext: `❌ settings.json is malformed JSON: ${msg}. Run \`bun ~/.claude/scripts/kai-reset.ts\` to rebuild from config files.`
    }));
    process.exit(0); // Don't continue to schema validation
  }
  
  // Phase 2: Does it match the schema?
  const schema = JSON.parse(readFileSync(schemaPath, "utf-8"));
  const errors = validate(settings, schema);
  if (errors.length > 0) {
    console.log(JSON.stringify({
      additionalContext: `⚠️ Config validation: ${errors.length} schema issue(s) in settings.json. Run \`bun ~/.claude/scripts/kai-doctor.ts\` for details.`
    }));
  }
}
```

**Files:**

| File | Action |
|------|--------|
| `hooks/ConfigValidation.hook.ts` | NEW — schema validation at startup |
| `config/hooks.jsonc` | Update — register new hook |

**Validation:**
- Valid settings.json → no output (silent pass)
- Invalid key → warning in session with fix suggestion
- Missing schema file → hook is no-op (graceful degradation)

### [7] `kai reset` — Rebuild Configuration from Source

**What:** A command that rebuilds all generated files from config sources. With `--hard`, also clears runtime state (sessions, cache, memory state) for a clean slate without reinstalling.

**Why:** When something breaks — corrupted settings.json, stale CLAUDE.md, broken hook state — users have no recovery path short of `rm -rf ~/.claude && bash install.sh`. `kai reset` gives a graduated recovery: soft reset rebuilds config, hard reset is a factory-reset-in-place.

**Implementation:**

```typescript
// scripts/kai-reset.ts
// Usage:
//   bun scripts/kai-reset.ts          # Soft: rebuild generated files
//   bun scripts/kai-reset.ts --hard   # Hard: also clear runtime state
//   bun scripts/kai-reset.ts --dry-run # Show what would be done

import { execSync } from "child_process";
import { rmSync, existsSync } from "fs";
import { join } from "path";

const PAI_DIR = process.env.PAI_DIR ?? join(process.env.HOME!, ".claude");
const isHard = process.argv.includes("--hard");
const isDryRun = process.argv.includes("--dry-run");

// Soft reset: rebuild generated files from config sources
const SOFT_ACTIONS = [
  { label: "Rebuild settings.json", cmd: `bun ${PAI_DIR}/hooks/handlers/BuildSettings.ts` },
  { label: "Rebuild CLAUDE.md", cmd: `bun ${PAI_DIR}/hooks/handlers/BuildCLAUDE.ts` },
  { label: "Reinstall dependencies", cmd: `bun install --silent`, cwd: PAI_DIR },
  { label: "Reinstall git hooks", cmd: `bash ${PAI_DIR}/scripts/install-hooks.sh` },
];
// NOTE: No shell operators (&&) in cmd strings — execSync without { shell: true }
// doesn't interpret them. Use the `cwd` option for directory changes instead.

// Hard reset: additionally clear runtime state
const HARD_ACTIONS = [
  { label: "Clear session state", path: join(PAI_DIR, "MEMORY/STATE") },
  { label: "Clear work PRDs", path: join(PAI_DIR, "MEMORY/WORK") },
  { label: "Clear staging", path: join(PAI_DIR, "MEMORY/STAGING") },
  { label: "Clear cache", path: join(PAI_DIR, "cache") },
  { label: "Clear sessions", path: join(PAI_DIR, "sessions") },
  { label: "Clear tasks", path: join(PAI_DIR, "tasks") },
  { label: "Remove onboarding flag", path: join(PAI_DIR, "MEMORY/STATE/.onboarding-complete") },
];

// Execute soft actions (always)
for (const action of SOFT_ACTIONS) {
  if (isDryRun) { console.log(`  [dry-run] ${action.label}`); continue; }
  console.log(`  → ${action.label}`);
  execSync(action.cmd, { stdio: "inherit", cwd: (action as any).cwd });
}

// Execute hard actions (only with --hard, with confirmation)
if (isHard) {
  // Count what will be deleted for informed consent
  const counts = HARD_ACTIONS
    .filter(a => existsSync(a.path))
    .map(a => {
      if (a.path.endsWith(".onboarding-complete")) return null;
      const stat = statSync(a.path);
      if (stat.isDirectory()) {
        const entries = readdirSync(a.path).filter(f => !f.startsWith(".")).length;
        return `${entries} files in ${basename(a.path)}`;
      }
      return basename(a.path);
    })
    .filter(Boolean);

  if (!isDryRun) {
    console.log(`\n  ⚠️  --hard will delete: ${counts.join(", ")}`);
    console.log("  Your config files, MEMORY/KNOWLEDGE, and WISDOM are preserved.\n");
    // TODO: prompt for y/N confirmation here
  }

  for (const action of HARD_ACTIONS) {
    // Safety: refuse to delete anything outside PAI_DIR
    if (!action.path.startsWith(PAI_DIR)) {
      console.error(`  ✗ Refusing to delete path outside PAI_DIR: ${action.path}`);
      continue;
    }
    if (isDryRun) { console.log(`  [dry-run] ${action.label}: rm ${action.path}`); continue; }
    if (existsSync(action.path)) {
      rmSync(action.path, { recursive: true, force: true });
      console.log(`  → ${action.label}`);
    }
  }
}
```

**Key decisions:**
- Soft reset is ALWAYS safe — only rebuilds generated files from tracked config sources
- Hard reset requires explicit `--hard` flag and confirms before deleting
- NEVER touches: config/*.jsonc, PAI/USER/, MEMORY/KNOWLEDGE/, MEMORY/WISDOM/, preferences.local.jsonc
- `--dry-run` shows what would happen without doing it
- Clearing the onboarding flag (hard only) re-triggers the first-session experience

**Files:**

| File | Action |
|------|--------|
| `scripts/kai-reset.ts` | NEW — graduated reset command |
| `install.sh` | Update — add `kai-reset` alias |

**Validation:**
- Soft reset fixes corrupted settings.json (rebuilds from config/)
- Soft reset fixes broken CLAUDE.md (rebuilds from template)
- Hard reset clears sessions/ and returns to "fresh" runtime state
- Hard reset preserves all user-customized files
- `--dry-run` produces accurate preview without side effects
- Running after hard reset + starting a session triggers FirstSessionOnboarding again

### [8] `kai upgrade` — Pull Latest and Rebuild

**What:** A single command that pulls the latest KAI version and rebuilds everything, handling the common upgrade workflow that currently requires 4 manual steps.

**Why:** Users must currently: `git pull` → `bun install` → `bun BuildSettings.ts` → `bun BuildCLAUDE.ts`. They forget steps, get stale state, or pull without rebuilding. One command does it all.

**Implementation:**

```typescript
// scripts/kai-upgrade.ts
// Usage:
//   bun scripts/kai-upgrade.ts          # Pull + rebuild
//   bun scripts/kai-upgrade.ts --check  # Check if update available (no action)

import { execSync } from "child_process";
import { join } from "path";
import { existsSync, readFileSync } from "fs";

const PAI_DIR = process.env.PAI_DIR ?? join(process.env.HOME!, ".claude");
const isCheckOnly = process.argv.includes("--check");

// Step 1: Fetch remote to see if updates exist
console.log("  → Checking for updates...");

// Resolve the tracked upstream (don't hardcode origin/main)
let upstream: string;
try {
  upstream = execSync("git rev-parse --abbrev-ref @{upstream}", { cwd: PAI_DIR, encoding: "utf-8" }).trim();
} catch {
  console.error("  ✗ No upstream branch configured. Run: git branch --set-upstream-to=origin/main");
  process.exit(1);
}
const [remoteName, ...branchParts] = upstream.split("/");
const branchName = branchParts.join("/");

execSync(`git fetch ${remoteName}`, { cwd: PAI_DIR, stdio: "pipe" });

const local = execSync("git rev-parse HEAD", { cwd: PAI_DIR, encoding: "utf-8" }).trim();
const remote = execSync(`git rev-parse ${upstream}`, { cwd: PAI_DIR, encoding: "utf-8" }).trim();

if (local === remote) {
  console.log("  ✓ Already up to date");
  process.exit(0);
}

// Show what's new
const commitCount = execSync(
  `git rev-list --count ${local}..${upstream}`,
  { cwd: PAI_DIR, encoding: "utf-8" }
).trim();
console.log(`  → ${commitCount} new commit(s) available on ${upstream}`);

if (isCheckOnly) {
  // Use --no-pager instead of pipe to avoid shell: true requirement
  execSync(`git --no-pager log --oneline -5 ${local}..${upstream}`, { cwd: PAI_DIR, stdio: "inherit" });
  process.exit(0);
}

// Step 2: Check for local modifications that would conflict
const status = execSync("git status --porcelain", { cwd: PAI_DIR, encoding: "utf-8" }).trim();
if (status) {
  console.log("  ⚠️  Local modifications detected:");
  console.log(status.split("\n").map(l => `    ${l}`).join("\n"));
  console.log("\n  Stash or commit before upgrading. Aborting.");
  process.exit(1);
}

// Step 3: Pull (use resolved upstream, not hardcoded origin/main)
console.log(`  → Pulling from ${upstream}...`);
execSync(`git pull ${remoteName} ${branchName} --ff-only`, { cwd: PAI_DIR, stdio: "inherit" });

// Step 4: Rebuild (with error recovery)
const rebuildSteps = [
  { label: "Installing dependencies", cmd: "bun install --silent", cwd: PAI_DIR },
  { label: "Rebuilding settings.json", cmd: `bun ${PAI_DIR}/hooks/handlers/BuildSettings.ts` },
  { label: "Rebuilding CLAUDE.md", cmd: `bun ${PAI_DIR}/hooks/handlers/BuildCLAUDE.ts` },
];

let rebuildFailed = false;
for (const step of rebuildSteps) {
  console.log(`  → ${step.label}...`);
  try {
    execSync(step.cmd, { stdio: "pipe", cwd: (step as any).cwd });
  } catch (e) {
    console.error(`  ✗ ${step.label} failed: ${e instanceof Error ? e.message : e}`);
    rebuildFailed = true;
  }
}

if (rebuildFailed) {
  console.error("\n  ⚠️  Pull succeeded but rebuild had errors.");
  console.error("  Run `bun ~/.claude/scripts/kai-reset.ts` to retry the rebuild.");
  process.exit(1);
}

// Step 5: Show version (graceful if VERSION file missing)
const versionPath = join(PAI_DIR, "VERSION");
let version = "unknown";
if (existsSync(versionPath)) {
  version = readFileSync(versionPath, "utf-8").trim();
} else {
  try {
    version = execSync("git describe --tags --always", { cwd: PAI_DIR, encoding: "utf-8" }).trim();
  } catch { /* leave as "unknown" */ }
}
console.log(`\n  ✓ Upgraded to v${version}`);
console.log("  Restart your Claude Code session to use the new version.");
```

**Key decisions:**
- Only does fast-forward pulls (`--ff-only`) — won't auto-merge
- Detects dirty working tree and refuses to upgrade (safe)
- `--check` mode just shows what's available (no changes)
- Reads tracked upstream from git config (not hardcoded `origin/main`) — works for users tracking custom branches or forks
- Runs `bun install` in case dependencies changed
- Rebuilds both settings.json and CLAUDE.md; if either fails, reports error with `kai-reset` recovery path
- Shows version from VERSION file with fallback to `git describe --tags` if VERSION is missing
- Does NOT auto-restart sessions (user chooses when)
- No shell operators or pipes in execSync — uses `--no-pager`, `cwd` option, and separate calls

**CheckVersion.hook.ts coordination:** Once `kai-upgrade --check` ships, CheckVersion.hook.ts
(which checks Claude Code npm version with 24h TTL) becomes partially redundant for the
"is KAI up to date?" question. Options: (a) retire CheckVersion entirely and fold its
Claude Code version check into `kai-doctor`, (b) keep CheckVersion for Claude Code but
add a `kai-upgrade --check` nudge alongside it, or (c) merge both checks into a single
VersionCheck hook. Decide at implementation time — the key constraint is that `kai-upgrade`
handles KAI repo updates while CheckVersion handles Claude Code runtime version, so they're
complementary, not duplicative.

**Extends existing tests:** `tests/Upgrade.test.ts` already exists — new tests for
kai-upgrade should be added there, not in a parallel file.

**Files:**

### [10] Release Audit Script (`kai-release-audit.ts`)

**What:** An automated script that replaces the ~15 manual grep checks in `docs/RELEASE-CHECKLIST.md` with a single executable that passes/fails.

**Why:** `verify-release.sh` handles the kai sync verification. But the release checklist has ~15 manual grep-based checks for PII, brand consistency, dead links, and count markers that are currently done by hand. Automating these eliminates human error in the release process.

**Checks to automate (from RELEASE-CHECKLIST.md):**
1. No PII patterns (email addresses, usernames, IP addresses) outside expected locations
2. Brand consistency — no stray "PAI" in user-facing KAI content
3. Dead internal links — `[[broken-ref]]` patterns that don't resolve
4. Marker region counts match actual file counts
5. No gitignored secrets accidentally committed (`.env` patterns, API keys)
6. skills-lock.json is up-to-date with installed skills
7. VERSION file matches manifest.json version
8. No TODO/FIXME/HACK in shipped hooks (configurable)
9. All hooks listed in config/hooks.jsonc actually exist on disk
10. No duplicate skill names across directories

**Implementation:**

```typescript
// scripts/kai-release-audit.ts
// Usage:
//   bun scripts/kai-release-audit.ts              # Run all checks
//   bun scripts/kai-release-audit.ts --fix        # Auto-fix where possible
//   bun scripts/kai-release-audit.ts --category pii  # Run specific category

interface AuditCheck {
  name: string;
  category: "pii" | "brand" | "integrity" | "hygiene";
  run: () => AuditResult;
}

interface AuditResult {
  pass: boolean;
  message: string;
  fixable?: boolean;
  details?: string[];
}

// Output:
//   Release Audit — kai v5.9.2

//     ✅ No TODO/FIXME in shipped hooks
//     ✅ Marker counts match file counts

//   9/10 passed, 1 fixable. Run with --fix to auto-correct.
```

## Alternative: Ship as v6.0.1 Instead?

**Arguments for v5.9.2:**
- No breaking changes — purely additive
- Benefits users NOW (before v6.0's longer development cycle)
- Tests the "self-service config" pattern before v6.0 takes it further
- Minimal dependencies (just needs v5.9.0 schema + v5.9.1 modules)

**Arguments for deferring to v6.0.1:**
- `kai setup` could use Workflow Templates (v6.0 [3]) instead of custom scripts
- The PAI→KAI rename (v6.0 [6]) means CLI commands might be `kai` in one repo and `pai` in another
- Could ship the whole "self-service config" story at once with the new primitives

**Recommendation:** Ship as v5.9.2. The CLI tools are useful immediately, and v6.0 Workflow Templates can eventually WRAP them (not replace them). The scripts remain the implementation; workflow templates become a convenience layer.
