## Phase A: Steering Enforcement + Graceful Session Exit

Full design for A1 in `docs/planning/steering-enforcement-design.md`. ~150 LOC + 15 tests.
A2 and A3 are new additions addressing the "how do I cleanly exit a session" problem.

### A1. PlanApprovalGuard.hook.ts (new)

Detects when a plan was presented in the last response and injects a reminder at next
UserPromptSubmit if the user hasn't explicitly approved.

**Detection:** ≥2 co-occurring signals in last assistant message:
- `## Execution Order` or `## Phase \d` section header
- Completion gate checklist (`- \[ \]` lines with gate language)
- Phase table with time estimates

**State:** `MEMORY/STATE/plan-pending.json` — written by StopOrchestrator when plan
detected; cleared by SessionCleanup or on approval detection.

**Injection:** `additionalContext` (soft — not a blocking decision). Short reminder:
"A plan was presented. Waiting for your go-ahead before executing."

**Approval detection:** "go ahead", "do it", "start", "execute", "approved", "yes", "proceed"
+ short single-phrase responses (≤5 words, no `?`, no new task verb-noun pair).

**Files:**
- `hooks/PlanApprovalGuard.hook.ts` (new, ~80 LOC)
- `hooks/handlers/PlanDetection.ts` (new handler): detect plan signals, write state (~30 LOC)
- `hooks/SessionCleanup.hook.ts`: delete `plan-pending.json` at session end (~5 LOC)
- `config/hooks.jsonc`: register PlanApprovalGuard on UserPromptSubmit
- `tests/PlanApprovalGuard.test.ts`: detection logic, approval parsing (~15 tests)

### A2. SessionCloseGuard.hook.ts — DEFERRED to v5.4.0

Natural-language exit detection (implicit phrase matching) deferred. Ship `/end` first,
learn from usage patterns, then build the implicit detection layer on top.
See NEXT-STEPS.md backlog for full design.

### A3. `/end` custom slash command — explicit session close

**The problem:** `/exit` is a Claude Code built-in with no hook surface. We can't intercept
it. So `/end` is a PAI-registered skill command that runs a full session close flow first,
then instructs you to type `/exit`.

**Flow when you type `/end`:**
1. PAI skill fires (via `Skill` hook, registered in skills/)
2. Reads session state: active PRD, STAGING count, memory written, ratings captured
3. Claude produces a structured closing summary
4. Outputs: "Session summary above. Type `/exit` to close, or keep going."

**What it surfaces:**
- Work completed this session (PRD criteria checked off, commits made)
- Memory captured: knowledge domains updated, failure patterns logged, relationship notes
- STAGING drafts pending `pai curate`
- Any mid-Algorithm PRD with unchecked criteria (HANDOFF reminder)
- Unsaved decisions or context worth noting

**Files:**
- `skills/PAI/end/SKILL.md` (new) — trigger: `/end`, "end session", "session end"
- `skills/PAI/end/Workflows/SessionClose.md` — the close workflow
- No hook changes needed — skill invocation handles everything

**Note:** `/end` produces the summary but doesn't call `/exit` automatically — that's
intentional. You might decide to keep going after seeing the summary.

## Phase R: Deliberate Research Mode

`deliberate.ts` already implements multi-model parallel invocation, multi-round debate,
synthesis, graceful degradation, and config-driven panels. R adds a research execution
mode to the existing script — no new abstraction layer needed.

**Dropped from original plan:**
- ~~R1: `ModelInvocation.ts` extraction~~ — `deliberate.ts` already has `invokeModel()`,
  `invokeClaudeModel()`, `invokeGemini()`, `invokeOpenAICompatible()`. Extracting them
  is refactoring overhead, not new capability.
- ~~R2: Add `tools` param to `Inference.ts`~~ — The `--tools ''` flag in Inference.ts is
  **intentional**: it makes Inference.ts a safe, contained inference primitive that can't
  trigger hooks or make filesystem changes from within hook subprocesses. Adding web search
  through this path would break the design. Web grounding goes through direct API calls
  in the model-specific invoke functions, not through the `claude` CLI subprocess.

### R1. `deliberate.ts --mode research`

New execution path in the existing script. Reuses `invokeModel()` infrastructure.

**Scatter-Gather-Synthesize** (vs. debate's Round 1/2/N position revision):
1. **Scatter** — each model independently answers the query with web grounding if available
2. **Cross-check** — identify claims appearing in ≥2 responses (higher confidence)
3. **Synthesize** — Claude Opus produces final answer with source attribution and confidence

```bash
bun deliberate.ts --mode research "What are the latest Claude Code hook capabilities?"
bun deliberate.ts --mode research --models gemini,gpt "Current state of WebLLM?"
```

Key difference from debate mode: no position revision rounds, no adversarial prompting.
Models answer once, synthesis weights by cross-source agreement.

### R2. Web grounding per model (Gemini + Grok only for v5.3.0)

Add grounding to the existing invoke functions — same REST API pattern already in place:
- **Gemini**: add `tools: [{ google_search: {} }]` to request body (Google Search grounding).
  Citations in `groundingMetadata.groundingChunks[].web.uri`.
- **Grok**: add `search_parameters: { mode: "auto" }` as top-level request field (xAI API).
  Grok routes through `invokeOpenAICompatible` — add conditional: if search enabled AND
  `config.id === 'grok'`, inject `search_parameters` into the request body.
- **Claude**: no web search via `claude` CLI subprocess (Inference.ts is intentionally
  tool-free). Claude contributes synthesis and reasoning from training knowledge.
- **GPT**: DEFERRED to v5.4.0. GPT web search requires the Responses API (`/v1/responses`)
  which has a different request/response schema from Chat Completions. Needs a new
  `invokeOpenAIResponses()` function. Two grounded sources (Gemini + Grok) plus Claude
  reasoning is sufficient coverage for v5.3.0.

Each grounded model returns search results in provider-specific format — extract and include
as citations in the synthesis output. Provider-specific parsing needed for each.

### R3. Skill docs
- `skills/Deliberate/Workflows/ResearchMode.md` — Scatter-Gather-Synthesize workflow
- Update `skills/Deliberate/SKILL.md` trigger words to include research mode invocation
