# v5.8.0 Release Plan — KAI-Generic Board Architecture

**Theme:** Board Maturity + KAI Portability
**Target:** v5.8.0 (after v5.5–5.7 ship smaller items)
**PRD:** `MEMORY/WORK/20260520-140000_v580-kai-generic-board-architecture/PRD.md`

---

## Overview

Refactor the KAI Board from a 946-line monolith into an adapter-based architecture that:
- Tracks session lifecycle (start, end, duration, exit status)
- Binds sessions explicitly to tasks
- Records phase transitions as events (timeline, not just current state)
- Runs identically on PAI (iTerm) and KAI (tmux/generic)

No database. No new runtime dependencies. Single-file HTML preserved.

## Phase 1: Session Lifecycle Schema (Foundation)

**Goal:** Add lifecycle tracking to work.json without breaking any consumers.

**Approach:** Strictly additive fields. Existing consumers ignore unknown keys.

### Schema Changes (work.json sessions map)

```typescript
interface SessionEntry {
  // EXISTING (unchanged)
  task: string;
  sessionName: string;
  sessionUUID: string;
  phase: string;
  progress: string;
  effort: string;
  mode: string;
  started: string;       // ISO timestamp
  updatedAt: string;     // ISO timestamp

  // NEW — v5.8 additions (all optional for backward compat)
  taskSlug?: string;           // Explicit binding to a WORK/ PRD slug
  lifecycle?: {
    startedAt: string;         // ISO — when session began
    endedAt?: string;          // ISO — when session ended (null = still active)
    durationMs?: number;       // Computed on end
    exitStatus?: "completed" | "abandoned" | "crashed" | "timeout";
    exitReason?: string;       // Brief note: "user closed", "context exhausted", etc.
  };
  events?: Array<{
    type: "phase_change" | "criterion_pass" | "session_start" | "session_end";
    at: string;                // ISO timestamp
    from?: string;             // Previous state (for phase_change)
    to?: string;               // New state
    detail?: string;           // Optional context
  }>;
}
```

## Phase 2: Terminal Adapter Interface (Extraction)

**Goal:** Extract iTerm-specific code into a pluggable adapter, add tmux as KAI default.

### Adapter Interface

```typescript
// scripts/adapters/terminal.ts
export interface TerminalAdapter {
  name: string;

  /** Detect which sessions are currently active */
  getActiveSessions(): Promise<Set<string>>;

  /** Launch a new terminal session for a task */
  launchSession(opts: {
    workDir: string;
    command: string;
    title?: string;
  }): Promise<{ success: boolean; error?: string }>;

  /** Check if this adapter is available on the current system */
  isAvailable(): Promise<boolean>;
}
```

### Concrete Adapters

```
scripts/
├── adapters/
│   ├── terminal.ts        # Interface + factory
│   ├── iterm.ts           # Current getActiveSessionUUIDs + osascript launch
│   └── tmux.ts            # tmux list-sessions + tmux new-window
```

**iTerm adapter** (`iterm.ts`):
- Moves current `getActiveSessionUUIDs()` logic (lines 210–257 of board.ts)
- Moves current osascript launch logic (lines 803–829 of board.ts)
- Depends on: `~/.claude/MEMORY/STATE/iterm-sessions/*.json`

**tmux adapter** (`tmux.ts`):
- `getActiveSessions()`: `tmux list-sessions -F "#{session_name}"` + check for claude process
- `launchSession()`: `tmux new-window -n "{title}" "cd {workDir} && claude"`
- `isAvailable()`: `which tmux` exits 0

**Factory** (`terminal.ts`):
```typescript
export function createAdapter(name: string): TerminalAdapter {
  switch (name) {
    case "iterm": return new ITermAdapter();
    case "tmux": return new TmuxAdapter();
    default: return new TmuxAdapter(); // fallback
  }
}
```

### Config Addition (board-config.json)

```json
{
  "terminal": "iterm",   // PAI default
  // KAI default: "tmux"
}
```

### Files Modified

| File | Change |
|------|--------|
| `scripts/adapters/terminal.ts` | NEW — interface + factory |
| `scripts/adapters/iterm.ts` | NEW — extracted from board.ts lines 210–257, 803–829 |
| `scripts/adapters/tmux.ts` | NEW — tmux implementation |
| `scripts/board.ts` | Import adapter, replace inline iTerm code |

## Phase 3: Work Loader Abstraction (Portability)

**Goal:** Make work directory scanning configurable so KAI can point to any directory.

### Loader Interface

```typescript
// scripts/loaders/work.ts
export interface WorkLoader {
  /** Scan configured directories for task PRDs */
  loadTasks(): Promise<WorkItem[]>;

  /** Update a task's phase */
  updatePhase(slug: string, phase: string): Promise<boolean>;

  /** Toggle a criterion */
  toggleCriterion(slug: string, criterionId: string): Promise<boolean>;

  /** Create a new task */
  createTask(title: string, description: string, effort: string, mode: string): Promise<string>;

  /** Find PRD path by slug */
  findPrd(slug: string): Promise<string | null>;
}
```

### Implementation

```
scripts/
├── loaders/
│   ├── work.ts            # Interface
```

## Phase 4: Task↔Session Binding + UI (Integration)

**Goal:** Connect phases 1-3 into the UI. Show session history per task, timeline, and duration.

### UI Changes (board.html)

1. **Task cards**: Show session count badge ("3 sessions, 47min total")
2. **Expanded task view**: Timeline of phase events with timestamps
3. **Session cards**: Show duration + exit status (green checkmark, red X, grey dash)
4. **Active indicator**: Use adapter-provided active detection (green pulse dot)

### Data Flow

```
PRDSync (hook) → work.json [events array] → board API → board UI
SessionAutoName → work.json [lifecycle.startedAt] → board API → session duration
SessionSummary → work.json [lifecycle.endedAt] → board API → completion badge
Board adapter → active detection → green pulse dot on session card
```

### New API Responses

`/api/work` response adds:
```json
{
  "sessions": [{
    "slug": "...",

    "events": [{ "type": "phase_change", "at": "...", "from": "observe", "to": "think" }],
    "taskSlug": "20260520-..."
  }]
}
```

## Migration Strategy

### Existing work.json Sessions

No migration needed. New fields are optional:
- Sessions without `lifecycle` → board shows "–" for duration
- Sessions without `events` → timeline view is empty
- Sessions without `taskSlug` → "Unbound session" in UI

New sessions get all fields automatically via updated hooks.

### Existing board-config.json

Additive only:
- Missing `"terminal"` → defaults to `"iterm"` (PAI) or `"tmux"` (KAI)
- Missing `"workRoot"` → defaults to first entry in `scanDirs`
