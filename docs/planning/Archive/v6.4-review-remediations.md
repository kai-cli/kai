## v6.4.0 — Write Coordination + Security Hardening

**Theme:** Fix data-loss risk, harden security classifier, establish E2E scaffold

### P0: SessionEnd Write Coordination

**Problem:** 9 async SessionEnd hooks fire simultaneously. If the process dies mid-execution, some complete and some don't. No way to know which ran.

**Solution:** Per-hook UUID sentinel files (not a shared counter — avoids TOCTOU race).

**Implementation:**
1. Create `hooks/lib/session-end-tracker.ts`
   - On hook start: write `MEMORY/STATE/session-end/{hook-name}.{session-id}.started`
   - On hook complete: rename to `.complete`
   - No shared counter — no TOCTOU race between hooks
2. Add staleness check to `LoadContext.hook.ts` (SessionStart)
   - Scan for `.started` files with no matching `.complete`
   - Log which hooks didn't finish, clean up stale sentinels
3. Update 9 SessionEnd hooks: add `markStarted()`/`markComplete()` calls (2 lines each)

**Validation:**
- Unit test: create stale sentinel → verify startup detects it
- Integration test: kill mid-execution → verify partial completion detected

---

### P1a: Security Classifier Fuzzing

**Problem:** `risk-classifier.ts` has confirmed bypass vectors:
- `echo "rm -rf /" | bash` → classified as `is_read_only` (sees `echo`)
- Variable expansion in destructive contexts: `rm $VAR`
- `curl ... | bash` — pipe to shell undetected

**Implementation:**
1. Create `tests/RiskClassifier.fuzz.test.ts`
   - Pipe-to-shell, base64-to-shell, path traversal, semicoloned chains
   - Property: any command containing a destructive action must NOT be `is_read_only`
   - 10K+ iterations
2. Harden classifier:
   - Pipe-to-shell detection (final pipeline command is `bash`/`sh`/`eval` → `is_risky`)
   - Base64-decode-to-shell detection
   - Path normalization (`..` resolution)
   - Unresolved variables → `is_risky` **only in destructive-command contexts** (not `ls $HOME`)
3. Document threat model in `docs/architecture/threat-model.md`

**Validation:**
- Fuzzer reproduces all 4 known bypass patterns
- All bypasses have regression tests
- No false positives on: `ls $HOME`, `cat $CONFIG`, `echo $PATH`

---

### P1b: E2E Scaffold (ship with P0)

**Problem:** No tests verify the hook stdin/stdout protocol end-to-end.

**Implementation:**
1. Create `tests/e2e/hook-harness.ts`
   - Spawns hook via `run-hook.sh` with controlled JSON stdin
   - Captures stdout/stderr/exit code, validates response schema
2. Initial scenarios (≥3):
   - SecurityValidator blocks `rm -rf /` (exit 2)
   - SecurityValidator allows `ls -la` (exit 0, `{"continue": true}`)
   - P0 crash detection: stale sentinel → startup → verify detection

**Validation:** Scaffold runs in CI.

## v6.4.1 — Full E2E + Security Audit

**Theme:** Complete E2E coverage, audit fail-open/closed behavior, input scanning

### Full E2E Suite

Expand scaffold to ≥10 scenarios:
- GitHubWriteGuard: block → approve → allow cycle
- SessionEnd composite: all hooks fire and complete
- Sync hook failure: what happens when a blocking hook crashes? (timeout behavior)
- Hook ordering: verify PreToolUse hooks execute in registered order

### Fail-Open/Closed Audit

**Rule:** Hooks that write to disk → fail-closed. Read-only → fail-open.
- Audit all 24 disk-writing hooks
- Verify error handlers don't silently swallow write failures
- Add explicit `fail-closed` behavior to write-path hooks that currently catch-and-ignore

### Tool Credential Declaration (CrewAI pattern)

**Problem:** Tools fail at runtime with opaque errors when environment variables are missing. DevTeam's `detectReviewCapability()` is a one-off; every script/skill should declare deps upfront.

**Implementation:**
1. `credentials.yaml` in script/skill dirs declares required + optional env vars
2. `hooks/lib/credential-validator.ts` validates at load time
3. DevTeam `executePhase()` validates before spawning (fail fast)
4. Missing optional → graceful skip with log; missing required → clear error

**Validation:**
- Missing required credential → error before agent spawn, not mid-execution
- Existing detectReviewCapability() becomes a special case

---

### Input Secret Scanning

Add check in `SecretScanner.hook.ts` (UserPromptSubmit) for common patterns in user prompts.
- Patterns: `sk-`, `ghp_`, `AKIA`, `-----BEGIN`, `Bearer eyJ`

### CLI Argument Injection Audit

Audit all `Bun.spawn` calls for argument injection vectors.
- Priority: `Inference.ts` (passes user prompts as CLI args to `claude` binary)

### Hook Contract Versioning

Add `hookProtocolVersion` field to `hook-io.ts` that warns on unrecognized fields rather than crashing. Prevents breakage when Claude Code evolves its hook schema.

## v6.6.0 — DevTeam Intelligence & Adaptive Behavior

**Theme:** Cost awareness, dynamic team composition, and blocked-agent detection
**Source:** Paperclip AI orchestration patterns (github.com/paperclipai/paperclip)

### Per-Phase Cost Tracking

**Problem:** No visibility into what DevTeam runs cost. Can't compare Opus vs Sonnet scoping ROI, can't set budgets, can't detect runaway agents.

**Implementation:**
1. Parse token usage from `claude` CLI output (use `--output-format json` which includes usage stats)
2. Add `CostTracker` class to `dev-team.ts`:
   - Captures input/output tokens per phase
   - Calculates cost using model pricing table (Opus: $15/$75, Sonnet: $3/$15, Haiku: $0.25/$1.25 per MTok)
   - Logs cost events to `run.jsonl`: `{"phase":"scope","event":"cost","input_tokens":8200,"output_tokens":3100,"cost_usd":0.28}`
3. Add soft/hard budget limits to TeamConfig:
   - `budgetSoftLimitUsd`: warn at this threshold (default: $2.00)
   - `budgetHardLimitUsd`: abort at this threshold (default: $5.00)
   - CLI flag: `--budget <usd>` sets hard limit
4. Report includes cost breakdown per phase and total:
   ```
   ## Cost Summary
   | Phase | Model | Tokens (in/out) | Cost |
   |-------|-------|-----------------|------|
   | scope | opus  | 8.2K / 3.1K     | $0.28 |
   | implement | sonnet | 12K / 8K   | $0.16 |
   | verify | sonnet | 6K / 2K      | $0.04 |
   | Total | — | 26.2K / 13.1K    | $0.48 |
   ```
5. Dry-run includes estimated cost range based on preset + model overrides

**Validation:**
- Cost appears in run.jsonl after each phase
- Soft limit produces warning but continues
- Hard limit aborts cleanly with partial report
- `--budget 0.50` aborts a multi-agent run before QA if Dev was expensive

---

### Dynamic Role Injection

**Problem:** If QA discovers a security vulnerability, there's no specialist to assess severity. If Dev's implementation is unusually complex, no architect validates the approach. Current teams are static.

**Implementation:**
1. Add `ConditionalRole` type to presets:
   ```yaml
   conditional_roles:
     - id: security-specialist
       trigger: "vulnerability|injection|XSS|CVE|OWASP"
       model: opus
       purpose: Assess severity, recommend remediation approach
       inject_after: verify
     - id: architect
       trigger: "refactor|redesign|migration|breaking change"
       model: opus
       purpose: Validate approach, identify coupling risks
       inject_after: implement
   ```
2. After each phase, scan agent output for trigger patterns (simple `.includes()` check)
3. If triggered, spawn conditional role agent with full context (PM findings + triggering output)
4. Conditional agent output appended to phase report; feeds into next phase
5. Log conditional spawns: `{"phase":"conditional","event":"triggered","role":"security-specialist","trigger_match":"injection"}`
6. SKILL.md updated: orchestrator informs user when a conditional role fires ("Security specialist triggered by QA finding — reviewing before merge.")

**Guardrails:**
- Max 1 conditional role per trigger per run (no infinite loops)
- Conditional roles cannot trigger other conditional roles
- Hard budget limit still applies (conditional spawns count toward total)
- User can disable with `--no-conditional-roles`

**Validation:**
- QA output containing "SQL injection vulnerability" triggers security-specialist
- QA output with no trigger words does NOT spawn extra agents
- Conditional agent's findings appear in final report
- Budget enforcement still works with conditional spawns

---

### Blocked Agent Detection

**Problem:** Timeout catches dead agents, but not agents that are running but producing no useful output (spinning, asking themselves questions, looping without progress). These consume budget without value.

**Implementation:**
1. Monitor stdout stream during `executePhase`:
   - Track bytes received per 30s window
   - If <100 bytes in 60s after initial output starts → "stall" warning
   - If <100 bytes in 120s → classify as "blocked" and enter recovery
2. Add stall detection to `executePhaseWithRecovery`:
   - Stall (60s silence mid-output) → log warning, continue watching
   - Block (120s silence mid-output) → SIGTERM + explicit-recovery retry with "You appeared to stall. Focus on completing the task directly without deliberation loops."
3. Distinct from timeout: timeout is total wall-clock, block detection is output-gap during execution
4. Log: `{"phase":"implement","event":"stall","silence_sec":65}` and `{"phase":"implement","event":"blocked","silence_sec":125}`

**Validation:**
- Unit test: mock proc that stops writing after 500 bytes → triggers stall → block → recovery
- Blocked recovery prompt includes "You appeared to stall" instruction
- Normal agents with occasional pauses (< 60s) are NOT flagged

---

### Adaptive Retry from Priority Signal

**Problem:** Currently all QA failures trigger the same retry behavior regardless of severity. A "Minor Concern" wastes a full retry cycle that could be deferred.

**Implementation:**
1. In `orchestrate()` verdict parsing, use priority signal to decide retry behavior:
   - `Critical Blocker` → always retry (current behavior)
   - `Standard Issue` → retry once, then escalate
   - `Minor Concern` → skip retry, note in report as "deferred", proceed to review
2. Add `--strict` flag to force retry on all severity levels (overrides adaptive behavior)
3. Report flags deferred items: "1 Minor Concern deferred (see QA report)"

**Validation:**
- QA returning "Minor Concern" with FAIL does NOT trigger Dev retry
- QA returning "Critical Blocker" with FAIL ALWAYS triggers retry
- `--strict` flag overrides and retries everything
- Deferred items appear clearly in final report

---

### Event-Driven Checkpointing (CrewAI pattern: Resume from Failure)

**Problem:** If dev-team.ts dies mid-QA, PM scoping and Dev implementation are lost. Must restart from scratch (10+ minutes wasted compute).

**Implementation:**
1. Typed checkpoint events: `phase_started`, `phase_completed`, `phase_failed`, `verdict`, `run_started`, `run_completed`
2. Persist phase outputs to `~/.claude/teams/{name}/checkpoints/{phase}.output.md`
3. Add `--resume <team-name>` flag: finds last completed phase, loads its output, continues
4. Checkpoint files cleaned up after successful `run_completed`

**Validation:**
- Kill mid-implement → resume picks up from implement (doesn't re-run PM)
- Kill mid-QA → resume picks up from QA with Dev output loaded
- Normal runs write checkpoints transparently

---

### Conditional Task Execution (CrewAI pattern)

**Problem:** DevTeam always runs PM→Dev→QA→Review in sequence. But investigation presets don't need QA, typo fixes don't need PM, small diffs don't need review.

**Implementation:**
1. Add `condition` field to preset phases (e.g., `condition: "output.implement.length > 500"`)
2. Simple expression evaluator (not eval — safe predicate matching against PhaseContext)
3. Skipped phases logged to run.jsonl
4. `--strict` flag ignores all conditions (escape hatch)

**Validation:**
- Trivial fix (< 500 chars Dev output) → QA skipped, logged
- 200-line refactor → QA and Review both execute
- `--strict` overrides and runs everything

---

### DAG Execution Planner (CrewAI pattern)

**Problem:** Sequential phase ordering prevents parallelism. QA and security-review both depend on implement but not on each other — should run simultaneously.

**Implementation:**
1. Phases declare `depends_on: [phase_ids]` instead of implicit ordering
2. Topological sort produces execution tiers (same tier = parallel)
3. Existing presets without `depends_on` default to sequential (zero breaking changes)
4. Dry-run shows execution plan tiers

**Validation:**
- Feature preset with QA + security-review → both run in parallel (same tier)
- Circular dependency → clear error at plan-build time
- Existing bug-fix preset works unchanged

---

### Run History Analytics

**Problem:** After 10+ DevTeam runs, data sits in `run.jsonl` files unused. No way to answer "what's my QA pass rate?" or "is Opus scoping worth the cost?"

**Implementation:**
1. `bun dev-team.ts --stats` command reads all `~/.claude/teams/*/run.jsonl`
2. Computes:
   - Success rate per preset (QA PASS on first attempt, after retries, escalated)
   - Average cost per preset (requires cost tracking feature)
   - Average duration per phase
   - Common failure patterns (group QA failure reasons)
   - Model override impact (pass rate with/without overrides)
3. Output: table + one-line recommendations ("Opus scoping shows 15% higher first-pass QA rate — worth the 5x cost on complex issues")
4. Optional `--since 7d` / `--preset bug-fix` filters

**Validation:**
- `--stats` produces output after ≥3 completed runs
- Filters work independently and combined
- Empty history → helpful message, not crash

---

### Inter-Agent Clarification Protocol

**Problem:** When Dev encounters ambiguity (PM criteria unclear, conflicting constraints), it can only guess forward and hope QA catches the mistake. Burns a full Dev+QA cycle on misunderstandings.

**Implementation:**
1. Dev prompt includes instruction: "If PM criteria are ambiguous or contradictory, emit a `### Clarification Needed` section with specific questions instead of guessing."
2. Orchestrator scans Dev output for `### Clarification Needed`
3. If found: re-invoke PM with Dev's questions + original context → PM produces clarified criteria
4. Feed clarified criteria back to Dev (counts as same attempt, not a retry)
5. Max 1 clarification round per attempt (prevents ping-pong)
6. Log: `{"phase":"implement","event":"clarification","questions":2}`

**Guardrails:**
- Clarification does NOT count toward retry_max (it's not a failure)
- Only Dev can request clarification (QA just fails/passes)
- If Dev produces BOTH implementation AND clarification section → ignore clarification (partial progress wins)

**Validation:**
- Dev output with `### Clarification Needed` triggers PM re-run
- Clarified criteria fed back to Dev without burning a retry
- Dev output without the section proceeds normally
- Max 1 clarification per attempt enforced

---

### Run Templates (Saved Configs)

**Problem:** Power users develop preferred configs (Opus scoping + 600s timeout for security work, Haiku QA + no-review for typos). Must type full flags every time.

**Implementation:**
1. `bun dev-team.ts --save-template "security-fix"` persists current flags to `~/.claude/teams/templates/security-fix.yaml`
2. Template stores: preset, model-overrides, timeout, priority, budget, no-review, strict, no-conditional-roles
3. `bun dev-team.ts --template "security-fix" --issue "..."` loads saved config (explicit flags override template values)
4. `bun dev-team.ts --list-templates` shows available templates
5. Templates are YAML for human editability

**Validation:**
- Save → load round-trip preserves all flags
- Explicit flag overrides template value
- `--list-templates` works with 0 and N templates
- Invalid template name → clear error

### Planning Observer / Adaptive Replanning (CrewAI pattern)

**Problem:** Algorithm runs a plan to completion. If step 3/5 reveals wrong assumptions, steps 4-5 execute anyway (wasting context). 

**Implementation:**
1. After each Algorithm phase, evaluate divergence between expected and actual outcome
2. If divergence > 0.3, trigger replanning: rewrite remaining steps based on new information
3. Only applies to Algorithm-mode (OODA-V cycles), not NATIVE mode
4. Replan decision logged for observability

**Integration:** Extends the meta-cognitive monitor from "check at end" to "check between steps." Same policy engine, applied incrementally.

**Prerequisite:** Algorithm decomposition (v6.5.1) — can't observe individual phases until extracted into separate functions.

## Rationale Appendix: Deliberation Decisions

### Q1: Two-Repo Sync — Keep A, Harden
Panel recommended monorepo (B) but blind spots undermine it: migration cost is weeks, a filter leak is irreversible, "gates must actually fail" favors reviewable diffs. Hardening A is cheaper and safer.

### Q2: Learning Lifecycle — Event-Source + Materialized View
All 6 models rejected time-based rotation as arbitrary. Constraint is context window budget, not storage. Hash-dedup + frequency + recency into a token-capped view. LLM-assisted pruning rejected (too expensive + non-deterministic at this scale).

### Q3: Algorithm Future — Meta-Cognitive Monitor
5/6 selected C. Algorithm is partially redundant given Claude's native extended thinking. Transform to post-generation linter with testable policy checks. Keep thin formatting spec. Context savings: ~18% of window reclaimed.

### Q4: SessionEnd Strategy — Gate + Batch
5/6 converged. Gate trivial sessions with zero-cost heuristic, batch surviving hooks into 1 Sonnet call. `/feedback` bypasses gate. Data quality matters more than compute cost.

### Q5: Concurrency — SQLite WAL (Conditional)
6/6 unanimous. But: no evidence of current corruption. Instrument first, migrate only if proven. `kai state` CLI preserves inspectability.
