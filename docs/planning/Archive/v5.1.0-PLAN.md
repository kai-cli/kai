## Step 0: Fix Broken KnowledgeSync Import (BLOCKER)

KnowledgeSync.hook.ts line 22 imports `./lib/config-loader` — that file does not exist in pai-config. It's a KAI_ONLY_FILE (sync-to-kai.sh line 118). The restore commit (`e8ce68e`) refactored KnowledgeSync to use config-loader without creating the file. Running `bun hooks/KnowledgeSync.hook.ts` fails with `Cannot find module`.

**Fix:** Create `hooks/lib/config-loader.ts` for pai-config with `loadDomainKeywords()`, `loadDomainDescriptions()`, `loadProjectMapping()`, `loadExcludedProjects()`, `getMaxDomainsPerSession()` — all reading from `config/domains.jsonc` with graceful fallbacks. This also enables C1 and C2.

Note: KnowledgeHarvester (`PAI/Tools/KnowledgeHarvester.ts`) has its own hardcoded `DOMAIN_DEFINITIONS` and does NOT use config-loader either. Consider migrating it as part of C1 or leaving it as-is (it's a manual tool, not a hook).

**File:** `hooks/lib/config-loader.ts` (create new)
**Unblocks:** KnowledgeSync (immediate), C1, C2

**KAI_ONLY decision (must resolve before Step 0 ships):**
`hooks/lib/config-loader.ts` is currently in `KAI_ONLY_FILES` (sync-to-kai.sh:118), meaning kai has its own version and pai-config's copy would never sync. Once Step 0 creates it here, there are two options:

- **Option A — Keep KAI_ONLY:** pai-config and kai maintain separate versions. Changes here don't propagate. Risk: drift as the two configs diverge over time.
- **Option B — Remove from KAI_ONLY_FILES:** pai-config's version becomes canonical and syncs to kai on next push. This is cleaner if the interfaces are identical (same 5 function signatures, same fallback behavior).

**Decision (resolved 2026-04-29): Option A — keep KAI_ONLY.** Kai's version has evolved separately: it has a module-level cache, `_resetCache()`, `loadDomainDefinitions()`, `loadDomainsConfig()`, and `parseJSONC()` beyond the 5 shared functions. Syncing pai-config's version would clobber these. The divergence is documented in sync-to-kai.sh with a comment on the KAI_ONLY entry.

### S2. `PAI/` vs `skills/PAI/` divergence — document and guard
38 vs 37 .md files with genuinely different content in each — this is an active split, not a stale mirror. Full merge/consolidation is out of scope for v5.1.0.

**v5.1.0 scope:**
1. Add `PAI/README.md` documenting the intended split (PAI/ = runtime system + tools + algorithm; skills/PAI/ = skill discovery and workflow docs)
2. Add a section to `scripts/verify-release.sh` that flags basenames appearing in one directory but not the other.

**S2 guard implementation spec:**
The check compares basenames (filename only, no path) across `PAI/*.md` and `skills/PAI/*.md`. A new file added to one side without the other triggers a failure. Implementation:

```bash
pai_basenames=$(find PAI/ -maxdepth 1 -name '*.md' | xargs -I{} basename {})
skills_basenames=$(find skills/PAI/ -maxdepth 1 -name '*.md' | xargs -I{} basename {})
pai_only=$(comm -23 <(echo "$pai_basenames" | sort) <(echo "$skills_basenames" | sort))
skills_only=$(comm -13 <(echo "$pai_basenames" | sort) <(echo "$skills_basenames" | sort))
```

Files in neither direction are fine. New asymmetric files that are **intentional** go in an allowlist constant at the top of the check. **Seed the allowlist from the current state** (all current asymmetric files are pre-approved; only new additions trigger failure). This avoids the check firing immediately on the existing 38 vs 37 divergence. Check content, not just counts — two directories can have the same count with completely different files.

**Current allowlist (verified 2026-04-28):**
```bash
# PAI-only (not in skills/PAI/):
PAI_ONLY_ALLOWLIST=(
  AISTEERINGRULES-EXAMPLES.md   # runtime steering examples, not a skill doc
  CONTEXT_ROUTING.md            # internal routing table, not user-facing
  MEMORY-CHANGELOG.md           # memory system history
  PRDFORMAT.md                  # Algorithm PRD format spec
)
# skills/PAI-only (not in PAI/):
SKILLS_PAI_ONLY_ALLOWLIST=(
  ARBOLSYSTEM.md
  BROWSERAUTOMATION.md
  DEPLOYMENT.md
  FEEDSYSTEM.md
  TERMINALTABS.md
)
```
Any basename outside these allowlists that appears in only one directory = CI failure.

**Full consolidation:** deferred to v5.2.0 once the intended purpose of each directory is established.
**Files:** `PAI/README.md` (update — already exists), `scripts/verify-release.sh` (add divergence guard section)
**Severity:** HIGH — divergence will worsen, but documenting intent + adding a guard is sufficient for now

### S4. Steering enforcement — deferred to v5.2.0
Current state: CLAUDE.md instructions say "plan means stop" but there's no mechanical enforcement. A session can blow through plan approval and self-execute.

**Why deferred:** Hook-based enforcement (PreToolUse:Edit/Write gate) has fundamental problems — "plan-mode state" has no representation in the system, and the hook would fire on every edit, requiring extremely precise state detection to avoid blocking normal work. False positives are session-breaking. This needs a design spike, not a quick fix.

**v5.1.0 interim:** Save a feedback memory documenting the compliance failure and the specific failure mode ("diagnosis flowing into obvious fix flowing into execution overrides the rule"). This is weak but avoids a design sinkhole.

**v5.2.0 scope:** Design spike on enforcement mechanism. Leading candidate: `UserPromptSubmit` hook that detects plan-presentation patterns in recent assistant output and injects a "waiting for approval" context reminder. Lower risk than PreToolUse:Edit blocking.
**Severity:** HIGH — but deferred with interim mitigation
