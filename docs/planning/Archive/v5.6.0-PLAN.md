## Feature A: Progressive Disclosure Memory

**Inspiration:** claude-mem (76K stars) — 3-layer progressive disclosure, 10x token savings.

**Problem:** MEMORY.md loads first 200 lines at session start regardless of relevance. As memory
grows, this creates bloat (irrelevant context loaded) or gaps (important memories past cutoff).

**Solution:** Three-layer architecture:

```
Layer 1: INDEX (always loaded, ≤50 lines)
├── One-line summaries of each memory topic
├── Loaded at every session start via LoadContext.hook.ts
└── Acts as routing table for deeper layers

Layer 2: TIMELINE (loaded explicitly)
├── Recent events, changes, temporal context
├── Loaded ONLY via /memory recent command (user-initiated)
├── Written by SessionEnd hook (append-only JSONL)
└── Last 7 days of activity summaries

Layer 3: DETAIL (loaded on demand)
├── Full memory content per topic
├── Loaded only when topic referenced in conversation
└── Existing topic files (debugging.md, api-conventions.md, etc.)
```

### Index Pruning Policy

The 50-line cap is enforced with a **priority eviction strategy**:

| Priority | Keep | Evict When |
|----------|------|-----------|
| P0 | Active project references | Never (user removes manually) |
| P1 | Feedback memories (behavioral rules) | Never (these prevent repeated mistakes) |
| P2 | Recent memories (≤30 days or referenced) | Reclassified to P3 when older than 30 days AND reference_count = 0 |
| P3 | Older unreferenced memories | When index exceeds 50 lines (lowest score evicted first) |

**P2 → P3 aging:** During the session-start metadata pass (same pass that updates
`last_accessed`), any P2 entry older than 30 days with `reference_count = 0` is reclassified
to P3. This makes it eviction-eligible on the next index overflow. Entries that have been
referenced at least once remain P2 indefinitely — active use prevents demotion.

**Entry metadata storage:** Sidecar file at `MEMORY/STATE/memory-meta.jsonl`:
```jsonl
{"file": "project_lwrt_firmware.md", "last_accessed": "2026-05-19", "reference_count": 7, "priority": "P0"}
{"file": "feedback_local_context_first.md", "last_accessed": "2026-05-15", "reference_count": 3, "priority": "P1"}
{"file": "reference_hook_internals.md", "last_accessed": "2026-05-10", "reference_count": 1, "priority": "P2"}
```

**Why sidecar, not inline in MEMORY.md:** MEMORY.md is read by Claude Code's native auto-memory
system which expects clean markdown. HTML comments or extended syntax would pollute the format.
A sidecar JSONL is invisible to the memory reader, trivial to update, and machine-parseable.

**Metadata update triggers (two access types):**
- **Index access** (entry appears in loaded index at session start): updates `last_accessed`
  only. Triggered by LoadContext.hook.ts during session start. Lightweight signal.
- **Detail access** (model reads the detail file): updates `last_accessed` AND increments
  `reference_count`. Triggered by `MemoryAccessTracker.hook.ts` (PostToolUse) which watches
  Read tool calls and checks if the target path matches a known memory detail file.
- `priority` assigned at creation (from section header) or manually overridden

**MemoryAccessTracker.hook.ts logic:**
- Fires on PostToolUse for Read tool
- Checks if the read path matches `~/.claude/projects/*/memory/*.md` (detail files)
- If match: updates the corresponding entry in memory-meta.jsonl
- Fast-path: skip if path doesn't contain `/memory/` (avoids overhead on non-memory reads)

**Why both types matter:** An entry surfaced in the index every day for 40 days but never
detail-read would have `last_accessed: today, reference_count: 0`. Its eviction score:
`(0 × -1) + (0 × 5) = 0`. Without the index-access update, that same entry would score
`(-40) + (0 × 5) = -40` and be evicted despite appearing useful. The index-access update
ensures frequently-surfaced entries aren't penalized for not needing detail reads.

**For migrated entries (no prior metadata):** All entries start with
`last_accessed: migration_date, reference_count: 0, priority: P2`. The eviction formula
then uses document order as tiebreaker for entries with identical scores (top = higher priority).

**Eviction mechanism:** When a new memory would push index past 50 lines:
1. Score all P3 entries by `(days_since_last_access × -1) + (reference_count × 5)`
2. Evict lowest-scoring P3 entry (move to detail layer only — not deleted, just unindexed)
3. If no P3 entries exist, warn user that index is full and suggest review

**The evicted entry remains accessible** in its detail file — it just stops loading at session
start. This is lossless pruning: data preserved, context reduced.

### Timeline Layer — Read Path

The timeline is **never loaded automatically**. Read triggers:
- User types `/memory recent` → loads last 7 days from timeline.jsonl
- User asks "what did we work on recently?" → model reads timeline.jsonl directly via Read tool
- No hook auto-loads it. No implicit injection. User-initiated only.

### Timeline Growth Management

**Growth estimate:** ~1 entry per session, ~200 bytes per entry = ~365 entries/year, ~73KB/year.
This is acceptable — no aggressive pruning needed.

**Trim-on-write policy (simple, in MemoryTimeline.hook.ts):**
- After appending a new entry, count total lines
- If lines > 500 (~18 months of daily use), truncate from the head (oldest entries removed)
- 500 entries is generous headroom; `/memory recent` only reads last 7 days anyway
- Trimmed entries are gone (not archived) — this is activity metadata, not knowledge

**Why 500, not 90 (7 days × ~13/day)?** Users may want to look back further than 7 days
occasionally ("what were we doing last month?"). 500 entries covers ~18 months of history
at typical usage rates. Beyond that, git log is a better tool for deep history.

### Migration (First-Run Behavior)

On first session after upgrade:
1. LoadContext reads existing MEMORY.md
2. If MEMORY.md exceeds 50 lines AND has no `## Index` header → trigger migration:
   a. Parse existing entries (each `- [Title](file.md) — description` line)
   b. Classify entries using **section headers as proxy for priority**:
      - Entries under `## Active References` → P0
      - Entries under `## Feedback` → P1
      - All other entries → P2 (no metadata exists to distinguish P2 from P3)
   c. Within P2, preserve **document order** (top = most recently added = highest priority)
   d. Write new MEMORY.md with the top 50 entries in priority order (P0 → P1 → P2 by position)
   e. Remaining entries are already in their detail files — just stop indexing them
3. Log migration event to timeline.jsonl
4. This is a one-time operation — subsequent sessions use the new format

**Why document order for P2:** Existing MEMORY.md entries have no `days_since_last_access` or
`reference_count` metadata. The eviction scoring formula can't run on data that doesn't exist.
Document order is the best available proxy — users naturally keep important items near the top.
After migration, new entries gain metadata (access timestamps) and the full eviction formula
applies going forward.

**No user confirmation required:** The migration is lossless (no data deleted, only unindexed).
If any important entry was evicted from the index, it remains in its detail file and can be
re-indexed by adding it back to MEMORY.md manually.

**Rollback:** If migration produces bad results, `git checkout -- MEMORY.md` restores prior state.

## Feature B: Instinct-Based Learning

**Inspiration:** ECC (187K stars) — "Continuous Learning v2" with instinct confidence scoring
and `/evolve` clustering. Also claude-reflect (1K stars) for correction detection patterns.

**Problem:** PAI captures learnings (InsightExtractor, WorkCompletionLearning, reflections JSONL)
but these are passive archives. They don't actively influence future behavior without the
Algorithm explicitly choosing FeedbackConsult. Learning happens but doesn't compound.

**Solution:** Instinct system — lightweight behavioral nudges that surface automatically:

```
User Message Received (UserPromptSubmit hook)
    ↓ (correction pattern matched + follows tool call)
Instinct Created/Reinforced
    ↓ (confidence scoring)
Surfaced at Session Start (if relevant + high confidence)
    ↓ (clustering over time)
/evolve → Formal Skill or CLAUDE.md Rule (user-invoked only)
```

### Instinct Schema

```jsonl
{
  "id": "inst_abc123",
  "text": "Use bun test --bail for faster feedback on first failure",
  "confidence": 0.85,
  "tags": ["testing", "workflow", "bun"],
  "created": "2026-05-19T16:00:00Z",
  "last_triggered": "2026-05-19T16:00:00Z",
  "last_decayed_at": "2026-05-19T16:00:00Z",
  "trigger_count": 3,
  "source": "correction|repetition|revert|rating",
  "context": "User said 'just run with --bail' after I ran full suite twice",
  "active": true
}
```

### Tag Assignment Strategy

Tags are assigned at instinct creation time using a **hybrid approach (deterministic + contextual)**:

**Mandatory tags (always assigned, zero-cost):**
1. **Project tag:** derived from cwd at capture time (e.g., `pai-config`, `linksys-wiki`)
2. **Directory tag:** immediate parent directory within project (e.g., `hooks`, `scripts`, `skills`)

**Content tags (keyword extraction from instinct text, zero-cost):**
3. Extract nouns/tools from the instinct text via simple heuristic:
   - Known tool names: `bun`, `git`, `npm`, `grep`, `curl`, `tsc`, etc. (hardcoded list of ~50)
   - Known domain terms: `testing`, `deployment`, `memory`, `hook`, `algorithm`, etc. (hardcoded list of ~30)
   - Match against instinct `text` field — any hit becomes a tag

**No LLM call for tagging.** Tags must be deterministic and free. The hardcoded vocabulary
approach is limited but sufficient for v5.6's tag-based clustering. If clustering quality
is poor, v5.7 can add embedding-based similarity (Feature C) to replace tag matching.

**Maximum 5 tags per instinct** (prevents dilution). Priority: project > directory > content.

**Example derivation:**
```
cwd: ~/Projects/pai-config/hooks/
text: "Use bun test --bail for faster feedback on first failure"

Tags: ["pai-config", "hooks", "bun", "testing"]
  - pai-config: from project name
  - hooks: from directory
  - bun: matched tool name in text
  - testing: matched domain term ("test" → "testing")
```

### Confidence Rules

- New instinct starts at 0.3
- Each repetition: +0.2 (capped at 1.0)
- Surfacing threshold: 0.5 (shown in context)
- Evolution threshold: 0.8 (eligible for /evolve)
- Decay: -0.1 per 30 days without trigger (floor 0.0)
- **Auto-archive at 0.0** — moved to `instincts-archived.jsonl`, no longer surfaced
- Hard cap: 100 active instincts maximum; at cap, lowest-confidence auto-archives

**Decay rationale (revised):** At -0.1/30 days, a 0.3 instinct archives in 3 months, a 0.8
instinct in 8 months. This is aggressive enough to prevent bloat while giving validated
instincts reasonable longevity. Auto-archive (not delete) means `/evolve` can still reference
them for pattern detection.

### Correction Detection — Conservative Patterns (v1)

**v1 ships regex-only. Semantic detection is a v5.7 enhancement.**

**High-confidence patterns (require 2+ signals to fire):**
```
Pattern 1: Explicit imperative correction
  Regex: /\b(no[,.]?\s+(don'?t|stop|never|quit|please don'?t))\b/i
  + Must follow a PAI tool call within last 3 messages
  + Scans full message (no ^ anchor — corrections often start with context:
    "for that last file, no don't...", "wait, stop doing that", "actually no...")
  → Without the "follows tool call" gate, "no, don't use Python" in abstract
    discussion would false-positive

Pattern 2: Repeated instruction (same session)
  Detection: User message contains instruction already given earlier in session
  Method: Exact substring match (≥20 chars) of a prior user message
  → Simple and precise — if user repeats themselves verbatim, it's a correction

Pattern 3: Low rating signal
  Detection: Rating hook captures ≤3 rating
  Method: Existing ratings.jsonl signal, extract session context for instinct text
  → Already implemented in capture pipeline, just needs instinct bridge
```

**Deferred to v5.7 (too complex/noisy for v1):**
- File-revert detection (requires file write ledger — see below)
- Semantic similarity of corrections (requires embedding infrastructure from Feature C)
- Tone analysis of user messages

**File-revert detection (v5.7 design note):** Requires a write ledger tracking which files PAI
modified in the current session. On user tool call that writes to the same file, diff against
PAI's version — if the diff reverts PAI's changes, that's a correction signal. Ledger would
be session-scoped (MEMORY/STATE/session-writes.jsonl) and cleared at session end.

### Disable Mechanism

**Feature flag in settings.json:**
```json
{
  "instincts": {
    "enabled": true,
    "surfaceAtStart": true,
    "captureCorrections": true,
    "maxSurfaced": 20
  }
}
```

If `instincts.enabled` is false, both capture and surfacing are disabled entirely.
If `instincts.surfaceAtStart` is false, capture continues but nothing is injected at start.
This allows "silent learning" mode where instincts accumulate without affecting sessions.

### /evolve Skill Specification

**Invocation:** User types `/evolve` (no auto-run in v5.6 — user-initiated only).

**Interface:**
```
/evolve              → Show instinct dashboard + cluster candidates
/evolve --promote    → Interactive: select cluster to promote to CLAUDE.md rule
/evolve --prune      → Interactive: review low-value instincts for archival
/evolve --stats      → Show instinct metrics (count, confidence distribution, age)
```

**Clustering algorithm (v1 — simple tag-based):**
1. Group instincts by shared tags (≥2 tags in common)
2. Within each group, find instincts with confidence ≥0.8 AND trigger_count ≥3
3. Present each qualifying group as a "cluster candidate"
4. User selects which to promote → AI drafts a CLAUDE.md rule or skill file
5. User approves/edits → written to target; source instincts archived

**Promotion target resolution:**
- Default: project-level CLAUDE.md (cwd's nearest CLAUDE.md or `.claude/CLAUDE.md`)
- If instinct tags span multiple projects → prompt user to choose target
- `--target global` flag writes to `~/.claude/CLAUDE.md` (cross-project behavioral rule)
- `--target skill` flag creates a new skill file in `.claude/skills/` instead of a CLAUDE.md rule
- User always confirms target before write (no silent promotions)

**v5.7 enhancement:** Replace tag-based clustering with embedding similarity (depends on Feature C).

### Instinct Decay Trigger

**Mechanism:** SessionStart check in LoadContext.hook.ts (not a cron job).

**Schema addition:** `last_decayed_at` field (ISO timestamp) tracks when decay was last applied.
This prevents double-application on frequent sessions.

**Logic:**
```typescript
// On session start, before surfacing:
const now = Date.now();
for (const instinct of activeInstincts) {
  const lastDecay = instinct.last_decayed_at
    ? new Date(instinct.last_decayed_at).getTime()
    : new Date(instinct.created).getTime();
  const daysSinceDecay = (now - lastDecay) / MS_PER_DAY;

  if (daysSinceDecay >= 30) {
    // Apply exactly the number of NEW 30-day periods since last decay
    const newPeriods = Math.floor(daysSinceDecay / 30);
    instinct.confidence = Math.max(0.0, instinct.confidence - (0.1 * newPeriods));
    instinct.last_decayed_at = new Date(now).toISOString();
    if (instinct.confidence === 0.0) {
      archiveInstinct(instinct);
    }
  }
}
```

**Why `last_decayed_at` not `last_triggered`:** Using `last_triggered` would recompute all
historical decay periods on every session start. With daily sessions and a 0.8 instinct
last triggered 90 days ago, the old code would apply 3 periods (−0.3) on EVERY session start
until the instinct is triggered again — archiving it in 3 sessions instead of 8 months.
`last_decayed_at` ensures each period is counted exactly once.

This runs at most once per session start, costs <5ms, and requires no external scheduler.

## Feature C: Embedding-Based Context Routing Fallback

**Inspiration:** Anthropic contextual retrieval research (67% failure reduction with hybrid
approach). Khoj's bi-encoder + cross-encoder pattern.

**Problem:** CONTEXT_ROUTING.md maps topics → file paths explicitly. When a query doesn't match
any rule, no context is loaded. The user must manually specify what to read.

**Solution:** Semantic fallback layer:

```
User Query
    ↓
Explicit Routing Rules (CONTEXT_ROUTING.md)
    ↓ (match found? → load directly)
    ↓ (no match? → fallback)
Embedding Similarity Search
    ↓ (above threshold? → load top-k chunks)
    ↓ (below threshold? → no injection, proceed without)
Context Injected
```

### Technical Decisions (resolved pre-build)

**Embedding model:** `jina-embeddings-v2-small-en` via `@xenova/transformers`
- 33M parameters (smaller than MiniLM), optimized for retrieval tasks
- Sequence length: 8192 tokens (handles full document sections)
- Better than MiniLM for technical/code content per MTEB benchmarks
- Alternative if performance insufficient: `nomic-embed-text-v1.5` (137M params, best open-source on MTEB)

**Similarity threshold: 0.45 (cosine)**
- Production semantic search typically uses 0.3-0.6
- 0.45 balances precision (avoids irrelevant injection) vs recall (actually fires)
- Configurable in settings.json — start at 0.45, tune based on observed hit rate
- Original 0.7 was too high — would produce near-zero fallback hits

**Chunk strategy: 200 tokens, 50-token overlap at section boundaries**
- Zero overlap is optimal for self-contained semantic units (per Chroma research)
- But knowledge files contain function definitions and multi-paragraph explanations that span boundaries
- Compromise: zero overlap WITHIN markdown sections, 50-token overlap ACROSS section boundaries
- Section detection via markdown headers (# ## ###) — natural semantic boundaries

**First-run model download handling:**
```
If index doesn't exist AND model not cached:
  1. LocalContextFirst skips fallback entirely (no blocking download)
  2. Logs warning to stderr: "Embedding model not installed. Run: bun scripts/EmbeddingIndex.ts --setup"
  3. Feature degrades gracefully to explicit-routing-only (current behavior)
  4. Model download happens only via explicit CLI invocation (user runs setup command)
```
No implicit 80MB download. No blocking session. User opts in via CLI.

**Scope of Indexing:**
- All files referenced in CONTEXT_ROUTING.md
- All files in MEMORY/KNOWLEDGE/
- All files in ~/Projects/Knowledge/ (if exists)
- NOT: source code, node_modules, git objects

**External Path Monitoring (~/Projects/Knowledge/):**

External paths outside the PAI config directory are tracked via a manifest file:
```
MEMORY/STATE/embeddings/manifest.jsonl
# Each line: {"path": "/abs/path", "mtime": 1716000000, "hash": "sha256_first8"}
```

`EmbeddingIndex.ts --incremental` walks all configured paths (internal + external), compares
mtime against manifest. Changed files are re-embedded; deleted files are removed from index.
This runs only on explicit CLI invocation (`bun scripts/EmbeddingIndex.ts --incremental`),
not automatically at session start.

**Known limitations (v5.6):**
- External files can go stale between explicit reindex runs. Automatic background reindexing
  is a v5.7 enhancement (could use a file watcher or periodic cron).
- **Bun + @xenova/transformers compatibility:** transformers.js relies on ONNX runtime which
  has had sporadic issues with bun's WASM implementation. If incompatible at build time,
  fallback options: (a) run embedding via `node` subprocess, (b) switch to `fastembed`
  (lighter ONNX wrapper with better bun support), or (c) pre-compute embeddings via node
  script and query the static index from bun. Verify compatibility in Phase 3 spike before
  committing to full implementation.

**Performance Budget:**
- Index build: ≤30s for full rebuild, ≤5s incremental
- Query latency: ≤200ms per fallback query
- Startup overhead: 0ms (Feature C has no session-start cost — fires mid-conversation only)

## LoadContext.hook.ts Integration Strategy

**Critical risk:** Features A and B modify LoadContext.hook.ts. Feature C does NOT — it operates
mid-conversation via a separate hook (see architectural decision below).

### Feature C Architecture Decision: Mid-Conversation Hook (Option A)

**Problem:** LoadContext runs at session start — before the user sends a message. There is no
query to embed against. Calling `semanticFallback(query)` with no query is architecturally
nonsensical.

**Resolution: Feature C operates via `LocalContextFirst.hook.ts` (PreToolUse), not LoadContext.**

Feature C's fallback triggers when:
1. The model attempts a Read tool call
2. The PreToolUse hook intercepts and applies **gating criteria** (see below)
3. If gating passes AND no explicit routing rule matches → run semantic fallback
4. Inject results as additional context before the tool executes

**Gating criteria (fast-path to avoid latency on non-context reads):**
```
SKIP (no fallback, pass through immediately) when:
  - Tool is NOT Read (Write, Edit, Bash, etc. always pass through)
  - Target path is inside the current project working tree (code reads, not knowledge)
  - Target path is a known existing file (not a speculative/exploratory read)
  - Embedding index doesn't exist (graceful degradation)

ENGAGE (run semantic fallback) when:
  - Tool IS Read
  - AND target path matches a knowledge path pattern (MEMORY/, Knowledge/, ~/Projects/Knowledge/)
  - AND the specific file doesn't exist OR the read appears exploratory
  - OR the model's recent messages contain a question about a topic with no routed context
```

This means the fallback fires on a narrow subset of Read calls — those targeting knowledge
areas where the model is exploring. Non-knowledge reads (source code, configs) pass through
with zero overhead.

This is architecturally correct because:
- There IS a query at this point (the model's tool call + recent conversation = intent signal)
- The gating ensures <1ms overhead on 95%+ of tool calls (simple path check)
- It integrates with the existing `LocalContextFirst.hook.ts` which already gates context lookups
- No startup latency impact whatsoever

**Revised hook ownership:**
- `LoadContext.hook.ts` → Features A and B only (session start: memory + instincts)
- `LocalContextFirst.hook.ts` → Feature C (mid-conversation: semantic fallback)

### Integration order (sequential, not parallel):

```
Phase 1 (Feature A): Modify LoadContext to load index-only memory
  └── Test: session starts with ≤800 tokens of memory context
  └── Merge point: replace memory loading section

Phase 2 (Feature B): Add instinct decay + surfacing AFTER memory load
  └── Test: instincts appear in session start context
  └── Merge point: new section after memory, before output assembly
  └── Dependency: needs A's reduced memory to stay within token budget

Phase 3 (Feature C): Modify LocalContextFirst to add embedding fallback
  └── Test: fallback fires when model seeks context not covered by routing rules
  └── Merge point: new fallback path in LocalContextFirst.hook.ts
  └── Dependency: graceful degradation if model not installed
  └── NO CONFLICT with Phase 1/2 (different file)
```

**Architectural approach:** Each feature adds a discrete, independently-testable function.
Each feature's logic lives in its own `hooks/lib/` file and is invoked as a single function
call from its respective hook:

```typescript
// LoadContext.hook.ts — final structure after Features A + B
const memoryContext = loadIndexMemory();           // Feature A
const instinctContext = surfaceInstincts();        // Feature B
output = assemble(memoryContext, instinctContext, existingDynamicContext);

// LocalContextFirst.hook.ts — final structure after Feature C
const explicitRoute = matchContextRoute(intent);   // Existing
if (!explicitRoute) {
  const fallback = semanticFallback(intent);       // Feature C
  if (fallback.confidence > threshold) {
    injectContext(fallback.content);
  }
}
```

## Pre-Build Validation (2026-05-20)

Four flags raised during plan review, all researched and resolved.

### Flag 1: MEMORY.md 200-Line Platform Behavior — Plan Still Adds Value

Claude Code's native behavior (confirmed May 2026):
- Loads first 200 lines (or 25KB) of MEMORY.md at session start — hard limit, not configurable
- Topic files loaded on-demand when Claude reads them
- Already a 2-layer progressive disclosure system

**Feature A still adds value:** shrinks index from 200→50 lines (5-8x token savings), adds eviction
scoring with priority classification (platform has none), adds timeline JSONL for session history
(platform doesn't track this). Feature A tightens and automates what the platform already supports.

### Flag 2: Correction Regex False Positives — Acceptable for v1

Pattern 1 + "follows tool call within last 3 messages" gate handles the main concern: code
discussions like "no, don't use rm -rf" when describing behavior won't fire because there's no
preceding PAI tool call the user is reacting to.

Residual edge cases (user says "no don't" about something else in a file PAI just wrote) are
acceptable because: confidence starts at 0.3, needs 2 reinforcements to reach surfacing (0.5),
one-off false positives decay and auto-archive, `/evolve --prune` handles cleanup.

The "< 20% false positive rate" success metric acknowledges imperfection. Semantic detection in
v5.7 addresses this properly.

### Flag 3: Bun + @xenova/transformers — Resolved, Pin v3.8.x

**Use `@huggingface/transformers@^3.8.1`** (formerly @xenova/transformers):
- Proven working in production Bun projects via onnxruntime-node native bindings (~12ms inference)
- `Xenova/jina-embeddings-v2-small-en` available in ONNX format (fp32, fp16, q8, q4)
- KAI runs hooks via `bun hookfile.ts` (standard execution), not `bun --compile`
- v4.x has `bun --compile` issues but v3.8.x is stable for standard execution

**Do NOT use:** v4.x (native addon bundling broken in Bun), WASM fallback (JSC OSR bugs), or
fastembed (accuracy concerns).

**Updated risk table entry:** "Bun + ONNX" risk is resolved. Phase 3 spike is a 5-minute
confirm-on-machine check, not a blocking uncertainty.

### Flag 4: TOTAL_STEPS in PAI-Install — Non-Issue

Current state: `TOTAL_STEPS = 9` with all 9 steps defined. Feature A doesn't add installer steps
(modifies LoadContext.hook.ts and adds memory-disclosure.ts). The installer already creates all
MEMORY/ directories. No conflict.
