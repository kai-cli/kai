# PAI v5.4.0 — Knowledge Intelligence + Memory Maturity

**Created:** 2026-05-15
**Theme:** On-demand retrieval, knowledge quality gates, lifecycle management, contradiction detection.
**Context:** v5.3.0 shipped PlanApprovalGuard, research mode, /end skill, and 529 tests.
KAI public release is v5.3.0. v5.4.0 matures the memory system — moving from "capture and inject"
to "retrieve, validate, and maintain." Inspired by gaps identified comparing against Daniel
Miessler's PAI knowledge graph approach, adapted to our filesystem-first philosophy.

**Prerequisite:** v5.3.0 shipped and KAI public release complete.

## Phase K: Knowledge Retrieval

The biggest gap: mid-session, if you need information from a domain that wasn't injected at
startup, there's no tool to find it. You have to know the path. This phase adds on-demand
retrieval without adding RAG/embeddings/vector stores.

### K1. `MemorySearch` tool — on-demand knowledge retrieval

**Problem:** LoadContext injects context at session start by project directory matching.
Mid-session queries about other domains hit a wall.

**Solution:** A CLI tool that searches KNOWLEDGE/ + CAPABILITIES/ + LEARNING/ using ripgrep,
ranks results by relevance, and returns a compressed summary fitting a token budget.

**Design:**
```
bun run PAI/Tools/MemorySearch.ts "openwrt feed compilation"
```

1. **Search pass:** `rg --json` across MEMORY/KNOWLEDGE/, MEMORY/CAPABILITIES/, CAPABILITIES.md
   with query terms. Score by match density + file recency.
2. **Rank:** Sort by score. Take top 5 files.
3. **Extract:** Pull the matching paragraphs (context window around matches).
4. **Budget:** If total exceeds budget (default 4000 chars), truncate lowest-ranked results.
5. **Output:** Structured markdown with file paths and relevant excerpts.

**Integration points:**
- Skill registration: `skills/PAI/Search/SKILL.md` — triggers on "search memory",
  "what do I know about", "check knowledge"
- Algorithm OBSERVE phase: optional auto-search when task domain doesn't match injected context
- Board API: `/api/search?q=` endpoint for UI search

**Files:**
- `PAI/Tools/MemorySearch.ts` (new, ~150 LOC)
- `skills/PAI/Search/SKILL.md` (new)
- `skills/PAI/Search/Workflows/SearchMemory.md` (new)
- `tests/MemorySearch.test.ts` (new, ~15 tests)

**Not building:** LLM-based compression (too slow for a retrieval tool). Embeddings. Vector stores.
Fast text search with ranked excerpts is sufficient for file-based knowledge at our scale.

### K2. Related-note linking

**Problem:** Knowledge files are isolated. Finding `firmware.md` doesn't lead you to
`usp.md` or `openwrt-sdk.md` even though they're deeply related.

**Solution:** Add `related:` frontmatter field to knowledge files. MemorySearch follows links
one hop deep — if your search hits `firmware.md`, it also surfaces the `related:` targets.

**Format:**
```yaml
---
domain: firmware
updated: 2026-05-15
tags: [openwrt, embedded, router]
related: [usp-and-standards, github-workflows]
---
```

**Validation:** A tool checks that all `related:` targets resolve to actual files.
Run during `pai curate` or on knowledge file modification.

**Files:**
- Schema definition in `MEMORY/KNOWLEDGE/_schema.md` (new)
- `PAI/Tools/KnowledgeLinker.ts` (new, ~80 LOC) — validates links, suggests new ones
- Migrations: Add frontmatter to existing KNOWLEDGE/ files

## Phase V: Validation + Quality Gates

### V1. Knowledge frontmatter schema

**Problem:** KNOWLEDGE/ files are freeform. No way to enforce consistency, detect incomplete
entries, or query by metadata.

**Required frontmatter:**
```yaml
---
domain: string          # Category (firmware, usp, flutter, devops, pai-system)
updated: YYYY-MM-DD    # Last meaningful update
tags: string[]         # 2-6 keyword tags for cross-referencing
related: string[]      # Slugs of related knowledge files (optional initially)
---
```

**Validation runs:**
- In `pai curate` before promoting STAGING → knowledge
- As a standalone tool: `bun PAI/Tools/ValidateKnowledge.ts`
- Report: missing fields, stale dates (>180 days), orphan tags

**Files:**
- `PAI/Tools/ValidateKnowledge.ts` (new, ~100 LOC)
- `tests/ValidateKnowledge.test.ts` (new, ~12 tests)

### V2. Contradiction detection (lightweight)

**Problem:** Nothing stops conflicting facts across knowledge files. "SDK uses kernel 5.15"
in one file while another says 6.1. Stale facts persist silently.

**Approach:** Tag-overlap analysis. For files sharing ≥2 tags, extract versioned claims
(regex: thing + version/date pattern) and flag conflicts.

**Design:**
- Scan all KNOWLEDGE/ files, group by tag overlap
- Within each group, extract "versioned facts" — lines containing version numbers, dates, or
  specific quantities attached to a named entity
- Compare: if the same entity has different values across files, flag as potential contradiction
- Output: Report file, not auto-fix. User reviews.

**Scoping:** This is NOT an AI inference task. It's mechanical: "file A says X=5.15,
file B says X=6.1, both tagged [openwrt]." Simple and useful.

**Files:**
- `PAI/Tools/ContradictionDetector.ts` (new, ~120 LOC)
- Integrated into `pai curate --check` flow
- `tests/ContradictionDetector.test.ts` (new, ~10 tests)

## Phase L: Lifecycle Management

### L1. Reference tracking (`last_referenced`)

**Problem:** Knowledge files accumulate forever. No signal for "actively used" vs "written once
and forgotten."

**Solution:** Use existing `memory-reads.jsonl` telemetry (already written by LoadContext at
line 622) to track which domains were served. Add a periodic analysis tool.

**Implementation:**
- `PAI/Tools/KnowledgeHealth.ts` — reads `memory-reads.jsonl`, computes per-domain:
  - Last referenced date
  - Reference count (last 30/90 days)
  - Trend (increasing/stable/declining)
- Surfaces in Board UI (`/api/knowledge-health`) and in `pai curate` output
- Files unreferenced for 90+ days get flagged (not deleted, not archived — flagged)

**Files:**
- `PAI/Tools/KnowledgeHealth.ts` (new, ~100 LOC)
- `tests/KnowledgeHealth.test.ts` (new, ~10 tests)
- Board integration: add `/api/knowledge-health` endpoint

### L2. Auto-consolidation (smart STAGING promotion)

**Problem:** `pai curate` is manual. STAGING drafts rot. We added a nudge but actual
consolidation still requires human intervention.

**Solution:** Automatic promotion for high-confidence patterns.

**Rules for auto-promotion (all must be true):**
- STAGING entry has been pending ≥14 days
- Pattern appears in ≥3 separate session captures
- No conflicting entries exist in WISDOM/FRAMES/
- Auto-promoted entries get a `source: auto-promoted` tag for later review

**Rules for auto-dedup:**
- STAGING entries with >80% text similarity → merge into single entry, keep highest count

**Guard rails:**
- Maximum 3 auto-promotions per consolidation run
- Log all auto-promotions to `MEMORY/STATE/auto-promotions.jsonl` for audit
- Never auto-promote entries containing "maybe", "might", "not sure" (low confidence)

**Trigger:** Runs in `SessionCleanup.hook.ts` (same place as `maybeRunSynthesisBackstop`).
Checks once per day.

**Files:**
- `PAI/Tools/AutoConsolidate.ts` (new, ~150 LOC)
- `hooks/SessionCleanup.hook.ts` — add `maybeAutoConsolidate()` call
- `tests/AutoConsolidate.test.ts` (new, ~15 tests)

### L3. SessionCloseGuard (deferred from v5.3.0)

**Problem:** `/end` requires explicit invocation. Natural-language exit patterns ("ok that's it",
"done for now", "wrapping up") should trigger the same close flow.

**Design:** Deferred from v5.3.0 A2 — see `docs/planning/steering-enforcement-design.md`.
After /end has been used for a few weeks and we have real usage patterns, implement the
implicit detection layer.

**Depends on:** Data from /end usage in v5.3.0 — what phrases precede it, what the
false-positive risk is for implicit detection.

## What We're NOT Building (and Why)

- **Vector store / embeddings:** Our knowledge base is <100 files. ripgrep at 1ms/query
  is faster than any embedding lookup. Revisit if we hit 1000+ files.
- **Full knowledge graph with BFS traversal:** One-hop `related:` links cover 90% of
  discoverability needs. Graph traversal adds complexity without proportional value at our scale.
- **LLM-powered compression in retrieval:** Adds 2-5s latency per search. Token budget
  trimming (cut lowest-ranked results) is instant and good enough.
- **Auto-Dream (server-side consolidation):** We don't have a persistent server process.
  SessionCleanup runs at session end — that's our consolidation window. Good enough.
- **Note lifecycle auto-archival:** Daniel archives after 90 days. We flag. Deletion/archival
  is a user decision — visibility is the system's job.

## Architectural Review (2026-05-15)

Full review of both v5.3.0 and v5.4.0 plans, examining whether the proposed solutions are
architecturally coherent, whether we're building the right things, and where gaps remain.

---

### The Fundamental Question: What Are These Plans Trying to Do?

**v5.3.0** = "Ship what we have, harden it, add one new capability (research mode)"
**v5.4.0** = "Make knowledge a first-class system instead of a bag of files"

Both are correct directions. But v5.4.0 has an architectural incoherence: it proposes 6
new TypeScript tools (MemorySearch, ValidateKnowledge, ContradictionDetector, KnowledgeHealth,
KnowledgeLinker, AutoConsolidate) that all operate on the same data (KNOWLEDGE/ files) but
don't share a data model. Each tool will re-parse frontmatter independently, define its own
types, and have its own understanding of what a "knowledge file" is.

**The missing abstraction is a shared knowledge schema layer.**

---

### Architectural Gap #1: No Unified Schema (the YAML problem you sensed)

Currently: `domains.jsonc` defines keywords/mapping. `knowledge-readback.ts` reads files.
`KnowledgeHarvester.ts` writes files. `KnowledgeSync.hook.ts` updates files. None of them
share a type definition for what a knowledge file IS.

v5.4.0 proposes adding frontmatter to KNOWLEDGE/ files (V1) but treats it as a validation
problem. It should be treated as a **data model** problem:

**Proposed: `hooks/lib/knowledge-schema.ts`** (~60 LOC)

```typescript
// The canonical type for a knowledge file
export interface KnowledgeFile {
  // Frontmatter (YAML)
  meta: {
    domain: string;
    updated: string;       // YYYY-MM-DD
    tags: string[];
    related?: string[];    // slugs of related knowledge files
    sections?: string[];   // auto-populated from ## headings
  };
  // Content
  body: string;
  // Derived (computed at load time, not persisted)
  path: string;
  slug: string;            // filename without extension
  headings: string[];      // extracted ## headers
}

// Parse a knowledge file from disk → typed object
export function parseKnowledgeFile(filePath: string): KnowledgeFile | null;

// Validate frontmatter completeness (returns errors)
export function validateKnowledgeMeta(meta: unknown): string[];

// Write a knowledge file back to disk (preserves body, updates frontmatter)
export function writeKnowledgeFile(file: KnowledgeFile): void;

// Load all knowledge files as typed objects
export function loadAllKnowledge(): KnowledgeFile[];
```

Every v5.4.0 tool imports from this one module. No re-parsing. No divergent type defs.
This is the 60 LOC that makes the other 600 LOC coherent.

**Why this matters for YAML:** Right now `domains.jsonc` defines domains, but the knowledge
files themselves have no structured metadata. Adding YAML frontmatter without a parsing layer
means every consumer re-implements frontmatter extraction. The schema module parses once,
validates once, provides typed access everywhere.

---

### Architectural Gap #2: The KnowledgeHarvester → KNOWLEDGE/ Pipeline Is Invisible

The plan assumes knowledge files are written by KnowledgeHarvester and consumed by
knowledge-readback. But looking at the actual system:

1. `KnowledgeHarvester.ts` scans ALL project memory files, uses LLM to distill → writes KNOWLEDGE/
2. `KnowledgeSync.hook.ts` runs at SessionEnd, checks mtimes, re-distills changed domains
3. `knowledge-readback.ts` reads files at session start by matching project → domain

**Problem:** KnowledgeHarvester writes freeform markdown (no frontmatter). KnowledgeSync
writes freeform markdown (same). If v5.4.0 adds frontmatter as a requirement, **the writers
need to be updated too** — not just the validators/readers.

**Missing from v5.4.0 plan:**
- Update `KnowledgeHarvester.ts` to emit frontmatter when generating domain files
- Update `KnowledgeSync.hook.ts` to preserve/update frontmatter when re-distilling
- Migration script: add frontmatter to existing 7 files (one-time)

Without this, you build a validator that immediately reports 7 failures and no tool produces
valid output. The validator validates nothing until the writers are fixed.

---

### Architectural Gap #3: `domains.jsonc` Should Be the Single Source of Truth

Currently:
- `domains.jsonc` defines domain names, keywords, project mappings
- `KnowledgeHarvester.ts` has BUILTIN_KEYWORDS fallback
- `KnowledgeSync.hook.ts` has BUILTIN_KEYWORDS fallback
- `knowledge-readback.ts` reads project mapping from config-loader

v5.4.0 adds:
- YAML frontmatter with `domain:` field on each knowledge file
- `tags:` field on each knowledge file
- `related:` field on each knowledge file

**Problem:** Now domain identity is defined in three places:
1. `domains.jsonc` (the project-to-domain mapping)
2. Knowledge file frontmatter (the file's self-declared domain)
3. BUILTIN fallbacks in tools

**Fix:** `domains.jsonc` becomes the authoritative schema:

```jsonc
{
  "definitions": {
    "firmware": {
      "description": "...",
      "keywords": [...],
      "requiredTags": ["openwrt", "embedded"],  // tags that MUST appear on this domain's file
      "relatedDomains": ["api-and-services"]    // expected cross-links
    }
  }
}
```

Then the knowledge schema module reads `domains.jsonc` to know what valid domains are.
Validation becomes: "does this file's `domain:` field match a key in `definitions`?"
Related-note linking becomes: "does this file's `related:` contain only valid domain slugs?"

One config file rules them all. No BUILTIN fallbacks needed.

---

### Architectural Gap #4: v5.3.0 Test Strategy Is Feature-Centric, Not Boundary-Centric

v5.3.0 Phase T lists functions to test:
- `trimOldEntries()`
- `synthesisToStagingContent()`
- `loadRatings(days?)`
- `maybeRunSynthesisBackstop()`

These are internal functions. But the **system boundaries** that actually break in production are:

1. **Config mutation boundary:** What happens when `BuildSettings.ts` receives malformed JSONC?
   (D3 partially addresses this, but no test plan for it)
2. **Hook stdin boundary:** What happens when hook receives malformed JSON from Claude Code?
   (The `payload-schema.ts` handles parsing, but only `PayloadSchema.test.ts` tests it)
3. **Inference budget boundary:** What happens when inference is called but budget is exhausted?
   (InferenceBudget.test.ts exists — good)
4. **File staleness boundary:** What happens when `memory-reads.jsonl` grows unbounded?
   (No rotation policy, no test for degraded performance at scale)
5. **Concurrent hook execution:** What happens when SessionEnd fires 7 async hooks simultaneously
   and multiple try to write to the same STATE/ files? (No test, real race condition risk)

**Missing test cases for v5.3.0:**
- `BuildSettings.ts` with corrupted/partial JSONC input
- `PlanDetection.ts` with edge cases: markdown that looks like a plan but isn't (README with
  checkboxes), plan split across multiple messages
- `deliberate.ts --mode research` with partial model failures (1 of 3 models times out)
- `/end` skill with empty session (no PRDs, no STAGING, no commits)

**Missing test cases for v5.4.0:**
- `MemorySearch.ts` with special characters in query (regex injection via ripgrep)
- `MemorySearch.ts` ranking stability (same query, same files = same order)
- `ContradictionDetector.ts` with temporal versioning (v1.12 → v1.18 is update, not conflict)
- `AutoConsolidate.ts` concurrent execution guard (two sessions ending simultaneously)
- `ValidateKnowledge.ts` with malformed YAML frontmatter (unclosed quotes, tabs vs spaces)
- Knowledge file read/write roundtrip: parse → modify → write → parse again = identical

### Architectural Gap #5: No Data Flow Diagram

The v5.4.0 plan describes tools in isolation. What's missing is the data flow:

```
SESSION START:
  domains.jsonc → knowledge-readback.ts → LoadContext.hook.ts → injected context

MID-SESSION (new in v5.4.0):
  user query → MemorySearch.ts → rg search → rank → extract → budget trim → output
                                    ↓ (one-hop follow)
                              related: links → additional files

SESSION END:
  project memory files → KnowledgeSync.hook.ts → LLM distill → KNOWLEDGE/*.md
                                                                     ↓
                                                        frontmatter preserved/updated

PERIODIC (manual or cron):
  pai curate → ValidateKnowledge.ts → report
             → ContradictionDetector.ts → report
             → KnowledgeHealth.ts (reads memory-reads.jsonl) → report
             → AutoConsolidate.ts → STAGING promotions

HARVEST (weekly):
  KnowledgeHarvester.ts → scan all project memory → LLM distill → KNOWLEDGE/*.md
```

This diagram reveals a dependency the plan doesn't make explicit:
**KnowledgeSync (SessionEnd) and KnowledgeHarvester (manual) both WRITE knowledge files.**
If both produce frontmatter, they need to agree on the schema. If one overwrites the other's
frontmatter, you get silent data loss. The knowledge-schema.ts shared module solves this.

### v5.3.0 Specific Gaps

**1. Research mode has no test harness for live API calls.**
`deliberate.ts --mode research` calls Gemini and Grok APIs. Tests can't hit live APIs.
Plan says nothing about mocking strategy. Either:
- Add `--dry-run` flag that returns fixture responses (test the synthesis logic)
- Or accept that research mode is only tested manually (document this explicitly)

What if the user says "yes, but also change X"? That's approval + new instruction in one

- "go ahead" → approve ✓

- "yes?" → question, not approval (has ?)
- "do it, and also add logging" → approve + extend

**3. `/end` skill depends on state that may not exist.**
It reads "active PRD, STAGING count, memory written, ratings captured." But in a short
session where you just asked a question and got an answer, none of these exist. The skill
needs a "nothing notable happened this session" graceful path. Plan doesn't mention it
explicitly.

**4. Phase S (release prep) is tangled with Phase A/R/T.**
If any coding phase slips, release prep is blocked. But S1-S3 (CONTEXT_ROUTING rewrite,
stale refs, README attribution) are independent of code. They could ship as a separate
pre-release commit that doesn't require A/R/T to be complete. Consider decoupling.

### v5.4.0 Specific Gaps

**1. MemorySearch ranking without metadata is just grep.**
The plan says "Score by match density + file recency." But with 7 files of ~200 lines each,
ripgrep will return matches in most of them for any non-trivial query. Without frontmatter
tags to weight by, the ranker is just "count matches per file." That's barely better than grep.

This reinforces: **V1 (schema + frontmatter) must precede K1 (search).**

**2. Board integration is mentioned but not scoped.**
K1 says "Board API: `/api/search?q=` endpoint for UI search." L1 says "Surfaces in Board UI."
But there's no Board phase. No LOC estimate. No file list. This is either:
- Trivial (add 2 endpoints to `scripts/board.ts`) — say so explicitly
- Non-trivial (UI component changes, search input, results display) — scope it

**3. Auto-consolidation has a philosophical tension with "flag, don't act."**
The "What We're NOT Building" section says "Note lifecycle auto-archival: Daniel archives
after 90 days. We flag." But L2 auto-promotes STAGING entries without user approval (just
with guard rails). These two positions are in tension. Either:
- Auto-consolidation is OK because it's STAGING→WISDOM (low stakes, reversible)
- Or it violates the "system flags, user decides" principle

The guard rails (3 max, ≥3 captures, no uncertainty language) partially resolve this, but
the plan should acknowledge the tension and state explicitly: "auto-promotion is acceptable
for STAGING because STAGING was already system-generated and just awaiting confirmation."

**4. L3 (SessionCloseGuard) shouldn't be in v5.4.0.**
It's listed as "deferred from v5.3.0" and "depends on data from /end usage." If v5.3.0
hasn't shipped yet, there's no usage data. L3 is not implementable until v5.3.0 has been
live for weeks. Move to v5.5 backlog or mark as "conditional: only if /end has 50+ uses
by the time L1/L2 are done."

### On YAML: You're Right, But It's a Data Model Problem, Not a Format Problem

The instinct that "we should use YAML better" is correct but the real issue isn't YAML —
it's that the system has config data in multiple places with no unifying type system:

- `domains.jsonc` → JSONC, defines domains
- `preferences.jsonc` → JSONC, defines runtime behavior
- KNOWLEDGE/ files → markdown, no metadata
- `.harvest-state.json` → JSON, tracks file mtimes
- `memory-reads.jsonl` → JSONL, tracks read telemetry
- `STAGING/*.md` → markdown with DraftMemory interface (defined in lib/staging.ts)

The fix isn't "more YAML" — it's "one shared type per entity that every tool imports."
YAML frontmatter is the right format for knowledge file metadata (human-readable, git-friendly,
standard). But the TypeScript types backing it should live in one module, not be re-derived
by each consumer.

`config/domains.jsonc` is actually well-designed — it's a single source for domain config
that `config-loader.ts` exposes. The knowledge layer just needs the same pattern:
one schema module that everything imports.
