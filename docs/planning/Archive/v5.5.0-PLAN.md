# PAI v5.5.0 — Algorithm Maturity + Structural Integrity

**Created:** 2026-05-15
**Theme:** Effort-tier formalization, closed-list capabilities, ISC quality rules, containment zones, unified daemon.
**Context:** Inspired by Daniel Miessler's PAI v5.0.0 (Algorithm v6.3.0, ISA system, Pulse daemon,
containment zones). After v5.4.0 matures retrieval and knowledge quality, v5.5.0 raises the
Algorithm and system architecture to match. Cherry-picks the best ideas from Daniel's system
while keeping our lighter-weight, practical approach.

**Prerequisite:** v5.4.0 shipped (knowledge retrieval, validation, lifecycle management).

**Source analysis:** Daniel's repo at github.com/danielmiessler/Personal_AI_Infrastructure.
Full comparison done 2026-05-15. His system is more comprehensive (45 skills, 171 workflows,
Sonnet-backed classifier, 19 closed-list capabilities) but also heavier. We take the
architectural insights without the weight.

## Phase A: Algorithm Enhancement

### A1. Effort Tier Completeness Gates

**Problem:** Our Algorithm has effort levels (Standard/Extended/Advanced/Deep) but no formal
requirement for what the PRD/ISA must contain at each level. A "Deep" run should demand more
upfront rigor than a "Standard" one, but currently both produce the same artifact structure.

**Daniel's approach:** E1-E5 tiers with hard completeness gates:
- E1 (<90s): Goal + Criteria only
- E2 (<10m): Problem, Goal, Criteria, Test Strategy
- E3 (<30m): Most sections
- E4 (<2h): All 12 sections
- E5 (2h+): All 12 + Interview workflow before BUILD

**Our adaptation:** Map to our existing effort levels with practical minimums:

| Our Level | Equivalent | PRD Minimum Required |
|-----------|-----------|---------------------|
| Standard | E1-E2 | Goal + Criteria (≥3 ISC) |
| Extended | E3 | Problem, Goal, Criteria (≥8 ISC), Constraints |
| Advanced | E4 | Problem, Vision, Goal, Criteria (≥15 ISC), Constraints, Test Strategy, Decisions |
| Deep | E5 | All sections + deliberate/research pass before BUILD |

**Implementation:**
- Add completeness check to Algorithm PLAN→BUILD gate
- If PRD doesn't meet minimum for current effort level, block BUILD and prompt for missing sections
- Soft block (additionalContext reminder), not hard block (don't break flow)

**Files:**
- `PAI/Algorithm/v3.14.0.md` (or next version) — add completeness gate to PLAN phase
- `hooks/handlers/PlanDetection.ts` — extend to detect under-specified PRDs at high effort
- `tests/AlgorithmCompleteness.test.ts` (new, ~12 tests)

### A2. Closed-List Thinking Capabilities

**Problem:** The model occasionally references thinking modes that don't exist in our system
("let me use my Deep Analysis capability" when no such thing is defined). Daniel calls these
"phantom capabilities" and treats them as a CRITICAL FAILURE.

**Daniel's list (19):** IterativeDepth, ApertureOscillation, FirstPrinciples, SystemsThinking,
RootCauseAnalysis, Council, RedTeam, Science, BeCreative, Ideate, BitterPillEngineering, Evals,
WorldThreatModel, Fabric patterns, ContextSearch, ISA, Advisor, ReReadCheck, FeedbackMemoryConsult

**Our adaptation:** Enumerate what we actually have and inject the closed list into the Algorithm.
Our capabilities map to skills + tools, not invented thinking modes:

**Proposed closed list (ours):**
- `Deliberate` — multi-model debate or research (skill)
- `Research` — multi-source web research (skill)
- `Agents/Council` — parallel perspectives via composed agents (skill)
- `FirstPrinciples` — deconstruct/rebuild reasoning (thinking mode)
- `RootCause` — systematic cause analysis (thinking mode)
- `RedTeam` — adversarial challenge of own conclusions (thinking mode)
- `ISC` — ideal state criteria articulation (Algorithm native)
- `VerifyByRunning` — live-probe verification (doctrine)
- `FeedbackConsult` — check LEARNING/ signals before approach (thinking mode)

**Rule:** "If a capability is not on this list, you don't have it. Never reference, invoke,
or imply capabilities not enumerated here."

**Implementation:**
- Add closed list to Algorithm file (OBSERVE phase preamble)
- Add to CLAUDE.md under a `## Available Capabilities` section
- No hook needed — this is a prompt-level constraint

**Files:**
- `PAI/Algorithm/v3.14.0.md` — add capabilities enumeration
- CLAUDE.md — add Available Capabilities section

### A3. Verification Doctrine Formalization

**Problem:** We have `feedback_verify_by_running` as a learned behavior, but it's not formally
part of the Algorithm. The model sometimes still asserts completion without tool-based evidence.

**Daniel's doctrine (4 rules):**
1. Never assert without tool-based evidence
2. Live-probe required (not just reading code — run it)
3. At E4/E5, cross-vendor audit (Cato)
4. Verification section in ISA records what was actually verified

**Our adaptation:**
- Formalize "verify by running" as a VERIFY phase hard requirement
- Add verification recording to PRD (what was tested, how, result)
- Skip cross-vendor audit (we don't have Cato) — but require deliberate for Deep effort

**Implementation:**
- Algorithm VERIFY phase: mandate at least one tool-based probe per ISC marked complete
- PRD template: add `## Verification Log` section (auto-populated by Algorithm)
- Hook: `CheckpointPerISC` concept — auto-commit when ISC transitions to complete

**Files:**
- `PAI/Algorithm/v3.14.0.md` — strengthen VERIFY phase
- PRD template update
- Optional: `hooks/VerificationLog.hook.ts` (PostToolUse, writes verification events)

## Phase I: ISC Quality Rules

### I1. ID-Stability Rule

**Problem:** When criteria are edited, added, or split during a session, IDs sometimes shift.
ISC-C3 becomes ISC-C4 if something is inserted above it. This breaks task tracking and
creates confusion in the Algorithm state.

**Daniel's rule:** ISC IDs never re-number on edit. Splits become ISC-N.M (e.g., ISC-C3.1, ISC-C3.2).
Deletions leave gaps. New additions get the next available number.

**Implementation:**
- Document in Algorithm file and PRD template
- AlgorithmTracker: validate ID format allows sub-numbering (ISC-C3.1)
- `parseCriterion()` regex update to handle dotted IDs

**Files:**
- `PAI/Algorithm/v3.14.0.md` — add ID-stability rule
- `hooks/AlgorithmTracker.hook.ts` — update CRITERION_PATTERNS for dotted IDs
- `tests/HookFunctions.test.ts` — add parseCriterion tests for ISC-C3.1 format

### I2. Mandatory Anti-Criterion

**Problem:** PRDs define what to build but rarely define what NOT to build. Scope creep happens
because there's no explicit boundary. Daniel requires at least 1 anti-criterion at every tier.

**Anti-criterion format:** `ISC-A1: System does NOT do X` — a testable negative constraint.

**Our adaptation:**
- Algorithm PLAN phase: require ≥1 anti-criterion (A-prefixed ISC) in every PRD
- AlgorithmTracker already tracks A-prefixed criteria — just enforce minimum count
- effortLevelUpdate logic can check: if criteria.filter(c => c.type === 'anti-criterion').length === 0, inject reminder

**Implementation:**
- Algorithm PLAN→BUILD gate: check for anti-criteria presence
- Soft reminder via additionalContext if missing (not a hard block)

**Files:**
- `PAI/Algorithm/v3.14.0.md` — add anti-criterion requirement
- Optional: `hooks/handlers/PlanDetection.ts` — detect PRD with 0 anti-criteria

### I3. Atomicity Requirement ("one binary probe per ISC")

**Problem:** Compound criteria like "ISC-C5: Tests pass and coverage exceeds 80%" are hard to
verify — one half might pass while the other fails. Each ISC should be verifiable with a
single tool invocation.

**Daniel's rule:** "If you can't name the single probe that verifies it, it's not atomic yet."

**Implementation:**
- Document in Algorithm file
- No automated enforcement (too subjective for a hook) — this is a prompting constraint
- Add examples of good vs bad ISCs to the Algorithm

**Files:**
- `PAI/Algorithm/v3.14.0.md` — add atomicity doctrine with examples

## Phase C: Containment + Release Safety

### C1. Structural Privacy Enforcement — DONE (patterns.yaml)

**Status:** Completed 2026-05-15. No new hook needed.

SecurityValidator.hook.ts already implements path-based access control (zeroAccess, readOnly,
confirmWrite, noDelete) but had no config file — it was failing open to permissive mode.

**Fix:** Created `PAI/USER/PAISECURITYSYSTEM/patterns.yaml` with zone-based rules:
- personal (readOnly): TELOS/, RELATIONSHIP/, SIGNALS/
- system (confirmWrite): settings.json, identity.jsonc, hooks.jsonc, patterns.yaml itself
- infrastructure (noDelete): KNOWLEDGE/, WISDOM/, Algorithm/, hooks/lib/, config/
- credentials (zeroAccess): ~/.ssh/, ~/.aws/credentials, ~/.gnupg/, settings.local.json

**Verified working:**
- Write to TELOS/ → BLOCKED (readOnly)
- Read from TELOS/ → allowed
- Write to KNOWLEDGE/ → allowed (shared zone)
- Delete KNOWLEDGE/ → requires confirmation (noDelete + rm -r confirm)
- Write settings.json → requires confirmation (confirmWrite)
- git status → trusted (fast-path allow)

No new hook, no new config format, no new tests needed — existing SecurityValidator.test.ts
covers the path matching logic. The patterns.yaml file is the configuration that activates it.

### C2. Two-Stage Release Gate

**Problem:** Our sync-to-kai.sh does everything in one shot: rsync, scrub, verify. If it
passes, we push. Daniel uses two-stage: stage (prepare) → publish (push), never auto-chained.

**Adaptation:**
- `sync-to-kai.sh --stage` — runs rsync + scrub + verify-release but does NOT push
- `sync-to-kai.sh --publish` — only runs after manual review of staged output
- Default behavior (no flag) does `--stage` only (safer default)

**Files:**
- `scripts/sync-to-kai.sh` — add --stage/--publish flags
- `scripts/verify-release.sh` — no changes (already exists)

## Architectural Review (2026-05-15)

Full review examining whether v5.5.0's proposals make architectural sense, overlap with existing
infrastructure, solve real problems, and are correctly sequenced.

---

### The Fundamental Question: What Is v5.5.0 Trying to Do?

v5.5.0 = "Make the Algorithm and system architecture more rigorous by borrowing structural
ideas from Daniel's system."

This is the right direction but has a subtle problem: **it conflates prompt-level doctrine
changes (cheap, fast) with infrastructure changes (expensive, risky).** The plan treats
A1-A3, I1-I3 (pure Algorithm/prompt changes) at the same priority as C1-C2, P1 (new hooks,
daemon refactoring). These are fundamentally different cost profiles:

| Type | Items | Cost | Risk |
|------|-------|------|------|
| Prompt doctrine | A2, A3, I2, I3, D1, D2 | ~0 (edit .md files) | Near zero |
| Algorithm code | A1, I1 | Low (hook regex + gate logic) | Low |
| New infrastructure | C1, C2, P1 | High (new hooks, config, daemon) | Medium-High |

**Recommendation:** Ship all prompt-doctrine items immediately (they're just .md edits).
Gate the infrastructure items behind real need.

### Phase A Review: Algorithm Enhancement

**A1 (Effort Tier Completeness Gates) — CORRECT but incomplete.**

The mapping is sensible. But the implementation is wrong:

> "Add completeness check to Algorithm PLAN→BUILD gate"
> "hooks/handlers/PlanDetection.ts — extend to detect under-specified PRDs at high effort"

PlanDetection runs on **UserPromptSubmit** — it sees the user's prompt, not the PRD content.
It detects whether a plan WAS presented (looking at last assistant output). It cannot inspect
PRD completeness because it doesn't read the PRD file.

The right hook point is **AlgorithmTracker.hook.ts** — it already tracks phase transitions and
reads session state. When it detects a PLAN→BUILD transition, it should check:
- Load PRD from `MEMORY/WORK/{slug}/PRD.md`
- Count sections present vs. sections required for current effort tier
- If insufficient, inject reminder via `additionalContext`

**But wait** — AlgorithmTracker is a **PostToolUse** hook on Bash (it detects phase from curl
commands). It doesn't produce `additionalContext` output (it writes state files asynchronously).
To inject a reminder at phase transition, you'd need either:
1. AlgorithmTracker writes state → next UserPromptSubmit hook reads state and injects warning
   (2-step, works with existing architecture)
2. A new sync hook on the phase transition event (doesn't exist in Claude Code's hook model)

Option 1 is the only viable approach. Document this clearly — it's not "add to PlanDetection,"
it's "AlgorithmTracker writes `needs-completeness-check.json` → PlanApprovalGuard-style hook
reads it and injects warning on next prompt."

**A2 (Closed-List Capabilities) — CORRECT, but almost free.**

This is literally just a list in the Algorithm .md file plus a section in CLAUDE.md. No hook,
no tool, no test needed. It's a 20-minute edit. Ship it with the next commit, don't wait for
v5.5.0. It should be in v5.3.0 Phase D (documentation).

**Severity of "phantom capabilities":** You already have capability selection in the Algorithm
(v3.13.0 lines showing MANDATORY prune, CRITICAL FAILURE for phantom selections). The closed
list just enumerates what's valid. This is already 80% there — just formalize the list.

**A3 (Verification Doctrine) — ALREADY EXISTS, just under-enforced.**

The Algorithm v3.13.0 already says:
- "Executable verification (MANDATORY for all tiers)" (line ~346)
- "If a check produces a command you can run, run it" (line ~346)
- "Adversarial self-check (RECOMMENDED for Standard, MANDATORY for Extended+)" (line ~384)

What's ACTUALLY missing vs. what the plan claims:
- Verification recording (Verification Log section in PRD) — this is new and useful
- "Cross-vendor audit" skipped — correct, we don't have Cato
- "Checkpoint per ISC" auto-commit — this is over-engineering (git noise)

**Revised A3:** Just add `## Verification Log` to PRD template and document format. Skip the
hook. The Algorithm already mandates tool-based verification — what's missing is RECORDING
what was verified, not enforcing that verification happens.

### Phase I Review: ISC Quality Rules

**I1 (ID-Stability) — CORRECT and needed.**

The current CRITERION_PATTERNS already handle:
```
ISC-(C\d+), ISC-(A\d+), ISC-([\w]+-\d+), ISC-(A-[\w]+-\d+)
```

Adding dotted sub-numbering (`ISC-C3.1`) requires updating patterns to:
```
ISC-(C\d+(?:\.\d+)?), ISC-(A\d+(?:\.\d+)?)
```

Small regex change + test update. Correctly scoped.

**I2 (Mandatory Anti-Criterion) — ALREADY IN THE ALGORITHM.**

The Algorithm v3.13.0 already says: "Anti-criteria (ISC-A prefix): what must NOT happen."

What's missing: enforcement of MINIMUM count. Currently it's documented but not enforced.
The plan proposes a soft reminder. That's correct and low-effort.

But: **where does the reminder inject?** Same problem as A1 — AlgorithmTracker detects
criteria counts but doesn't produce additionalContext. You need the same 2-step pattern:
write state file → next sync hook reads and injects.

**I3 (Atomicity) — Pure documentation, zero implementation.**

The plan correctly says "No automated enforcement (too subjective for a hook)." This is just
adding examples to the Algorithm .md file. Do it now, don't wait for v5.5.0.

### Phase C Review: Containment Zones — THE BIG QUESTION

**C1 (Structural Privacy Enforcement) — Solves a problem that `SecurityValidator` already
partially solves, but at a different abstraction level.**

Current state:
- `SecurityValidator.hook.ts` loads `patterns.yaml` for path-based access control
  (zeroAccess, readOnly, confirmWrite, noDelete)
- **BUT patterns.yaml doesn't exist on disk** — SecurityValidator fails open to permissive mode
- `sync-to-kai.sh` has EXCLUDE_PATHS defining what's personal vs. public

The containment zone concept maps to what SecurityValidator already does:
- `personal` zone → readOnly or zeroAccess in patterns.yaml
- `shared` zone → normal access
- `public` zone → normal access

**The real fix:** Create `patterns.yaml` (which SecurityValidator already expects!) with zone
definitions, rather than building a new ContainmentGuard hook. You already have the hook
infrastructure — you just never populated the config.

```yaml
version: "5.5"
philosophy:
  mode: defensive
  principle: "Personal data stays personal; shared data requires intent"

paths:
  zeroAccess:
    - "~/.ssh/"
    - "~/.aws/credentials"
  readOnly:
    - "PAI/USER/TELOS/"
    - "MEMORY/RELATIONSHIP/"
    - "MEMORY/LEARNING/SIGNALS/"
  confirmWrite:
    - "config/identity.jsonc"
    - "settings.json"
  noDelete:
    - "MEMORY/KNOWLEDGE/"
    - "PAI/Algorithm/"

bash:
  blocked: [...]
  trusted: [...]
```

**Recommendation:** Don't build ContainmentGuard.hook.ts. Instead:
1. Create `PAI/USER/PAISECURITYSYSTEM/patterns.yaml` (the file SecurityValidator already looks for)
2. Populate it with zone-based path rules
3. SecurityValidator immediately enforces containment — zero new code

**Cross-validate with sync-to-kai.sh:** The EXCLUDE_PATHS in sync-to-kai.sh should be a
superset of what patterns.yaml marks as `readOnly` + `zeroAccess`. Add a test that verifies
this (grep EXCLUDE_PATHS, grep readOnly, compare sets).

**C2 (Two-Stage Release Gate) — Correct and easy.**

Adding `--stage` / `--publish` flags to sync-to-kai.sh is a 20-line change. The current
script already runs rsync → scrub → verify in sequence. Just add an early exit after verify
if `--stage` is passed. Clean, low-risk.

### Phase P Review: Pulse-Lite — THE WRONG ABSTRACTION

**P1 proposes renaming board.ts → pulse.ts and adding cron + event stream + voice.**

Problems:

1. **board.ts already HAS SSE** (line 523, endpoint at `/api/events`, Content-Type
   `text/event-stream`). The plan claims "add /api/events SSE endpoint" — it already exists.

2. **"Consolidate Board + Cron into Single Process" is a solution looking for a problem.**
   Current cron-like logic lives in hooks (maybeRunSynthesisBackstop in SessionCleanup).
   This runs at session end — which is when the user is actually generating data worth
   processing. A persistent daemon running cron jobs when no session is active would:
   - Consume resources for no benefit (nothing to consolidate if no session happened)
   - Need a separate launchd service (ops burden)
   - Duplicate logic that already works in hooks

3. **Voice notifications via `say` command** — useful for exactly one scenario (long-running
   background agents completing). This doesn't need a daemon — it needs a one-line addition
   to the relevant agent completion hook.

4. **The rename (board.ts → pulse.ts) provides zero value.** It breaks existing references
   (MAINTENANCE.md, memory files mentioning board.ts, KAI Board memory entries).

**Recommendation:** Drop P1 entirely from v5.5.0. If you want scheduled tasks, add them to
SessionCleanup (where they already live). If you want voice, add `say` to the specific event
hooks that warrant notification. A unified daemon is premature at this scale — you don't have
the event volume to justify it.

If a daemon becomes needed later (v5.7+), it would be because you're running background
agents continuously or serving multiple users. Neither is happening now.

### What's Actually Missing from v5.5.0

**1. Algorithm version bump management.**

v5.5.0 proposes Algorithm v3.14.0 but doesn't address: how do we deploy a new Algorithm
version? Currently `v3.13.0.md` is hardcoded in CLAUDE.md (the Read target). Changing to
v3.14.0 means:
- Writing `PAI/Algorithm/v3.14.0.md`
- Updating CLAUDE.md to reference it
- Testing that the new Algorithm doesn't regress behavior

There's no test for "Algorithm produces correct output format." An eval suite for Algorithm
behavior would be more valuable than any single feature in this plan.

**2. The ModeClassifier doesn't know about effort levels.**

ModeClassifier classifies prompts as MINIMAL/ALGORITHM/NATIVE. The Algorithm then determines
effort level internally. But v5.5.0 A1 proposes effort-tier completeness gates.

Question: should ModeClassifier pre-hint effort level? It already detects complexity signals
(word count, multi-step markers). If it also emitted `effort_hint: extended` in additionalContext,
the Algorithm could validate earlier. Currently the Algorithm determines effort after OBSERVE,
which is correct — but means the completeness gate only fires mid-session, not at entry.

This isn't blocking but worth noting as a v5.6 refinement: ModeClassifier → effort hint →
Algorithm validates earlier.

**3. No regression test for Algorithm format output.**

27 test files exist but NONE test Algorithm output format. This is the highest-impact code
in the system (every ALGORITHM response goes through it) with zero automated validation.
A basic eval: "given this prompt, does the Algorithm produce output matching the expected
phase structure?" would catch more bugs than any other test.

**4. SecurityValidator patterns.yaml doesn't exist.**

This is both a v5.5 blocker and an opportunity. SecurityValidator currently fails open
(permissive mode) because no patterns file exists anywhere. This means:
- Path protection is NOT active
- Command blocking is NOT active
- The entire security hook is essentially a no-op

Creating patterns.yaml should be v5.5.0 Phase 0 — it activates the security infrastructure
that's been built but never configured.

### The YAML Observation Applied Here

You asked about YAML — here's where it matters for v5.5.0:

1. **patterns.yaml is the right format** for SecurityValidator's path rules. Human-readable,
   git-diffable, supports comments (with the `yaml` package already imported). The fact that
   this file doesn't exist yet is the biggest structural gap in the security system.

2. **domains.jsonc (JSONC) is the right format** for typed config that BuildSettings processes.
   Don't convert to YAML — the build pipeline already parses JSONC.

3. **YAML frontmatter in .md files** (proposed in v5.4.0 for knowledge files) is the right
   format for document metadata. But ensure ONE parser module — don't let each tool re-parse.

The pattern is: **YAML for human-edited config that isn't processed by BuildSettings. JSONC
for machine-merged config that feeds settings.json. TypeScript types for runtime validation
of either.**
