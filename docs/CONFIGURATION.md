# Configuration Guide

## Architecture

KAI uses domain-specific config files that merge into a single settings.json:

```
config/identity.jsonc          → Who you are, who KAI is
config/preferences.jsonc       → Env vars, tech stack, runtime settings
config/hooks.jsonc             → Hook registrations
config/permissions.jsonc       → Tool allow/deny/ask rules
config/notifications.jsonc     → Alert channels and routing
config/domains.jsonc           → Knowledge domain definitions
config/preferences.local.jsonc → Machine-specific overrides (gitignored)
```

## The Build Pipeline

```
config/*.jsonc → BuildSettings.ts → settings.json (generated, gitignored)
```

`settings.json` is what Claude Code actually reads. You never edit it directly —
it's rebuilt automatically at session start if any config file changed, or
manually via:

```bash
bun ~/.claude/hooks/handlers/BuildSettings.ts
```

## When to Use Each File

| I want to... | Edit this file |
|---|---|
| Change my name or timezone | `config/identity.jsonc` |
| Add an environment variable (global) | `config/preferences.jsonc` |
| Add an environment variable (this machine only) | `config/preferences.local.jsonc` |
| Add an MCP server | `config/preferences.local.jsonc` |
| Allow a new tool | `config/permissions.jsonc` |
| Set up notifications | `config/notifications.jsonc` |
| Change knowledge domains | `config/domains.jsonc` |
| Override something just on this machine | `config/preferences.local.jsonc` |

## preferences.local.jsonc

This file is for anything machine-specific that shouldn't be in git:
- MCP servers only available on this machine
- API keys (alternative to shell profile)
- Dev server ports
- Project paths that differ between machines

It merges ON TOP of preferences.jsonc — same keys override, new keys are added.

Example:
```jsonc
{
  "env": {
    "MY_CUSTOM_VAR": "value",
    "DEV_API_URL": "http://localhost:8080"
  },
  "mcpServers": {
    "my-server": { "url": "http://localhost:3000/mcp" }
  }
}
```

## Environment Variable Precedence

```
process.env (shell profile)  >  .env file  >  preferences.local.jsonc  >  preferences.jsonc
```

Shell profile exports always win. The `.env` file is a convenience for keys
you don't want to export globally.

## After Any Config Change

```bash
bun ~/.claude/hooks/handlers/BuildSettings.ts
```

This also runs automatically at session start if config files have changed
(via the BuildSettings handler registered in hooks.jsonc).

## Never Edit Directly

- `settings.json` — generated, overwritten on rebuild
- `CLAUDE.md` — generated from CLAUDE.md.template + settings
- `skills-lock.json` — generated by skills-lock.ts

## Config Starters

The `config/starters/` directory contains archetype templates for domains.jsonc.
These are applied during installation and can be re-applied:

```bash
cp config/starters/fullstack-domains.jsonc config/domains.jsonc
bun ~/.claude/hooks/handlers/BuildSettings.ts
```

Available archetypes: generic, full-stack, data-science, devops.
