## The Algorithm 3.14.0

Core: transition from CURRENT STATE to IDEAL STATE using verifiable criteria (ISC). Goal: **Euphoric Surprise** — 9-10 ratings.

### /goal for Background Sessions

**When /goal applies:** Background sessions only — sessions dispatched via Agent View without a human in the loop. Interactive sessions use human verification loops after each phase; background sessions use /goal as the convergence mechanism.

**Relationship to ISC:** /goal is the session-level completion condition (outer loop). ISC is the step-level verification criteria (inner methodology). They compose: the /goal references ISC as the success definition.

**Deriving /goal from ISC:** The /goal should be the top-level success condition that encompasses all ISC criteria. It's the conjunction of all critical ISC + the verification evidence that matters most.

**Example:**
- ISC includes:
  - ISC-C1: `bun test` exits 0 with all tests passing
  - ISC-C2: No TypeScript errors (`tsc --noEmit` exits 0)
  - ISC-C3: Login flow renders in browser without errors
- Derived /goal: `all tests pass, tsc reports no errors, and the login flow works end-to-end`

**How it works:** Setting a /goal triggers the Algorithm internally. The agent runs OBSERVE→THINK→ACT→VERIFY as normal, but loops until the /goal condition is satisfied. The /goal becomes the exit criterion — when it evaluates to true, the session ends.

**When to set /goal:** At Algorithm entry, if `mode: background` in PRD frontmatter. Construct the /goal from the ISC generated in OBSERVE, after Quality Gates pass. The /goal should be executable — a command or check the agent can run to know it's done.

**Anti-pattern:** /goal that references subjective criteria ("code is clean") or unverifiable states ("user is happy"). The /goal must be binary testable, just like ISC.

### PRD as System of Record

**The AI writes ALL PRD content directly using Write/Edit tools.** PRD.md in `MEMORY/WORK/{slug}/` is the single source of truth. The AI is the sole writer — no hooks, no indirection.

**What the AI writes directly:**
- YAML frontmatter (task, slug, effort, phase, progress, mode, started, updated; optional: iteration)
- All prose sections (Context, Criteria, Decisions, Verification)
- Criteria checkboxes (`- [ ] ISC-1: text` and `- [x] ISC-1: text`)
- Progress counter in frontmatter (`progress: 3/8`)
- Phase transitions in frontmatter (`phase: execute`)

**What hooks do (read-only from PRD):** A PostToolUse hook (PRDSync.hook.ts) fires on Write/Edit of PRD.md and syncs frontmatter + criteria to `work.json` for the dashboard. **Hooks never write to PRD.md — they only read it.**

**Every criterion must be ATOMIC** — one verifiable end-state per criterion, 8-12 words, binary testable. See [ISC-Methodology.md](ISC-Methodology.md).

### Atomicity Rule

Each ISC must be verifiable with a single tool invocation. If you can't name the one command/check that proves it, split it.

**Good (atomic):**
- ISC-C1: `bun test` exits 0 with all tests passing
- ISC-C2: `/api/health` endpoint returns 200
- ISC-C3: No TypeScript errors (`tsc --noEmit` exits 0)

**Bad (compound — split these):**
- ~~ISC-C4: Tests pass and coverage exceeds 80%~~ → split into C4 (tests pass) + C5 (coverage ≥80%)
- ~~ISC-C5: API responds correctly and handles errors~~ → split into C5 (200 on valid) + C6 (4xx on invalid)
- ~~ISC-C6: UI renders and is accessible~~ → split into C6 (renders without error) + C7 (axe audit 0 violations)

**Anti-criteria** (ISC-A prefix): what must NOT happen.

- **Removal Protocol (MANDATORY when deleting files, features, hooks, or references):** Deleting code is only half the operation. The other half is finding every reference to what was deleted. Before marking a removal as done:
  1. `grep -rn "FeatureName"` across all tracked text files — fix or remove every hit

- **Rename/Rebrand Protocol (MANDATORY when renaming identifiers, brands, or paths across files):** Renaming in N known files while missing M unknown files is a guaranteed multi-round defect. Before marking a rename as done:

  3. For count changes (e.g., removing a hook changes "37 hooks" to "35 hooks"): grep for the OLD number across all docs — update every occurrence, not just the first one found
  4. Gate: zero remaining matches before proceeding. If matches remain in legitimate contexts (e.g., directory names), document each exception explicitly.

  4. For removals: grep for the removed feature's name in all tracked files
  5. Gate: zero unexpected matches. If legitimate matches remain (e.g., in archive/ dirs), document each exception.

  This check answers: "did I get EVERY instance?" Observational review of edited files is insufficient — run the grep.

  1. **What was fixed** (the specific change)
